# churl ‚Äî Architecture

## Crate layout

### `churl-core` (library)

Zero TUI dependencies ‚Äî ever. This constraint is enforced by code review and CI; adding `ratatui`/`crossterm`/`termion` to this crate is always wrong.

| Module | Responsibility |

Every live proxy/TLS/cookies change rebuilds this single client (`App::rebuild_client`) ‚Äî there is never more than one, and it always reflects current session state. Auth injected via `auth::apply_auth` (M5): header effects skipped when an enabled user header with the same name exists (the user's header wins), query effects appended after enabled params. No `{{var}}` templating (M6); URL/headers/body/auth used verbatim |
| `history` | SQLite via `rusqlite` (bundled); schema managed via migrations at startup. **R1 D2 durability/concurrency**: `open` sets `busy_timeout=5000` (`BUSY_TIMEOUT_MS`) + `journal_mode=WAL` (fails loud `WalUnavailable` on FS fallback; asserted for file-backed DBs only), and `migrate()` runs the whole loop inside ONE `BEGIN IMMEDIATE` transaction (re-reads `user_version` under the write lock, applies still-pending migrations, commits once ‚Äî all-or-nothing; a concurrent opener blocks then no-ops). Migration 3 (M7.5) adds a SEPARATE `load_batches` table (`LoadBatchSummary` / `insert_load_batch` / `recent_load_batches`): a load run writes exactly ONE summary row there, never to `history`, so the per-endpoint history view is never flooded; migration 4 appends the `mean_ms` column via `ALTER TABLE` (migrations are append-only ‚Äî never edit an applied one); migration 5 (M8) adds a `cookies` table (`workspace TEXT PRIMARY KEY, jar_json, updated_at`) storing one persistent cookie-jar blob per canonicalized workspace root (`cookie_jar`/`save_cookie_jar`), loaded on workspace open and saved after mutating sends / toggle-off / clear / exit. **M8.1 off-thread writer**: the cookie-jar write is offloaded to `history::CookieJarWriter` ‚Äî a dedicated OS thread owning its OWN `HistoryStore` connection to the same file (a harmless second connection: WAL is already set, the migration runner re-reads the version and no-ops), fed coalesced `(workspace, jar_json, updated_at)` snapshots over a `Mutex<HashMap>` + `Condvar` (mirrors the `highlight` off-thread worker pattern). The UI thread serializes the jar (cheap, in-RAM) then hands off the snapshot, so the blocking `conn.execute` ‚Äî which can wait up to the WAL `busy_timeout` under cross-process contention ‚Äî never stalls it. Invariants: **final-state-wins** (the queue is a `workspace ‚Üí latest snapshot` map and the single writer thread drains in order, so no older snapshot lands after a newer one; bounded by one row per key), **no lost writes on quit** (`shutdown` drains the queue then joins the thread ‚Äî the run loop calls it after the final `persist_cookie_jar`), and **no clobber on failure** (serialization skips the enqueue on error so the writer only ever receives a good blob, and `save_cookie_jar`'s atomic upsert leaves the prior blob intact on a DB error, surfaced via `take_error`). History writes stay synchronous (out of scope). **R1 D4b pruning**: `insert`/`insert_load_batch` prune-on-insert to the newest `HISTORY_ROW_CAP`=10 000 rows/table (indexed `DELETE ‚Ä¶ WHERE id <= max_id - CAP`); `recent()` reads unaffected |
| `config` | `~/.config/churl/config.toml` (incl. flat `[keys]` override strings, `timeout_secs`, `max_body_bytes` ‚Äî resolved via `Config::timeout()`/`Config::max_body_bytes()`; the M7.5 `[load]` guardrail caps via `Config::load_caps()`; the M8 `proxy`/`insecure`/`cookies` launch defaults via `Config::{proxy,insecure,cookies}()`) and per-workspace `churl.toml` (also `proxy`/`cookies`); never contains secrets. Secrets heuristics: `looks_like_secret_name`, `is_template_placeholder`, `secret_violations` (manifest), `auth_secret_violations` (endpoint auth, M5), `proxy_has_credentials` (M8 ‚Äî a persisted proxy must be credential-free; the save path refuses `user:pass@`, never strips) |
| `cookies` | (M8) `ChurlCookieJar` ‚Äî a thin `RwLock<cookie_store::CookieStore>` implementing reqwest's `cookie::CookieStore` trait (mirrors reqwest's own `Jar`, but over an **owned** store we can serialize). Cross-origin safety is a store property: `cookies()` does RFC 6265 domain/path/`Secure` matching, so a cookie set by host A is never sent to host B ‚Äî this layers *under* the manual cross-origin `Cookie`-header strip in `http`. `to_json`/`load_json` persist only persistent+unexpired cookies; `list`/`delete`/`clear` drive the Options overlay + `churl cookies` CLI. **M8.1 poison recovery**: every guard acquisition goes through `store_read`/`store_write`, which `unwrap_or_else(|p| p.into_inner())` instead of `.expect(...)` ‚Äî a panic elsewhere while holding the write guard poisons the lock, and recovery keeps every later cookie op working rather than turning one panic into a permanent crash-on-every-jar-access wedge |

**Bracketed paste.** `tui::init` requests bracketed paste (`EnableBracketedPaste`, disabled again in `restore`), so a multi-line clipboard paste (e.g. a browser "Copy as cURL") arrives as a single `Event::Paste(text)` instead of a key stream whose embedded newlines would submit a prompt early. The run loop routes it to `App::handle_paste`, whose routing **mirrors `handle_key`'s precedence exactly** ‚Äî because bracketed paste is enabled globally, every surface that accepts typed keys must also accept a paste, or the payload would silently vanish. Covered: help-search, the fuzzy pickers/palette (`PickerState::push_str` + refilter), body-search, the CRUD/new-endpoint prompt, the env-editor sub-inputs (session-var / profile-name / row-field, via `EnvEditorState::paste` which live-mirrors the row), the Options proxy field, the sequence editor (add-step filter / rule field, `SequenceEditorState::paste`), the load-runner numeric field (digit-filtered, `LoadRunnerState::paste`), the inline URL editor, and the request-row `FieldEdit` (both `LineEditor::insert_str`). The edtui multi-line surfaces (Body editor, URL popup) get the **raw** payload via `on_paste_event` and are gated on an insert-capable (non-Normal) mode so the text lands at the cursor. `LineEditor`/picker surfaces get newlines normalised to `\n` (`normalize_paste_newlines`: single-line editors never want a raw CR, and the curl prompt's `\`-continuation logic keys on LF); the edtui body path keeps CRLF so a pasted body is preserved. With the full curl now intact in the prompt buffer, the existing submit ‚Üí `import_curl` path handles the `\`-continuations and `\[\]` glob-escapes. When no text surface is active (Normal mode with no inline editor, jump, a y/n confirm, a pending leader, the method menu) the paste is dropped ‚Äî never replayed as keys.

**Import secret capture.** `import_curl` remaps `Authorization: Bearer <token>` / `-u user:pass` to first-class `Auth::Bearer`/`Auth::Basic` with the secret replaced by a `{{token}}`/`{{password}}` placeholder (no secrets in workspace files). `ImportResult.captured_secrets` now carries the extracted `(var_name, real_value)` pairs (empty when the value was already a placeholder or absent). The TUI import path (`create_endpoint_from_curl`) binds each into a RAM-only **Session var** via `write_session_var` so `{{token}}` resolves and the endpoint is immediately sendable; the file on disk still holds only the placeholder, and the real value is never echoed. An existing same-named session var is overwritten for now (dedup/rename deferred ‚Äî see ROADMAP). The CLI (`run_import`) has no live session, so it leaves the placeholder and keeps the "supply via profile/env" warning; it never prints the captured value.

**CLI `churl import` input forms** (`Import.curl: Vec<String>`, `trailing_var_arg`): **0 tokens** (or a lone `-`) reads the whole command from **stdin** (`pbpaste | churl import`), guarded by an `IsTerminal` check that refuses with a usage hint rather than hanging on an interactive TTY; **1 token** is a full command STRING ‚Üí `import_curl` (shlex-tokenises); **>1 tokens** are raw shell-tokenised args ‚Üí `import_curl_tokens` parses them directly (no re-split), so `churl import curl 'url' -H '‚Ä¶'` works with `-H` etc. captured rather than parsed as churl flags. `--name`/`--out` must precede the command. Both string and token paths share one parser: `import_curl(s) = import_curl_tokens(shlex::split(normalize_continuations(s)))`.

Since M3 `AppMsg` also carries `Response { generation, outcome, meta }` (a stale generation ‚Äî after cancel/resend ‚Äî is dropped) and `Highlighted { hash, lines }`. The app channel is **bounded** (`mpsc::channel(APP_CHANNEL_CAPACITY=1024)`, R1 D4a) so a fast/high-`total` load run can't flood it with unbounded owned `Response`/`Highlighted` bodies; every sender is a spawned async task (`.send().await`) or the highlight OS thread (`blocking_send`), never the render/UI thread, so a full queue applies backpressure without stalling input. Render functions stay pure (`fn render(frame, area, &state)`); the response pane's `render` additionally *returns* an optional `HighlightJob` for the caller to enqueue (top-level `render` already takes `&mut App`). Highlight caches live per-`EndpointBuffer` (capped at 64) and are evicted from the previously-active buffer on every buffer switch (`set_active_buffer`, R1 D4c), bounding total cached lines to (active-buffers √ó 64). Under `TestBackend` no highlight worker exists, so snapshots deterministically show plain text.

**History-write reliability (R1.5 B).** On **quit**, the `run` loop exits and the tokio runtime drops, aborting any in-flight `send` task before its `Response` can land ‚Äî so `record_inflight_on_quit()` runs at the end of `run` (runtime still alive) and writes an interrupted history row (`write_history(meta, None, None)`, mirroring `cancel_request`) for **every** buffer with an `in_flight` request, so a request completed-on-the-wire but quit before landing isn't silently lost (B1). A **persistent** history/load-summary write failure is surfaced stickily, not just as a 250 ms toast: `App.history_write_failures` counts consecutive `insert`/`insert_load_batch` failures (`note_history_write` resets to 0 on any success, called from both write sites), and once it reaches `HISTORY_FAIL_THRESHOLD` (=1) the statusline shows a persistent `‚öÝ history not recording` flag (`theme.status_error`) until a write succeeds (B3) ‚Äî a healthy session never arms it, so all 95 snapshots are unaffected.

Key routing precedence (per key event, in `Mode::Normal` the crokey map is authoritative):

0. Jump-mode (`f` by default) is an overlay-level mode alongside search/palette: it consumes every key ‚Äî a label char focuses that pane / selects that explorer row, `Esc` (or the Jump key again) cancels, everything else is ignored.
1. 2. 3. Request pane, Body tab, edtui in a non-Normal mode: keys go to edtui with the same Ctrl-S/Ctrl-C interception (M4, DECISIONS.md).

### Response viewer pipeline (M7)

The response viewer composes a reformat pass plus three pure transforms over the logical lines (`components/response/`); the reformat/sanitize/index stages run once per response (on build / toggle), while the fold‚Üíwrap‚Üíviewport geometry is memoized per-frame by the R1.5 A3 geometry cache (below) so an idle tick over a multi-MB body no longer re-walks it:

```
raw body bytes ‚Üí lossy-UTF-8 decode  (kept verbatim as `raw_text` ‚Äî copy reads this)
  ‚Üí reformat pass    (JSON-only, pretty-by-default; malformed ‚Üí raw fallback)
  ‚Üí sanitize pass    (strip ANSI ¬∑ expand tabs @4 ¬∑ CRLF‚ÜíLF ¬∑ controls‚Üí`¬∑`) ‚áí displayed `text`
  ‚Üí line index       (`line_offsets` over the DISPLAYED text ‚Äî navigation matches what's shown)
  ‚Üí fold filter      (JSON-only; folded regions elided to a `‚ãØ N lines` header)
  ‚Üí wrap expansion    (optional; each display row = a char sub-range of one logical line;
                       unwrapped rows are windowed to `[h_scroll, h_scroll+width)`)
  ‚Üí viewport slice    (scroll offset + height)
```

**Reformat pass (M7.7 Stage A).** `reformat_body_if_needed(text, syntax, pretty, sort_keys)` runs before the fold/wrap/viewport stages. When `pretty` is on and `syntax == Json` it parses the body as a `serde_json::Value` and re-emits it with `to_string_pretty` (JSON-only in v1 ‚Äî zero new deps since `serde_json` is already a workspace dep); on **any** parse error it returns the raw text unchanged (silent fallback ‚Äî the viewer never errors on a malformed body). json-ish content-types default to pretty on arrival, everything else to raw. `p` (`Action::TogglePretty`) flips raw‚Üîpretty: it rebuilds `text`/`line_offsets` from `raw_text`, bumps `generation` (invalidating the syntect viewport cache), and **resets folds** (openers are position-based; the layouts differ). `pretty` is also hashed into `viewport_hash`. The separate `raw_text` field is what copy (`y`/`Y`) reads, so copy stays byte-exact regardless of the toggle ‚Äî reformatting drives display only.

**Optional A‚ÜíZ key sort (M7.7 Stage A follow-up).** A `sort_keys` flag (default **false**, per-response, nothing persisted) layers an optional alphabetical key sort onto the pretty view. `s` (`Action::ToggleSortKeys`) flips it via `ResponseView::toggle_sort_keys`, which mirrors `toggle_pretty` exactly (rebuild `text`/`line_offsets` through the reformat path, bump `generation`, reset folds, clear search); `sort_keys` is also hashed into `viewport_hash`. When set (and pretty-on + JSON), `sort_value_keys` recursively rebuilds every `serde_json::Value::Object`'s entries in `String::cmp` order before `to_string_pretty` ‚Äî **arrays keep element order**, only object keys sort. It works because the workspace's `preserve_order` feature makes the rebuilt map re-emit in the insertion (= sorted) order written. Sort is **independent of `pretty`**: it only affects display while pretty is on; the app handler guards on pretty-JSON-body and otherwise no-ops with a `sort: pretty JSON only` notice. Default stays wire order; copy is unaffected (still `raw_text`, byte-exact).

**Sanitize pass + horizontal window (M7.7 close-out).** `sanitize_for_display(text)` runs on the reformatted text *before* `index_lines` (never on `raw_text`, so copy stays byte-exact): a single left-to-right scan strips ANSI escapes (CSI/OSC/lone-ESC ‚Äî zero-width; blocks cursor moves, recolours, OSC-52 clipboard writes), expands `\t` to the next multiple of `TAB_WIDTH = 4` (a documented const, not config; columns reset at each `\n`), collapses `\r\n`‚Üí`\n` (a lone `\r` is a control char, so no phantom blank lines), and replaces every other C0/DEL/C1 control with `¬∑` (U+00B7). Correct JSON is already clean; the teeth are on the raw / parse-failed / `text/plain` path. Separately, an `h_scroll` char-column offset on `ResponseView` bounds unwrapped long-line render cost: `expand_wrap` windows each unwrapped **normal-line** row to `[h_scroll, h_scroll + width)` (fold-headers keep full width; wrap unchanged) ‚Äî row *count* is unchanged, so the display-row‚Üîlogical mapping stays intact. `H`/`L` (aliased to Left/Right; `Action::ScrollBodyLeft`/`Right`, remappable) pan it wrap-off-only; `render_done` clamps it to the widest visible line (mirror of `clamp_scroll`) and reports `clamped_h_scroll` for write-back; search-into-view pans a far-right match into the window (`ensure_column_visible`). `h_scroll` resets on build/generation-bump/`ToggleWrap` and keys `viewport_hash`.

Cursor and scroll are **display-row** indices (post-fold, post-wrap); the cursor follows-and-scrolls at render time. Search matches are stored against *logical* lines (byte ranges) and mapped through the pipeline for navigation, auto-unfold, and highlight overlay. Per-view UI state (view mode, folds, wrap, pretty, sort-keys, search, `h_scroll`) lives on `ResponseView` so it resets on each new response; cursor/scroll/viewport geometry live in a `response::ResponseGeometry { scroll, cursor, total_rows, viewport_height, viewport_width }`. Fold regions come from `components::fold::scan_regions` (single O(n) string-aware pass over the displayed text, cached lazily). Highlighting is deferred under wrap (wrapped bodies render plain); unwrapped bodies keep full off-thread syntect highlighting with a duplicate-enqueue guard (`App::pending_highlight`).

**Render geometry cache (R1.5 A3).** The fold‚Üíwrap‚Üíviewport geometry ‚Äî the fold-filtered `Vec<Visible>` (O(logical lines)), the wrap-expanded `Vec<DisplayRow>` (O(total body chars)), and `max_h_scroll`'s widest-visible-line scan (O(total visible chars)) ‚Äî was recomputed **every frame, including idle 250 ms ticks**, so a multi-MB body cost O(body) per tick even when nothing changed. It is now memoized in a single-slot `GeomCache` (a `RefCell` field on `ResponseView`, so interior-mutable through the pure `&self` render + geometry accessors) with one slot per product, each keyed on the signature that captures exactly its inputs: the display-rows slot on `RowsSig { visible_sig, wrap, width, h_scroll }`, the widest-line slot on `MaxHScrollSig { visible_sig, wrap, width }` (deliberately **omitting** `h_scroll` ‚Äî the widest line is pan-independent, so a pure horizontal scroll still hits it), where `visible_sig = { generation, headers, fold_sig }` covers every displayed-text rebuild (`generation` ‚Äî pretty/sort bump it), the body‚Üîheaders `view_mode` switch (`headers`), and the folded-opener set (`fold_sig`, the same sorted-openers hash the highlight `viewport_hash` uses). A signature match ‚áí a recompute would be byte-identical, so serving the memo is behaviour-preserving (the 95 `tui_snapshot` snapshots prove it); a miss recomputes and **replaces** the slot (never grows), and the whole cache dies with the view when a new response replaces it ‚Äî matching the per-`EndpointBuffer` highlight cache's bound + generation discipline. `line_numbers` is not a separate key input: it feeds the pipeline only by shrinking the effective `width` (gutter) before the geometry calls, so keying `width` covers it. **Deliberately deferred:** the audit's cheaper alternative ‚Äî a dirty-flag gate on `terminal.draw` ‚Äî is **not** shipped, because gating the whole draw risks a stale UI if any tick-driven update (in-flight spinner, elapsed-`ms` readout, message expiry) is missed; the geometry cache removes the per-frame O(body) recompute while every tick still redraws, keeping animations live.

**Unified runner response viewer (drive-test note #2).** `ResponseGeometry` is carried by the main-pane endpoint buffer AND by both runner states (`LoadRunnerState`/`SequenceRunnerState`). `App::active_response_surface()` resolves `Main | LoadRunner | Sequence` from `self.mode` + the runner's `focus`; `active_response()` / `active_response_geometry()` (+ `_mut`) route through it, so the `response_*` handlers are mode-aware without per-handler branching. A runner's Response-region key is dispatched through the SAME `lookup_ctx(_, PaneCtx::Response)` + `response_*` handlers the main pane uses (`App::try_route_runner_response_key`, called before delegating to the runner) ‚Äî full parity (headers/wrap/pretty/sort/line-numbers/fold/search/copy/hscroll + cursor nav), zero duplication. The two runners no longer carry their own `handle_response_key`. The highlight cache/guard stays on the active endpoint buffer (shared by all three surfaces). Body search opened over a runner restores `App::body_search_return` on close so it returns to the runner, not to Normal.

**Honest, yankable failed row (drive-test #4a).** A transport failure is `ResponseState::Failed { error, meta }` (never `Done`), so it carries no `ResponseView`. Because the copy handler `response_copy_view` reads through `response_view_mut()` (which is `Some` only for `Done`), `y` was a silent no-op on a failed row. The fix stays in the shared viewer: the `response::render` `Failed` arm renders an honest panel ‚Äî the request method+URL (from `meta`), the error, and an explicit "no response ‚Äî status, body, timing unavailable" line (a transport failure genuinely has none; nothing is fabricated) ‚Äî and `ResponseState::failure_copy_text() -> Option<String>` yields the copyable blurb (request line + error) for `Failed`, `None` otherwise. `response_copy_view` gains one state-branch: copy the view when `Done` (unchanged byte-exact path), else `enqueue_clipboard(failure_copy_text())`. Both live in the single shared surface, so the main pane and both runners get it without per-site duplication. `Dropped` (memory-evicted load rows) has no body bytes and returns `None` ‚Äî `y` stays a deliberate no-op there.

**Line-number gutter (drive-test note #8).** A `line_numbers: bool` on `ResponseView` (default **true**, toggled by `#` / `Action::ToggleLineNumbers`) draws a dim, right-aligned left gutter showing each display row's **1-based displayed logical line number**. The number shows on the first display row of each visible entry (tracked by `visible_idx`, robust to `h_scroll`); wrapped-continuation rows show a blank aligned gutter; fold-headers show the opener line's number. `ResponseView::gutter_width()` = digits of the **total displayed line count** + 1 pad (stable while scrolling; `0` when off). **Width seam**: `render_done` subtracts `gutter_width` from `body_area.width` **once** and threads the reduced width through `max_h_scroll`, `expand_wrap` (wrap split + `h_scroll` window), and `RenderOutcome::viewport_width` ‚Äî the caller feeds `viewport_width` back into `logical_at_display_row` / `display_row_for_logical` / `ensure_column_visible`, so cursor‚Üîlogical mapping and search-into-view use the same gutter-adjusted width. Render-only: `raw_text`, copy, and `line_offsets` are untouched (`viewport_hash` keys on `line_numbers` so a toggle re-highlights at the new width). The gutter also appears in the load-runner / sequence-runner responses via the shared `ResponseView`.

**Shared smart-case matcher (M7.7 Stage B).** The body-search matching algorithm lives in one place ‚Äî `response::smart_case_matches(lines, query) -> Vec<(line, start, end)>` (smart-case: all-lowercase = case-insensitive, any uppercase = case-sensitive; unicode case-fold offsets via `lowercase_with_offsets`/`map_lowered_offset`). `ResponseView::compute_matches` delegates to it, and the `?` help overlay's `/` search calls it over the help lines' plain text, so the two searches can never diverge. The match-span painter `build_search_spans` (current match reversed, others dim+underlined) is shared too. **Help-overlay search** (`Mode`-free ‚Äî the help overlay intercepts keys ahead of the mode machine) is **highlight-and-jump, never a hiding filter**: state is `help_search: Option<help::HelpSearch>` + `help_search_editor: LineEditor` + `help_search_input: bool` on `App`; `/` opens it inside `handle_help_key`, `handle_help_search_key` mirrors `handle_body_search_key` (typing recomputes + jumps, `n`/`N` cycle/wrap, Esc clears keeping the overlay open, Enter commits keeping matches), and `help::render` paints highlights + a `k/N` counter in the title while leaving every row present.

## On-disk format

```
<workspace>/                    # a git repo the user owns == the ROOT COLLECTION (M7.9)
  churl.toml                    # root-collection meta: name + [vars] + [[profiles]] (no secrets)
  <endpoint>.toml               # root-level endpoints attach directly to the root collection
  <collection>/                 # a directory = a collection; nests to arbitrary depth (M7.9)
    folder.toml?                # optional sub-collection metadata + flat [vars] defaults (M6)
    <endpoint>.toml             # one file per endpoint; explicit `seq` for ordering
    <sub-collection>/           # collections nest recursively ‚Äî no depth cap
      folder.toml?
      <endpoint>.toml
  sequences/                    # reserved dir (M7.4) at the ROOT ONLY ‚Äî request sequences,
    <sequence>.toml             #   NOT a collection; a nested `sequences/` IS a collection
```

M7.9 folded the workspace into the **root collection**: everything is a collection, the workspace root *is* the root collection, collections nest to arbitrary depth, and endpoints attach to any node (root included). `churl.toml` is the root collection's meta (name + `[vars]` + `[[profiles]]`); sub-collections carry only `folder.toml` (vars). The root additionally owns the global `sequences/` store and profiles. Var resolution walks the endpoint's collection ancestor chain leaf‚Üíroot (inherit-and-override) ‚Äî see the `template` module and DECISIONS.md "Variable scoping". A pre-M7.9 flat workspace loads unchanged (its `churl.toml` `[vars]` already are the root vars; root endpoints are net-new files) ‚Äî no migration.

Sequence file shape (M7.4):

```toml
seq = 0
name = "Auth flow"
on_error = "halt"               # halt (default) | continue

[[step]]
seq = 0
endpoint = "auth/login.toml"    # workspace-relative endpoint path
persist = ["token"]             # note #6: these rules' captured values feed the
                                #   in-memory Session scope (surviving the run);
                                #   omitted/empty => every rule is Run-only (ephemeral)
[step.extract]
token = "$.data.token"          # var name -> extraction expression
user_id = "$.data.user.id"

[[step]]
seq = 1
endpoint = "users/me.toml"      # its request uses {{token}} ‚Äî resolved from the extracted scope
```

**In-memory Session store (note #6).** `persist` lists only the rule *names*; the captured *values* are written into a process-lifetime, in-memory Session store (`App::session_vars`, keyed by canonical workspace root) and are **NEVER written to disk** ‚Äî a security feature, so captured secrets never touch the filesystem and evaporate on exit. The Session scope is the highest-precedence resolver layer (`session > cli > ‚Ä¶`) for both sequence-run and standalone-send resolution, so a token captured by a `login` sequence resolves in a later standalone `{{token}}` send. The `<leader>v` env editor shows the store as a read-only, masked `Session` group with a clear action.

Endpoint file shape (M1):

```toml
seq = 1                     # explicit ordering within the collection
name = "Get user"

[request]
method = "GET"
url = "https://api.example.com/users/{{id}}"

[[request.headers]]         # array-of-tables; `enabled = false` to disable a line
name = "Accept"
value = "application/json"

[request.body]              # optional; type = text|json|form (default text)
type = "json"
content = '{"q": true}'

[request.auth]              # optional (M5); type = basic|bearer|apikey
type = "basic"              # secret values must be {{var}} placeholders ‚Äî
username = "alice"          # save_endpoint/endpoint_to_toml refuse literals
password = "{{password}}"
```

Saves are format-preserving: comments and ordering in hand-edited files survive a churl round-trip (see DECISIONS.md for merge semantics and edge cases). Every save is also **atomic and durable** (R0): the single write in `save_value` funnels through `persistence::atomic_write` (temp sibling file ‚Üí fsync ‚Üí atomic `rename` ‚Üí parent-dir fsync), so a crash mid-write can never tear the source-of-truth file. The error contract is unchanged (`PersistenceError::Write`).

Workspace `churl.toml` and collection `folder.toml` (M6):

```toml
# churl.toml (the root collection's meta, M7.9)
name = "my-api"

[vars]                        # root-collection template defaults, outermost scope (no secrets)
base_url = "https://api.example.com"

[[profiles]]                  # per-environment; profile vars beat collection vars
name = "prod"
[profiles.vars]
host = "prod.example.com"

# <collection>/folder.toml
[vars]                        # collection-level defaults, environment-independent
page_size = "50"
```

Global config `~/.config/churl/config.toml` (M6 adds theming):

```toml
theme = "dark"                # built-in: "dark" (default) | "light"

[theme_colors]                # per-slot overrides (named ANSI or #rrggbb)
title = "cyan"                # unknown slot / bad colour = loud startup error
jump_label = "#ffcc00"        # (table is [theme_colors], not [theme.colors] ‚Äî see DECISIONS.md)

[keys]                        # action remaps: "combination" = "action-name"
"ctrl-p" = "open-palette"
```

High-churn state (history, cookies, cached responses) lives in:
```
$XDG_DATA_HOME/churl/state.sqlite
```
Never in the workspace ‚Äî workspace = safe to commit/sync; state.sqlite = never committed.

Render-side caches (line-offset index, wrap layout, viewport-only syntect highlighting) are in-process only ‚Äî not persisted.

## Performance budget

| Concern | Target | Mechanism |
|---|---|---|
| Cold start | < 100 ms | Lazy collection loading; lazy syntect init |
| 1 MB response body | Smooth scroll | Virtualised line render; line-offset index; viewport-only highlighting |
| Syntax highlighting | < 23 ms per viewport | `syntect` + `two-face` colours, off-thread, cached by viewport hash |

No HTTP-semantic caching by design ‚Äî churl is a development tool; stale responses are a footgun. History covers recall.

## Tutorial scaffold (`churl tutorial`)

`crates/churl/src/tutorial.rs` ‚Äî the `Tutorial` subcommand scaffolds a demo workspace so a first-time user can send a request in under a minute.

The scaffold writes through the real `churl-core::persistence` seams ‚Äî no hand-written TOML strings for endpoint or folder files:

1. `save_workspace_manifest(root, &ws)` ‚Äî writes `churl.toml` with `name`, `vars` (`base_url`), and a `dev` profile.
2. `create_collection(root, "examples")` ‚Äî creates the collection directory.
3. `save_collection_meta(coll_dir, &Default::default())` ‚Äî writes `examples/folder.toml` (empty vars).
4. Three `create_endpoint(coll_dir, name)` + `save_endpoint(path, &ep)` calls ‚Äî writes the endpoints through the format-preserving merge serializer.

Endpoints target [httpbingo.org](https://httpbingo.org): Get Anything (GET `/anything?name=churl`), Post JSON (POST `/post` with JSON body), Bearer Auth (GET `/bearer` with `{{token}}` placeholder auth). The scaffold refuses a non-empty directory.

## Release pipeline

Tag-triggered GitHub Actions workflow (`.github/workflows/release.yml`):
- `taiki-e/upload-rust-binary-action@v1` ‚Äî cross-compiles, strips, archives, and uploads per-target binaries with SHA-256 checksums.
- Targets: `aarch64-apple-darwin`, `x86_64-apple-darwin`, `x86_64-unknown-linux-musl` (static), `aarch64-unknown-linux-musl` (static), `x86_64-pc-windows-msvc`.
- musl static targets are viable because the HTTP stack is rustls (pure Rust TLS) and SQLite is bundled ‚Äî no system-lib dep.

`install.sh` (repo root, POSIX sh) ‚Äî the `curl|sh` installer: detects OS/arch ‚Üí release target triple, downloads the `.tar.gz` + `.sha256` from GitHub Releases, verifies checksum, extracts to `~/.local/bin`. Options: `--to DIR`, `--force`, `--dry-run`.
