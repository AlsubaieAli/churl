## 2026-07-11 (R1.5 A2 — picker consolidated into a data-carrying `enum Picker`)

- **The one shared picker overlay is now a data-carrying `enum Picker` — PR 3 of the A2 fold, closing audit H3** (R1.5 A2, audit F1/H3). `picker: Option<PickerState>` + the three parallel item `Vec`s (`profile_choices: Vec<Option<String>>`, `workspace_choices: Vec<PathBuf>`, `sequence_choices: Vec<PathBuf>`) + `search_targets: Vec<(usize,usize)>` + `palette_actions: Vec<Action>` + the three one-shot `bool`s (`load_runner_after_pick`, `sequence_pick_runs`, `auth_picker`) are **all deleted** and replaced by `picker: Option<Picker>`. Each variant carries its own items + one-shot intent ALONGSIDE the shared `PickerState` finder: `Search { state, targets, after_pick }` / `Palette { state, actions }` / `Profile { state, profiles }` / `Workspace { state, paths }` / `Sequence { state, paths, runs }` / `Auth { state }`. The illegal states removed: the picker *kind* was previously inferred from `Mode` + which `Vec` was non-empty + which `bool` was set, and the selected index addressed whichever `Vec` matched **by convention** — a wrong-`Vec` pairing or stale index was an out-of-bounds hazard. Now the index can only address its own variant's list, and a stale intent can't leak across kinds (`(kind, wrong-items)` and `(kind, stale-flag)` are unrepresentable).
- **One-shot flags became variant fields, consumed in the same accept path** (R1.5 A2). `load_runner_after_pick` → `Picker::Search { after_pick }` (set by `open_load_runner_pick` right after `open_search`, so the intent can't outlive its picker); `sequence_pick_runs` → `Picker::Sequence { runs }` (deciding run-vs-edit on accept); `auth_picker` → the distinct `Picker::Auth` variant (matched, not a side-flag disambiguated against the profile/command palette). `close_overlay`'s five field-clears collapse to `self.picker = None` (dropping the picker drops every item `Vec` AND intent). `accept_overlay` `take`s the `Picker`, reads `state().current()`, resets the mode, then `match`es the owned variant — no `PickerMode` tag or `mem::take` of side fields; an empty-result Enter still drops the picker (and its intents) exactly as before.
- **`PickerState` stays the shared inner `state` field; `Mode::Search/Palette/WorkspacePicker/SequencePicker` stay as-is** (R1.5 A2, scope control). The finder/query/filter/selection remains one `PickerState`, reached uniformly via `picker_state()`/`picker_state_mut()` (render + overlay-key routing + tests), so the render path and key handling stay kind-agnostic. The four `Mode::*` picker variants are the routing/render axis (key routing at the `Search|Palette|WorkspacePicker|SequencePicker => handle_overlay_key` arm; the render overlay) and are **NOT** folded into `Picker` in this PR — that would double-model the same axis; `Picker` consolidates the picker's own item/intent state only. (`Mode::Palette` still backs both the command palette AND the profile/auth pickers — those are now `Picker` variants, so the accept path no longer disambiguates by an empty-`Vec` convention.) Test seams `picker_after_pick()`/`picker_sequence_runs()`/`picker_profiles()`/`picker_workspaces()` (`#[cfg(test)]`) replace the deleted field reads. Behaviour-preserving: `cargo test --all` **866/866 per-binary identical**, all **95 `tui_snapshot` snapshots byte-identical** (never `INSTA_UPDATE`). Remaining A2 folds (the hidden `help_open`/`leader`/`url_popup` overlay axis) are deferred to later R1.5 PRs.

## 2026-07-11 (R1.5 A2 — sequence state folded into `Mode::Sequence { view, editor, runner }`)

- **The unified sequence surface's three parallel `App` fields are folded into the `Mode::Sequence` variant — PR 2 of the A2 fold, extending PR 1's env_editor/load_runner pattern** (R1.5 A2, audit F1/H2/H3). `Mode::Sequence` (payload-free) → `Mode::Sequence { view: SeqView, editor: Option<SequenceEditorState>, runner: Option<SequenceRunnerState> }`; the `sequence_runner`/`sequence_editor`/`sequence_view` fields on `App` are **deleted** (the `sequence_abort: Option<AbortHandle>` field STAYS — it is not the surface's state, it must survive a Run→Edit face flip and is aborted on cancel/re-run/close). `editor`/`runner` remain `Option` **inside** the variant on purpose: both faces are legitimately optional (a `<leader>s r` run opens the Run face with no editor built yet; an edit session never allocates a runner until the first run). The illegal state the fold removes is "`sequence_editor: Some` while `mode != Sequence`" — now unrepresentable. Opening the surface is a single transition (`self.mode = Mode::Sequence { .. }`); close (`self.mode = Mode::Normal`) drops the whole payload, so `close_sequence_surface`'s three field-clears collapse to nothing (the generation bump still runs first, while the runner is live). The `SeqView::Run` `is_none()→close` guard and its `.expect("checked above")` collapse into one `sequence_runner_mut()` accessor; the delete-confirm's defensive "drop a stale editor/runner pointing at this file" block is **removed** (it ran in `Mode::Confirm`, where the sequence payload is definitionally absent — unreachable by construction).
- **Mode-aware accessors resolve the folded sequence state incl. the body-search park; `SeqView` stays `Copy`/`PartialEq`** (R1.5 A2). Six accessors mirror PR 1's `load_runner*`: `sequence_runner()`/`sequence_runner_mut()`/`sequence_runner_in_mut(mode, parked)` (the associated-fn form lets `active_response_mut`/`_geometry_mut` borrow the runner disjoint from `orphan_response`), `sequence_editor()`/`sequence_editor_mut()`, and `sequence_view() -> Option<SeqView>` (returns the `Copy` face by value, dropping the `self.mode` borrow so callers can act via `&mut self`). Only `Mode` is non-`PartialEq`; `SeqView` keeps `Copy`/`PartialEq`, so face comparisons (`view == SeqView::Run`) are unchanged — only reads move from `self.sequence_view` to `self.sequence_view().unwrap()` / the arm's `view`. Face/state mutation writes back through the arm (`if let Mode::Sequence { view, runner, .. } = &mut self.mode`).
- **Body-search over the sequence Response region reuses PR 1's `mem::replace` park verbatim — NO new parking machinery** (R1.5 A2, reviewer-attention item). Body-search can open over the Run-face Response region (tests `sequence_response_search_and_copy` / `sequence_response_search_targets_runner`). PR 1's `body_search_return = mem::replace(&mut self.mode, Mode::BodySearch)` already MOVES the *whole* current mode — so a `Mode::Sequence { runner: Some(..), .. }` parks and restores with zero new code. Only the mode-aware resolvers were extended to also match the parked sequence: `sequence_runner()`/`sequence_runner_in_mut` consult `body_search_return` while `Mode::BodySearch` (mirroring `load_runner()`), and `active_response_surface` reads `{ view: SeqView::Run, runner: Some(runner), .. }` out of the effective (live-or-parked) mode. `on_sequence_step` needed one reorder: its stale-generation gate now runs through the immutable `sequence_runner()` (borrow dropped) before the `self.sequence_abort = None` field write, because a live `sequence_runner_mut()` borrow now aliases `self`. Behaviour-preserving: `cargo test --all` **866/866 per-binary identical** (PR 1's baseline count), all **95 `tui_snapshot` snapshots byte-identical** (never `INSTA_UPDATE`). Remaining A2 folds (picker, hidden overlay axis) are deferred to later R1.5 PRs.

## 2026-07-11 (R1.5 A2 — mode-owned state folded into `Mode` variants)

- **Each modal overlay's owned state now lives IN its `Mode` variant, not in a parallel `Option` field — illegal `(Mode::X, None)` states removed** (R1.5 A2, audit F1/H2/H3, incremental PR 1 of the fold: `env_editor` + `load_runner`). `Mode::EnvEditor` → `Mode::EnvEditor(EnvEditorState)` and `Mode::LoadRunner` → `Mode::LoadRunner(LoadRunnerState)`; the `env_editor: Option<EnvEditorState>` and `load_runner: Option<LoadRunnerState>` fields on `App` are **deleted**, so "active editor/runner with no data" is unconstructable. Opening a mode is now a **single transition** (`self.mode = Mode::EnvEditor(state)`), and close (`self.mode = Mode::Normal`) drops the state automatically. The defensive `is_none() → Mode::Normal` guards at each modal entry and the two `.expect("checked above")` panics they protected are **gone** — unreachable by construction. The handlers reach the state through `self.mode` (`if let Mode::EnvEditor(state) = &mut self.mode`), producing the component's owned `Outcome` **inside** a scoped `&mut self.mode` borrow that is dropped before any `&mut self` follow-up (`notify`/summary-write) runs — preserving the existing "borrow → Outcome → drop borrow → act" shape.
- **`Mode` is now non-`Copy` and non-`PartialEq`/`Eq`; comparisons migrated to `matches!`** (R1.5 A2, the mechanical cost). The added payloads (`EnvEditorState` is `Clone`-only, `LoadRunnerState` is neither `Copy` nor `Clone`) make `Mode` non-`Copy`; we deliberately did **NOT** derive `PartialEq` on the states (a discriminant-only `PartialEq` would silently equate two different runner states), so every `self.mode == Mode::X` / `!= Mode::X` site became `matches!(self.mode, Mode::X)` / `!matches!(...)`, and every place that *copied* the mode (`let m = self.mode;`, the top-level key-router `match self.mode`, render's `overlay_mode`) now matches by **reference** and dispatches on an owned lightweight tag (small `Copy` `PromptPurpose`/`ConfirmPurpose` are dereferenced out before the `&mut self` call; the payload arms re-borrow the state out of the mode inside their block). ~11 production comparison/copy sites + ~110 test-assertion sites migrated; assignments (`self.mode = Mode::Normal`) needed no change.
- **`body_search_return` parks the whole mode (payload included) via `mem::replace` — the one subtle behaviour-preserving carry-over** (R1.5 A2, reviewer-attention item). Body-search can open OVER a load-runner Response region; previously the runner state persisted in its own field while `self.mode` flipped to `Mode::BodySearch`. Now that the runner state lives IN the mode, flipping to `BodySearch` would **drop** it. Fix: `body_search_return: Mode` (still typed `Mode`) is loaded via `std::mem::replace(&mut self.mode, Mode::BodySearch)` — it now **owns** the parked `Mode::LoadRunner(state)` for the overlay's lifetime, and restore is `self.mode = mem::replace(&mut self.body_search_return, Mode::Normal)`. `load_runner()`/`active_response_surface`/render consult `body_search_return` while `Mode::BodySearch` is active so the runner surface stays reachable through a `/` search — observable behaviour identical (the only change is the runner state is now dropped exactly when `Mode::Normal` is set, not earlier and not via a separate field). Behaviour-preserving throughout: `cargo test --all` 865/865 per-binary identical, all 95 `tui_snapshot` snapshots byte-identical. Sequence + picker folds are deferred to later R1.5 PRs.

## 2026-07-11 (A1 — semver hardening: `#[non_exhaustive]` policy)

- **`#[non_exhaustive]` on churl-core's error enums and growable-data structs — NOT on behavioral domain enums** (audit finding A1). Two-bucket policy for the published `churl-core` crate: (1) **APPLY** to every public `thiserror` error type — `ExtractError`, `SequenceError`, `ConfigError`, `HttpError`, `HistoryError`, `PersistenceError`, `ImportError`, `InterchangeError`, and the `ParseMethodError` tuple struct — plus the growable returned-data struct `LoadStats`. These grow additively (new error variants, new stats fields) and downstream match-with-catch-all / field-addition is idiomatic and non-breaking, so `#[non_exhaustive]` lets the library add to them without a semver-major bump. (2) **DO NOT APPLY** to the behavioral domain enums the TUI matches exhaustively for correctness — `Method`, `Auth`, `BodyKind`, `ApiKeyPlacement`, `OnError`, `LoadCheck`, `ReqOutcome`, `StepResult`, `KeyEntry`. For these, exhaustive matching in the consumer is a **safety feature**: adding `Method::Connect` should force a compile error at every curl-export / render / dispatch site rather than silently falling into a wildcard arm. The pre-1.0 semver convenience does not outweigh losing that compiler-enforced completeness. Default-to-skip for any type whose classification is ambiguous. Behaviour-preserving: `#[non_exhaustive]` never restricts the defining crate, and the `churl` binary needed **zero** new wildcard arms — its only cross-crate error match (`PersistenceError`, two sites) already carried a catch-all, and it builds `LoadStats` via `::default()` only (no cross-crate struct literal), so nothing downstream broke. Other public data structs (`Config`, `Response`, `Timing`, history/interchange summaries, etc.) were deliberately left annotation-free this round: they are consumer-constructible or not-yet-shown-to-grow, and the default is to skip absent a concrete additive-growth need. Test count unchanged (865/865, per-binary identical).

## 2026-07-11 (M7.11 — modularization: module boundaries)

- **Extractions target a *child* module of the owning type's module — never a sibling** (M7.11, the module-boundary rule) — Rust privacy is "visible to the defining module **and its descendants**", so a helper moved into a **child** of `app` (e.g. `app/render.rs`, `app/pure.rs`) keeps full access to `App`'s 50 **private** fields and every private method with **zero visibility changes**. The trap (and the one adversarial-review question for every modularization PR — "is this a pure move?"): extracting to a *sibling* (a top-level `tui/render.rs`) would force `pub(crate)` onto ~36 `App` fields, which is a behaviour/visibility change, not a pure move. Rule for the whole M7.11 tail: **every extraction lands in a child module of the type's own module**; if a move seems to *need* a `pub(crate)` field, it was extracted to the wrong place. No `macro_rules!` anywhere in the tree, so there are no macro-hygiene hazards to the move.
- **The god-file `app.rs` becomes the directory module `app/` — `mod.rs` is the spine** (M7.11 PR 2) — `crates/churl/src/tui/app.rs` → `crates/churl/src/tui/app/mod.rs` (a `git mv`, so rename detection keeps the diff reviewable; `tui.rs`'s `pub mod app;` and the existing `#[cfg(test)] mod tests;` sibling both resolve unchanged). The foundation layer split off first: **`app/render.rs`** holds the render layer (`render` + `leader_popup`/`render_collapsed_stub`/`prompt_hint`/`confirm_text`) and **`app/pure.rs`** the `self`-free helpers (`split_query`/`percent_decode`/`merge_query_params`, the `auth_field_*`/`toggle_auth_placement` Auth-tab helpers, the `default_export_path`/`export_target` export-path helpers). Both children take `use super::*;` to inherit the spine's imports and private types. `render` is re-exported (`pub use render::render;`) so the public path `churl::tui::app::render` — used by the `tui_snapshot` integration test — is unchanged; in-crate call sites and `tests`' `use super::*;` resolve every moved helper unqualified via `use` re-imports in `mod.rs`. Strictly behaviour-preserving: identical `cargo test --all` count (865/865, per-binary identical), the 95 `tui_snapshot` snapshots byte-identical (a snapshot diff would have meant the move wasn't pure), zero `pub(crate)` widening.

## 2026-07-11 (R1 — durability & concurrency)

- **Reserved-name disambiguation, not rejection** — a create/rename whose slug equals a reserved churl stem is *bumped* (`churl` → `churl-2.toml`) exactly like a filename clash, rather than erroring or overwriting. Two reserved sets, split by kind: `RESERVED_FILE_STEMS = [churl, folder]` guards endpoint/sequence **files** (a `churl.toml` endpoint parses as the workspace manifest and the note silently vanishes; a `folder.toml` endpoint parses as collection metadata); `RESERVED_DIR_NAMES = [sequences]` guards collection **directories** (the `sequences/` dir is excluded from `collections()`, so a collection named `sequences` would vanish from the tree). A collection named `churl`/`folder` is deliberately NOT bumped — those are files, not dirs, so no collision. The display `name` in the file is untouched; the disambiguated on-disk stem is surfaced in the TUI create/rename confirmation message (no silent no-op). The reserved sets are kept in sync with `MANIFEST_FILENAME`/`FOLDER_FILENAME`/`SEQUENCES_DIRNAME` by a unit test. `create_collection` still errors `AlreadyExists` on a genuinely-existing dir (import reuse relies on it) — reserved-slug bumping is a separate concern layered before that check.
- **SQLite: WAL + busy_timeout(5000ms) + BEGIN IMMEDIATE migration lock** — `HistoryStore::open` sets `busy_timeout=5000` (const `BUSY_TIMEOUT_MS`) first, then `journal_mode=WAL`, and fails loud (`HistoryError::WalUnavailable { actual }`) if the filesystem silently fell back to another mode (asserted for file-backed DBs only; an in-memory DB legitimately reports `memory`). The whole migration loop runs inside ONE `BEGIN IMMEDIATE` transaction: the write lock is taken up front, `user_version` is re-read *inside* the lock, only still-pending migrations apply, `user_version` is bumped, and it commits once — all-or-nothing. A concurrent opener blocks on the lock (up to `busy_timeout`), then re-reads the now-current version and no-ops. Refines "SQLite migrations: `PRAGMA user_version`" (2026-07-05): still no migrations table, but the per-migration transactions become one lock-guarded run.
- **Element-wise array-of-tables merge (supersedes the "unequal → replaced" clause)** — the format-preserving save's `merge_items` now merges `[[array-of-tables]]` element-wise *regardless of length* (`merge_arrays_of_tables`): recurse into the overlapping prefix (each survivor keeps its decor), append new tables from the fresh doc, truncate removed trailing ones. Replaces the old wholesale `*old = new.clone()` on any length change, which dropped every `# comment`/whitespace/order on surviving siblings. This is what makes the module's format-preserving promise hold for e.g. a hand-commented `[[request.headers]]` array across an add/remove. Supersedes the "equal-length … merged element-wise (unequal → replaced, comments lost)" clause in the 2026-07-05 "Persistence: serde reads, hand-rolled merge writes" decision.
- **Three unbounded-growth bounds (config-free consts)** — bounded memory/disk over an arbitrarily long session: (1) **App channel** `mpsc::channel(1024)` (const `APP_CHANNEL_CAPACITY`) replaces `unbounded_channel` — all four `AppMsg` senders are spawned async tasks (`.send().await` = natural backpressure) and the highlight worker is a dedicated OS thread (`blocking_send`), so no sender is on the render/UI thread where an await would stall input. (2) **History pruning** caps `history` and `load_batches` at the newest `HISTORY_ROW_CAP`=10 000 rows each, prune-on-insert via an indexed `DELETE … WHERE id <= max_id - CAP` (no per-insert full-table subquery); `recent()` reads are untouched. (3) **Highlight cache** is evicted from the previously-active buffer on every buffer switch (`App::set_active_buffer`), since only the active buffer enqueues highlight jobs — bounding total cached lines to (active-buffers × 64) instead of (all-buffers × 64). Open-buffer LRU capping was intentionally NOT added (buffers are user-opened and bounded by intent; the three growth vectors above are the actual unbounded ones).

## 2026-07-05

- **Name: churl** — "curl" in a TUI shell; short, memorable, not taken on crates.io at decision time.
- **Default branch: master** — matches the host platform's default; no organisational convention overrides it for this personal project.
- **HTTP client: reqwest 0.13 + rustls** — async-native, pure-Rust TLS, no OpenSSL dep. Coarse timing (connect + total) is sufficient for M0–M3; a custom hyper connector (DNS/connect/TLS split timing) is documented as the later upgrade path when fine-grained waterfall data is wanted.
- **TOML library: toml_edit** — format-preserving; users edit endpoint files by hand and expect their comments and ordering to survive a round-trip through churl. `toml` (serde-only) would silently destroy formatting.
- **File-per-endpoint + explicit `seq` field** — learned from Bruno: monolithic collection files (Slumber, Insomnia) create merge conflicts and make git history noisy. One file per endpoint scales to large collections; `seq` provides stable ordering without relying on filesystem order.
- **Templating: hand-rolled `{{var}}` substitution** — no JS runtime (ATAC carries a ~10 MB binary bloat for Deno); no Tera/Handlebars (overkill for URL/header/body variable injection). Precedence: CLI `--var` flag → named profile → process environment. Secrets are process-env only; never written to synced files.
- **Secrets hygiene: never in workspace files** — workspace = a git repo users commit and sync. Secrets go in process env or a local `.env` file the user adds to their `.gitignore`. churl never reads, writes, or suggests committing secrets.
- **Local state: rusqlite (bundled)** — history, cookies, and cached sessions are high-churn and relational. `sled` (considered) was abandoned upstream. SQLite with bundled compilation avoids a system dep and is universally available. Schema migrations run at startup.
- **Syntax highlighting: syntect + two-face, lazy, off-thread, viewport-only** — ~23 ms per viewport is the measured cold-start risk; background thread + viewport hash cache keeps the render loop smooth. Whole-document highlighting of a 1 MB response is never done.
- **Vim-modal editing: edtui** — `tui-textarea` has been unmaintained since 2024; `edtui` is actively developed and ships modal vim bindings out of the box.
- **Keybindings: crokey + semantic Key→Action map** — follows gitui/helix model; users remap actions, not raw keys. Prevents keybinding spaghetti as the feature set grows.
- **Fuzzy search: nucleo** — same engine as Helix; outperforms skim/fuzzy-matcher at scale; async-native.
- **HTTP mocking in tests: wiremock** — spins a real HTTP server in-process; tests exercise the full reqwest stack without network access.
- **Snapshot tests: insta + ratatui TestBackend** — deterministic pixel-level assertions for TUI rendering; catch accidental layout regressions.
- **No HTTP-semantic caching** — churl is a development/testing tool. Returning a cached response when the user hits "send" is a footgun. History (SQLite) covers the recall use case.
- **Async runtime: tokio** (arrives M2) — not added in M0. Plain synchronous `crossterm::event::read()` is sufficient for the placeholder TUI; the full `tokio::select!` event loop comes with request execution in M2/M3.
- **Persistence: serde reads, hand-rolled merge writes** — loads use `toml_edit::de` into serde types; saves serialize to a fresh document (normalized to `[[array-of-tables]]`) and merge it into the existing file's `DocumentMut`: unchanged values untouched, changed scalars keep the old decor, stale keys removed, equal-length arrays-of-tables merged element-wise (unequal → replaced, comments lost). Saving canonicalizes explicit `enabled = true` away.
- **SQLite migrations: `PRAGMA user_version`** — no migrations table; each migration runs in its own transaction that bumps `user_version`. Simple, idempotent across reopens, and rollback-safe per step.
- **Platform paths: `dirs` crate** — config at `<config_dir>/churl/config.toml`, state at `<data_dir>/churl/state.sqlite`; hand-rolling XDG-vs-macOS-vs-Windows resolution is not worth the bugs.
- **Secrets enforcement: name-marker heuristic at save time** — `save_workspace_manifest` refuses profile variables whose name contains a secret marker (token/secret/password/api_key/…) with a literal value; `{{...}}` template placeholders pass. Loads never refuse (it's the user's file); only churl-initiated writes are gated.
- **Fuzzy engine: `nucleo-matcher`, not `nucleo`** (refines "Fuzzy search: nucleo" above) — same scoring engine, synchronous API. The high-level `nucleo` crate's threaded streaming matcher is overkill for workspace-sized endpoint lists and would drag a worker-pool lifecycle into the TUI.
- **Key routing precedence** — (1) an open overlay (search/palette) consumes every key; (2) request pane focused with edtui in a non-Normal mode → all keys to edtui; (3) otherwise crokey `KeyMap` first, unmapped keys fall through to edtui when the request pane is focused, and mapped navigation keys forward their original event to edtui there. edtui owns insert/visual modality — the app never re-implements vim.
- **Keybinding overrides: flat `[keys]` string map in `churl-core::config`** — core carries only `"combination" = "action-name"` strings (keeps core TUI-free); the `churl` crate parses both sides at startup and fails loudly on unknown entries rather than silently skipping them. Overrides layer on top of the `key!`-built defaults.
- **Explorer laziness matches persistence laziness** — startup lists collection directories only; endpoint files parse on first expand or on search-overlay open, protecting the < 100 ms cold-start budget.
- **BackTab default binding built explicitly** — crossterm always reports Shift-Tab as `BackTab` + `SHIFT`, and crokey's `key!(backtab)` does not add the modifier (its string parser does); the default map constructs `KeyCombination::new(BackTab, SHIFT)` directly.
- **Task-level cancellation, not core-level** — `churl-core::http::execute` is a plain runtime-agnostic `async fn` with no `AbortHandle` in its signature. The TUI spawns it with `tokio::spawn` and keeps the `AbortHandle`; a monotonic `generation` counter on `App` tags each request, and a landed `AppMsg::Response` whose generation ≠ the current in-flight generation is dropped (stale result after cancel+resend). Ctrl-C cancels an in-flight request instead of quitting; `q`/`Esc` always quit. This keeps core free of any runtime dependency (refines the M0 "AbortHandle per in-flight request" wording, which lived in core in the original plan).
- **reqwest `rustls` feature, not `rustls-tls`** — reqwest 0.13 renamed the pure-rustls feature to `rustls` (with the connector selected via `ClientBuilder::tls_backend_rustls()`); the M0 decision's `rustls-tls` name was the 0.12 spelling. Same intent (pure-Rust TLS, no OpenSSL), current name.
- **No `{{var}}` templating in M3** — request execution uses the URL, headers, and body verbatim. Templating (the `{{var}}` precedence chain) is an M5 deliverable; wiring it into execution before the substitution engine exists would be dead code.
- **No response body-size cap in M3** — the full body bytes are read (`.bytes()`) and decoded lossily for the viewer; giant responses can bloat memory. Deferred to M6 (open question in MILESTONES) rather than guessed at now.
- **Response viewer: no wrap, truncate at pane width** — the virtualised viewer slices only the visible lines out of a line-offset index and lets ratatui truncate long lines at the pane edge. Line wrapping and horizontal scroll are deferred (open questions in MILESTONES); virtualisation (never materialising all lines) is what protects the 1 MB-body budget, and wrapping would complicate the offset math without buying that.
- **Viewport-stateless syntax highlighting** — the off-thread worker starts a fresh `HighlightLines` at the top of each viewport (highlighting only the visible lines, cached by a viewport hash of generation/scroll/height/syntax). This is a known imperfection: multi-line constructs (block comments/strings) that open above the viewport are mis-coloured. Accepted for the render-loop budget it buys; whole-document highlighting of a 1 MB response is never done. Foreground RGB only (background skipped so the theme doesn't fight the pane); theme is two-face's Nord.
- **History written on every terminal outcome** — success, failure, and cancel each insert a `NewHistoryEntry` synchronously in the event handler (a fast local SQLite write; acceptable on the UI thread). `status` is `Some(code)` only on success; `duration_ms` comes from response timing (`None` on cancel/failure). Insert failure warns on the statusline, never crashes; a failed history *open* disables history for the session (also non-fatal).
- **Body-size cap in M4** (supersedes "No response body-size cap in M3") — stream the response body with a default 10 MB cap, config-overridable (`max_body_bytes`); `Response` gains a `truncated` flag surfaced in the status line. Owner decision at the post-M3 plan review: it is a safety hole (accidental giant download freezes/OOMs the TUI), not polish, so it lands with M4's core work rather than waiting for release polish.
- **Auth is its own milestone (M5), not part of M4** — owner decision at the post-M3 plan review. M4 imports `-u`/`Authorization:` as plain headers; M5 adds the minimal auth model (basic/bearer/API-key) and remaps. Keeps M4 lean at the cost of one remap step. - **curl import flag policy: strict, lossless, no file reads** — any flag outside the supported set is a hard `UnknownFlag` error (fail loudly; silently dropping a flag imports a *different* request). `-F`/`--form` multipart and `@file` data payloads are `Unsupported` errors — churl never reads files during an import. `-u` lands as a plain `Authorization: Basic <base64>` header with a warning (M5 remaps to first-class auth). The URL's query string stays in the URL — import never explodes it into `Param`s (lossless and simple). `-L`/`-s`/`-v`/`-S` are accepted silently (no request semantics; redirects are already the client default); `--compressed`/`-k` accepted with "ignored" warnings; `-o` consumes and discards its value with a warning. Body kind derives from content + headers: `{`/`[` prefix or JSON Content-Type → json, `k=v&…` shape → form, else text; `--json` forces json, implies POST, and adds `Accept: application/json` when absent.
- **curl export: shlex-quoted single line, round-trip stable** — `export_curl` emits `curl [-X METHOD] -H '…' --data '…' <url+enabled params>` with every argument through `shlex::try_quote` (paste-safe, no line continuations); only enabled headers/params are emitted. `-X` is omitted for a body-less GET but kept for a GET *with* a body so `import_curl(export_curl(e))` reproduces the same method/URL/headers/body (import infers POST from a body). Enabled params are percent-encoded and appended to the URL query, matching import's query-stays-in-URL rule.
- **`ExecuteOptions` + streamed body cap** (implements "Body-size cap in M4") — `execute(client, request, &ExecuteOptions)` streams the body via `chunk()`, accumulating at most `max_body_bytes`; a chunk that would exceed the cap is cut at the boundary, `Response.truncated` is set, and the connection is dropped. An exact-cap body is *not* truncated. The status line appends `· truncated at <size>`; history is unchanged (it never stored body size). Caller resolves the knobs from config (`Config::max_body_bytes()`, `Config::timeout()` → `build_client(timeout)`).
- **Insert-mode interception resolves through the keymap** (implements "Insert-mode exception for Ctrl-S/Ctrl-C") — in the non-Normal-edtui pass-through, a key with the CONTROL modifier is looked up in the crokey keymap and dispatched iff it resolves to `Send` or `Quit` (the Quit path already downgrades Ctrl-C to cancel while a request is in flight); everything else still goes to edtui untouched. Lookup-not-hardcode honours user remaps; the CONTROL requirement guarantees no text-input key can be stolen. This remains the single exception to "edtui owns non-Normal modes".

Rationale at the time: zero native deps (`arboard` pulls `objc`/`x11`), works over SSH/tmux, matches lazygit's route; `base64` already a dep. **Why it was wrong**: a *raw* OSC 52 does not reach the system clipboard through tmux/screen without passthrough-wrapping, and macOS Terminal.app ignores OSC 52 entirely — yet the code reported success unconditionally, so copy-as-curl and response copy silently failed for tmux users. Payload cap (1 MB, `MAX_COPY_BYTES`, char-boundary) and the `copied first 1.0 MB of 4.2 MB` note carry over.
- **Clipboard: native (arboard) primary + OSC-52-with-passthrough fallback** (M7.5.3, reverses the M7-wave-1 decision above) — the owner needs copy to work across almost all terminals, multiplexers, **and** operating systems. `tui/clipboard::copy(payload, out) -> CopyOutcome` now tries the **native OS clipboard first** via `arboard` (macOS NSPasteboard, Windows, Linux **X11** — the only path that reliably reaches the system clipboard regardless of terminal; native Wayland is NOT enabled — arboard's `wayland-data-control` feature is off — so a pure-Wayland session uses the OSC 52 fallback, a follow-up if native Wayland is wanted), and only if that is unavailable/errors falls back to **OSC 52 wrapped for the active multiplexer**: tmux (`$TMUX`) → DCS passthrough `ESC P tmux ; <osc52-with-each-ESC-doubled> ESC \` (the payload's leading ESC doubled → two ESCs, not three); GNU screen (`$STY` or `$TERM` starts `screen`) → `ESC P <osc52> ESC \`; else raw. **Honesty**: the silent `let _ =` is gone — the run loop reports the real `CopyOutcome` (native/osc52 = "copied…", total failure = "copy failed"); the resolved-vars secret warning stays. `arboard = { version = "3.6", default-features = false }` on the **churl bin only** (never churl-core) — it was already a transitive dep via edtui, so it adds no new packages and its Linux backend (`x11rb` → pure-Rust `rustix`, no libxcb-dev) already compiled on ubuntu CI (no `ci.yml` change needed). Linux caveat: the X11 selection clears when the owning process exits; churl is a long-lived TUI so a copy persists for the session — acceptable, and we do **not** spawn a persistence daemon. **Testing seam**: native copy is never called from `cargo test` (headless CI has no display) — unit tests cover only the pure logic (OSC 52 framing, tmux/screen passthrough wrapping, multiplexer detection over injected env values, payload capping, `CopyOutcome`); the copy message is now built as a deferred *success* message and asserted via `App::pending_copy_message`, since the real copy runs in the terminal-owning run loop.
- **Wrap index built eagerly per render, highlighting deferred under wrap** (M7 wave 1) — soft-wrap expands the visible map into display rows with a `unicode-width`-aware greedy split (O(total lines), same order as the existing line-offset build; viewport rendering stays virtualised). **Fallback taken**: under wrap, response bodies render as *unhighlighted* plain text — slicing highlighted syntect spans at wrap boundaries (split spans by display width) was judged not worth the complexity for this milestone. Unwrapped mode keeps full off-thread highlighting. Recorded as a deviation from the pinned design's preferred path.
- **JSON fold regions scanned once per response by a string-aware scanner** (M7 wave 1) — `components::fold::scan_regions` does a single O(n) pass tracking in-string/escape state, pushing `{`/`[` openers onto a stack; a closer forms a region only with a *matching* opener kind (`}`↔`{`, `]`↔`[` — a mismatch drops the opener without a region) and only when opener and closer lines differ. Braces inside string literals are inert; unbalanced/truncated JSON never panics (unclosed openers at EOF yield no region). Scanned lazily on the first fold action, cached on the view. Only `SyntaxToken::Json` bodies fold (the single content-type mapping point stays authoritative — no per-format forks); non-JSON `o`/`O` no-op with a `folding: JSON responses only` notice (the Headers view of a JSON response gets its own `folding: body view only`).
- **Search feedback uses two mechanisms deliberately** (M7 wave 1, accepted deviation — the pinned design asked for one) — while a search is live the stats title carries `k/N matches` (persistent state, and it survives the message row being occupied by the `/query` input), and each `n`/`N` navigation additionally posts `match k/N` to the message row (transient event, confirming the jump just taken). The two answer different questions and neither position can cover both; keeping both is deliberate.
- **Case-insensitive search maps offsets through a per-char lowering table** (M7 wave 1 fix round) — `lowercase_with_offsets` builds the lowered haystack alongside one `(lowered_offset, original_offset)` entry per original char (+ sentinel); every lowered match position maps exactly back to original byte offsets (`map_lowered_offset`; offsets inside a multi-byte lowercase expansion round outward so the mapped range always covers the match). No length heuristics, no whole-line fallback — per-char length shifts whose totals cancel (e.g. `İ` 2→3 with `ẞ` 3→2 on one line) match correctly.

## 2026-07-07 (M7 review round 4 — vim motions in the edtui editors)

- **Missing vim motions implemented churl-side as cursor mutations on `EditorState`** (M7 review round 4) — edtui 0.11.3 lacks `W`, `B`, `f<char>`, `F<char>`, `t<char>`, `T<char>` entirely and binds first-non-blank only as `_` (not `^`). edtui's custom-keybinding register (`EditorEventHandler`) maps *fixed* key sequences, so it cannot express `f<any-char>` (a two-key motion whose second key is arbitrary). The alternative (forking edtui or extending its register upstream) was rejected as out of scope for a pure-TUI review round.
- **`f` shadows the global Jump key inside the Body editor** (M7 review round 4, extends the M6.6 overlay-shadowing precedent) — in Request/Body/Normal, `vim_ext::handle_key` runs before the leader/keymap steps, so `f` resolves to find-char inside the Body editor rather than the global Jump action. Precedent: M6.6's decision that `1`–`4` shadow pane-focus digits inside the Request pane. Jump-mode stays reachable from every other pane (guard test `f_from_explorer_still_enters_jump`), and Tab leaves the Body tab. The precedence also lets a pending find's *next* char reach `vim_ext` even when it is Space (leader) or a mapped key. `w` (Save) and `/` (endpoint search) are explicitly out of scope and keep their Body-tab meaning. Companion fix: `handle_url_popup_key` is now mode-aware so edtui's `/`-search can execute (it used to commit on any Enter regardless of edtui mode) — Search-mode keys pass wholesale to edtui; Enter commits only outside Search.

## 2026-07-08 (M7.1 — collection interchange)

- **JSON interchange hand-rolled against `serde_json::Value`, no Postman schema crate** (M7.1) — `churl_core::interchange` maps Postman Collection v2.1 to/from the churl model by walking untyped `Value`s. Rationale: real-world exports are messy (missing/renamed/extra fields, auth stored as arrays *or* objects, bare-string URLs), and a strict typed `#[derive(Deserialize)]` schema would reject valid files. Import is deliberately *liberal* — unknown methods default to GET with a warning, unsupported constructs degrade to warnings, never hard errors. `serde_json` is the one new dependency (workspace + churl-core).
- **Selectable export dialect: Postman v2.1 (round-trip) OR churl-native JSON** (M7.1, resolves the design-session open question) — `JsonDialect {Postman, Native}` chosen per export action. Postman v2.1 is the interoperable, lossy-but-round-tripping format (`import_postman_v21(export_postman(x))` reproduces method/url/headers/body/auth — round-trip test enforced); churl-native is a thin lossless envelope (`{ churl_version, name, collections: [{ name, endpoints: [Endpoint...] }] }`) reusing the model's serde derives, for full-fidelity backup/transfer between churl workspaces. No native round-trip importer yet (native import is not a stated requirement) — the native shape is for backup and is trivially re-parsable.
- **Nested Postman folders flatten into a single collection name on import** (M7.1) — churl collections are one directory deep (no nested collection dirs), so a request nested under `A > B` imports into a collection named `A / B` (the `folder_path` joined with `" / "`, then slugified by persistence). Root-level requests land in a collection named after the collection's `info.name`. Nested collection directories are a deliberate post-release backlog item — flattening keeps the imported layout navigable today without a model change.
- **Query stays in the URL on Postman import** (M7.1, mirrors the M4 curl-import decision) — `request.url.raw` is imported verbatim including its query string; import does **not** explode the query into `Param`s. Keeps import lossless and simple, and matches the existing curl-import behaviour (`query_string_stays_in_url`). Postman `url.query[]` (when present) is ignored in favour of `raw`.
- **Postman body-mode + auth-secret handling on import** (M7.1) — body modes: `raw` → Text/Json (by `options.raw.language == "json"`), `urlencoded` → Form (`k=v&…`, raw join — symmetric with export's raw split so round-trips are lossless), and `formdata`/`file`/`graphql` drop the body with a warning. Auth `basic`/`bearer`/`apikey` map to the churl first-class kinds; literal secret values are placeholder-ized (`{{password}}`/`{{token}}`/`{{api_key}}`) exactly like the M5 curl-import policy so `save_endpoint` never refuses them, and export refuses any endpoint still carrying a literal secret auth value (reuses `config::auth_secret_violations`). Collection-level `variable[]` is not imported (warned — define them as workspace/profile vars instead).
- **Import shared between CLI and TUI via `interchange::write_import`** (M7.1) — both `churl --import-collection <file>` (writes into the cwd workspace, prints a summary, then launches the TUI — fails loudly with a nonzero exit on a bad file rather than launching half-imported) and the in-TUI palette import call the one core writer, which creates one collection per flattened folder and writes each endpoint via `create_endpoint` + `save_endpoint`. The bin's `import_collection_into` is the thin testable seam over it.
- **`C` (shift-c) is the only new default keybind; interchange import/export stays palette-only** (M7.1) — copy-as-curl earns a Request-pane overlay binding (`C` is free globally and in the overlay; verified against the `?` help guard test, which renders it automatically); the resolved-vars copy variant and all import/export actions are rare and need a path/curl prompt anyway, so they live in the command palette only. Export writes are guarded to stay inside the workspace root via a lexical-normalize + `starts_with(canonical_root)` check (`export_target`); imports read any user-typed path (their own file). Copy-as-curl has two palette variants: default emits `{{var}}` verbatim, the "resolved vars" variant substitutes through the M6 resolver first (explicit secrets-caution opt-in that may place a real secret on the clipboard).

### M7.1 review-round fixes (2026-07-08, main-session adversarial review)

- **`write_import` bootstraps a `churl.toml` when the target dir has none** (M7.1 review, fix F1) — `churl --import-collection` into a bare directory previously wrote collection dirs + endpoint files but no manifest, so the launched TUI called `open_workspace` → `None` and showed an empty explorer (the import was invisible, contradicting the spec's "opens the TUI with the collection loaded"). `write_import` now writes a minimal manifest (`name` = `info.name`, or `imported`) when `churl.toml` is absent, and leaves an existing manifest untouched (in-TUI imports already have one). Covered by `write_import_bootstraps_manifest_in_bare_dir`.
- **Export path guard hardened against symlinked components** (M7.1 review, fix F2) — `export_target` did a purely lexical `..`/absolute check, which a symlinked directory *inside* the workspace (`exports -> /etc`) could tunnel through since the subsequent `write` follows links. It now also canonicalizes the deepest existing ancestor of the target and re-checks `starts_with(canonical_root)`, rejecting symlink escapes (`export_target_rejects_symlinked_component`, unix). The `..`/absolute vectors were already blocked.
- **Slug-collision on import warns instead of silently merging** (M7.1 review, fix F4/F5) — distinct flattened collection names that slugify to the same directory (`"A B"` vs `"a-b"`, a nested path colliding with a literal folder, empty folder names) were silently merged into one collection with no signal. `write_import` now detects when two distinct names resolve to the same dir and pushes a warning into the summary; the reported collection count is the number of distinct directories. Empty Postman folder names default to `folder` (previously only *absent* names did). Covered by `write_import_warns_on_collection_slug_collision`.
- **Known limitation — the secret gate covers only `auth.*`, not literal secrets in headers/URL/body** (M7.1 review, F3, deferred by design) — import placeholder-izes and export refuses only first-class auth secrets (`Basic.password`, `Bearer.token`, secret-named `ApiKey.value`), matching the M5 persistence policy (headers were never gated). A literal token living in an `Authorization` header or URL query is therefore imported to disk and exported to JSON verbatim. This is symmetric with existing behaviour and not a regression, but a shared JSON export is a new, highly shareable surface — extending secret detection to header values is a policy decision left to the owner rather than changed asymmetrically here. Export still refuses first-class auth secrets in both dialects.
- **Structured Postman `url` without `raw` imports empty, with a warning** (M7.1 review, F6) — a `url` object using only the `{host, path, query}` arrays (no `raw`) now imports with an empty URL *and* a warning, rather than silently. Real v2.1 exports include `raw`; this keeps the "liberal import" promise honest. Form-body round-trip (F7) remains lossy for values containing literal `&`/`=` (raw join/split, no percent-coding) — documented, low impact.

## 2026-07-08 (M7.3 — environments & vars editor + crash fix)

- **Manifest-in-collection crash fix + lenient endpoint load** (M7.3) — `Collection::endpoints()` skipped `folder.toml` but not `churl.toml`, so a nested-workspace layout (a collection dir that is itself a workspace root — e.g. `churl --import-collection` from the demo root) had its `churl.toml` parsed as an `Endpoint` (`missing field 'request'`), and the `?` propagated to abort the whole TUI load. Fix is two-layer: (1) `endpoint_files()` (shared by both load methods) now skips `MANIFEST_FILENAME` as well as `FOLDER_FILENAME`; (2) a new `Collection::endpoints_lenient() -> CollectionLoad { endpoints, warnings }` degrades a per-file parse failure to a `"skipped <file>: <reason>"` warning instead of aborting — only the directory `read_dir` itself stays a hard error (a vanished collection is a real failure, one bad file is not). `endpoints()` stays strict for callers that want it (existing `malformed_endpoint_error_names_the_file` test unchanged); the TUI explorer/search load paths use the lenient variant and the app drains `ExplorerState::take_warnings()` into the M6.7 message row after expand/search-open/reload (fail loudly, never silent). Chosen over "don't treat a workspace-subdir as a collection" because the lenient path also covers the general "one hand-corrupted endpoint must not nuke the workspace" robustness the milestone asked for.
- **Save-prune verified, no save-path change needed** (M7.3, the CRITICAL correctness gate) — the design flagged that an editor "delete" is broken if the format-preserving save only overlays. Investigation: `merge_tables` **already prunes** — it removes keys present on disk but absent from the saved struct (its first loop), and an unequal-length `[[profiles]]` array is wholly replaced (comments on survivors lost, the entry gone). So deleting a var key, deleting a profile, renaming a profile/var, and emptying a scope's `[vars]` all correctly disappear from disk. No production change; instead the gate is locked down by round-trip tests that re-read the file and assert the removal (`deleting_a_workspace_var_key_removes_it_from_disk`, `deleting_a_profile_removes_its_table_from_disk`, `renaming_a_profile_leaves_no_old_name_on_disk`, `renaming_a_var_key_leaves_no_old_name_on_disk`, `deleting_a_collection_var_key_removes_it_from_disk`, `deleting_the_last_var_removes_the_vars_table`), with surviving keys' comments asserted intact.
- **Editor is a self-contained modal in the `churl` crate; profile CRUD lives in the editor's working state, not core** (M7.3) — all new UI state (`EnvEditorState` and friends) lives in `components/env_editor.rs`; `churl-core` gained nothing UI. The editor holds each scope as an ordered `Vec<(String,String)>` (insertion order + in-place rename UX; the `BTreeMap` shape is only rebuilt at save). Profiles are created/renamed/deleted purely in that working state and persisted by rebuilding `Workspace { name, vars, profiles }` and calling `save_workspace_manifest` — no profile-CRUD helpers were added to core (the design's permitted-but-not-required option), keeping the resolver/persistence seams untouched (M9 guardrail).
- **Large-popover layout over a dedicated screen** (M7.3, owner's "own layout OR large popover" choice) — a near-full-screen (90%×90%) centered modal with a left scope column (grouped `WORKSPACE`/`COLLECTIONS`/`PROFILES`, active profile `●`) and a right var-row column, same visual family as the other overlays (thick border, `Clear`), degrading gracefully on tiny terminals (clamped column widths, zero-area guards). It grabs every key as a new `Mode::EnvEditor` routed to `handle_env_editor_key`, the same routing tier as Search/Palette/Picker. Trigger: `<leader>v` (`v` was free) + palette command; both go through one `Action::OpenEnvEditor`.
- **Discard guard is editor-local, not the global `guarded_load` machinery** (M7.3) — dirty is derived (`scopes != snapshot`, same pattern as the request-pane `loaded_snapshot`); closing while dirty sets `pending_close` and renders `prompt::render_confirm` (`s` save-then-close only if the save took, `d` discard, `esc` stay). This stays inside the editor rather than threading through `Mode::Confirm`/`pending_load` (that seam is endpoint-load specific), per the design.
- **Precedence display is computed relative to the selected scope's context** (M7.3, the headline feature) — the resolver's collection layer is per-endpoint, so a single global "winner" for a var name is ambiguous across collections. Resolution rule for the annotation: build the chain from cli vars, the **active** profile only, the **selected collection** (only when a collection scope is being viewed), and workspace, in Resolver precedence order (cli > profile > collection > workspace). A row is tagged ` ✓` when the selected scope is the highest-precedence definer, else ` → <winner>`; the footer shows the full chain (`base_url: profile dev — · workspace ✓`, where `—` means the active profile is shown but does not define the name). Rows of an **inactive** profile are tagged ` (inactive)` (that environment never resolves). This is precise and non-ambiguous for the collection-scope case (matches send-time semantics for that collection) and honest for the workspace/profile case (collections are context-dependent and deliberately not folded into those views). **Honesty refinement (2026-07-08 review, fix 2):** a **workspace** `✓` winner whose name is *also* set in ≥1 collection renders ` ✓*` (asterisk) with a footer legend `* also set in a collection — overridden there per-request`, because the collection layer (rank 2) outranks workspace (rank 3) for that collection's endpoints — a bare `✓` would overstate the win. The asterisk is workspace-only: a profile/cli winner outranks collections, so it stays a precise `✓`, and the collection-scope view is already exact.
- **Secrets gate reused verbatim; save validates before writing anything** (M7.3) — save rebuilds the manifest + each collection meta and runs `config::secret_violations` / `collection_secret_violations` up front; any literal secret-named var refuses the **whole** save (writes nothing, keeps the editor dirty, names the offenders), exactly the M5/M6 policy — just made legible in the editor footer. Consequence surfaced during testing: a workspace whose `churl.toml` was hand-edited to contain a secret literal cannot be saved from the editor at all (the manifest as a whole is refused), which is the existing persistence contract, not a new behaviour.
- **Active profile applied on save** (M7.3, minor deviation from the design's "on save/close") — `x` changes the editor's `active_profile` mirror for the precedence display; the app's real active profile is updated only when the edits are **saved** (a discard/close leaves it untouched). Active profile is session-only state, but coupling its persistence to save keeps "nothing takes effect without `w`" consistent and avoids applying a profile the user then discarded.
- **Editor modal keys are hardcoded (not `KeyMap`-remappable), like the other overlays** (M7.3, accepted deviation) — the design asked the footer hint to be built "from the actual bindings, not a stale string". The env-editor keys (`j/k/a/d/r/n/x/w/q`, edit-mode `enter/tab/esc`) are modal-internal, exactly like Search/Palette/MethodMenu keys, and are not in the user-remappable `KeyMap`. The footer hint is built live from the focused-pane / edit state (it changes with focus and mode) and is kept adjacent to the handler that implements the same keys as the single source of truth — but the keys themselves are fixed, matching every other overlay in the app rather than introducing a new remap surface for one modal.

- **`Esc` reverts an in-progress field edit** (fix 1) — the editor live-mirrors each keystroke into the working row (so the render + dirty state track typing), which meant `Esc` left the mirrored text behind instead of cancelling. `EnvFieldEdit` now snapshots the field's pre-edit value (`original`) in `begin_edit`; `cancel_edit` restores it (and still drops a fresh, never-committed empty row). The live preview stays; only the revert-on-cancel was missing.
- **Duplicate var names refuse the whole save** (fix 3) — a scope can transiently hold two rows with the same name (visible while editing); on save `rows_to_map` collapses them last-wins, which would silently drop a visible row. Save now runs a `duplicate_name_violation` check first (mirroring the secret gate): the first duplicate trimmed non-empty name refuses the entire save, writes nothing, and names the scope + var (`duplicate var name 'x' in Workspace`).
- **Active-profile selection folds into dirtiness** (fix 4) — `x` (set active profile) was invisible to `is_dirty`, so `x` then `q` closed with no discard prompt and silently dropped the selection. `EnvEditorState` now snapshots `active_profile` at open (`snapshot_active_profile`); `is_dirty` is also true when the active profile differs; the discard path restores it; `w` refreshes the snapshot after applying. The app still only mutates its real active profile on a successful save.
- **Known limitation — multi-file save is not atomic** (M7.3, accepted fast-follow) — save validates *everything* up front (duplicates + secrets across all scopes) so **no secret can leak and no partial write is silently wrong**, but the writes themselves are sequential (`churl.toml`, then each changed `folder.toml`); a mid-loop IO error can leave disk half-updated, and since the app's live-refresh (`app.workspace` re-open + explorer reload) runs only on `EnvSaveResult::Ok`, a partial failure leaves `app.workspace` stale relative to disk. It is reported loudly (`EnvSaveResult::Failed` names exactly which files were written and which failed; the editor stays open and dirty). Hardening to a temp-write+rename or reload-on-partial is deferred to a later pass — IO errors mid-save are rare and the failure is loud, not silent.

## 2026-07-08 (M7.4 — request sequences / API E2E testing)

- **Sequences persist as `sequences/<slug>.toml`, a reserved workspace dir (not a collection)** (M7.4) — one file per sequence, `seq`-ordered, mirroring the file-per-endpoint philosophy. `persistence::SEQUENCES_DIRNAME = "sequences"` is excluded from `OpenWorkspace::collections()` (same spirit as skipping `folder.toml`/`churl.toml`), so a sequences dir never surfaces as a fake collection (regression test `sequences_dir_is_not_a_collection`). `Sequence { seq, name, on_error, steps: Vec<SequenceStep> }` serializes steps as `[[step]]` with a nested `[step.extract]` map; `SequenceStep { seq, endpoint, extract }` where `endpoint` is a workspace-relative path. CRUD seams mirror the endpoint ones (`load/save/create/rename/delete_sequence`, slug + `next_seq`, empty-name refusal); no secrets gate is needed — a sequence file holds endpoint refs and extraction *expressions* (var names → expressions), never secret values. `OpenWorkspace::sequences() -> SequenceLoad { sequences, warnings }` uses the M7.3 lenient posture (one unparseable file degrades to a warning, never aborts).
- **Extraction is a documented, dependency-free subset over `serde_json` — no JSONPath crate** (M7.4) — `sequence::extract_value(&Response, expr)` supports exactly: `status` (numeric status as string), `header:<Name>` (case-insensitive header lookup, error if absent), and a JSON path `$.a.b[0].c` / `a.b[2]` (leading `$.` optional) parsed by a hand-rolled byte scanner (breaks only on ASCII `.`/`[`/`]`, so unicode keys are safe and it never panics). Leaf coercion: JSON string → its value, number/bool → display, object/array → compact JSON; a `null` leaf, missing key, out-of-range index, type mismatch, non-JSON body, or malformed expression are each a distinct `ExtractError`. Rationale: a JSONPath crate is a heavyweight dep for a testing tool's needs; the subset covers the real chaining cases (tokens, ids, headers) and stays inside the "no new heavyweight deps" rule. Any single extraction rule failing fails the whole step (subject to `on_error`).
- **The run engine lives in `churl-core`; the resolver stays the single seam via a prepended "extracted" scope** (M7.4, the M9 plugin guardrail) — `sequence::prepare_step` builds the resolver in canonical order with one extra scope **prepended at highest precedence**: `extracted > cli > profile > collection > workspace` (then process env). Resolution is never forked — it is the same `Resolver`/`substitute_request` seam every other send uses, with one ephemeral scope. **Extracted wins** so a chained value is never shadowed by a same-named ambient var (wiremock-proven by `extracted_var_beats_workspace_var`); documented precedence choice. The collection scope is loaded fresh per step from *that step's* endpoint's collection, so a multi-collection sequence resolves each step against its own collection vars.
- **Shared run seams — what's identical vs what the TUI mirrors** (M7.4; wording corrected in the fix round) — the core exposes three shared seams both `run_sequence` and the live TUI driver call: `classify_response(&Response, &SequenceStep) -> (StepResult, extracted)` (status ≥ 400 → `Failed`; else run extraction → `Ok`/`ExtractError`), `should_halt(&StepResult, OnError) -> bool` (the halt/continue decision), and `prepare_step`/`extract_step`/`ordered_steps`. **What the seam does *not* cover, and the TUI therefore mirrors, is exactly two trivial lines**: the transport-error → `StepResult::HttpError(string)` mapping (the sequence `AppMsg` carries `Result<Response, String>`, the error stringified at the task boundary like the normal `AppMsg::Response`, so the TUI can't hand core an `HttpError` value) and the skipped-tail marking on halt. `run_sequence`'s convenience `classify_step(&Result<Response, HttpError>, …)` wraps `classify_response` for the value-typed error it *does* have. Drift is guarded by the cross-check test `sequence_transition_matches_core`: four scripted scenarios (all-ok+extract / halt-on-500 / continue-past-500 / extract-error-halts) run through the real `run_sequence` (wiremock HTTP) **and** the App's `on_sequence_step` driver (injected outcomes) and must produce the same per-step `StepResult` sequence + same extracted map. Earlier wording ("one classify seam") overstated the sharing — corrected here.
- **Step endpoints cannot escape the workspace root — lexical *and* canonical** (M7.4; hardened in the fix round) — `resolve_step_path` rejects `..`/absolute/drive-prefix components lexically (fast pre-check), **then** canonicalizes the deepest existing ancestor of the resolved path (following symlinks) and requires it to stay under the canonical workspace root. Batch/sequence history is an M7.5 concern (concurrent-runs milestone already owns "record batch runs without flooding per-endpoint history").

## 2026-07-08 (M7.5 — concurrent requests / load testing) — closes the M7.1–M7.5 block

- **Bounded concurrency via `futures`' `buffer_unordered`, not a manual `FuturesUnordered` + `Semaphore`** (M7.5, deviation from the design's literal wording) — the design specified "a `FuturesUnordered` bounded by a `tokio::sync::Semaphore`". A semaphore-guarded `FuturesUnordered` needs the launch loop to *also* drain the set while waiting for a permit (otherwise the permit-acquire await deadlocks, since permits only free as in-flight futures complete). `futures::stream::iter(0..total).map(execute-copy).buffer_unordered(concurrency)` gives the **identical** guarantee — at most `concurrency` request futures are polled at once — with no manual permit bookkeeping and no deadlock foot-gun, and it *is* a `FuturesUnordered` internally. The non-negotiable ("bounded concurrency must actually bound") is met and **proven** by a wiremock test that records each request's arrival timestamp and computes the peak count of overlapping `[arrival, arrival+hold]` intervals, asserting it never exceeds the cap (and is ≥ 2, so the bound is meaningfully exercised). Both `run_load` and the live TUI launcher use this same construct, so they cannot drift.
- **Cancel = abort the ONE launcher task; no detached per-request `tokio::spawn`** (M7.5, the batch-cancel non-negotiable) — the TUI fires the whole batch from a *single* `tokio::spawn`ed launcher that owns the `buffer_unordered` stream; its `AbortHandle` is `App::load_abort`. Aborting it drops the stream, which drops every in-flight reqwest future (reqwest cancels on drop) — that is the entire batch-cancel mechanism. There is deliberately no per-request spawn that could outlive the abort. Proven live by `load_cancel_aborts_the_batch_live` (multi-thread wiremock, concurrency 2, slow server): a cancel right after launch means the server sees only a handful of the 20 copies — the un-launched ones never fire. A per-run `run_generation` (bumped on cancel/re-run/close) additionally drops any straggler result that lands after a cancel.
- **Absolute-target pacing (not sleep-since-previous)** (M7.5) — `interval > 0` paces launches by making copy `i` sleep until `start + i·interval` before calling `execute`. Because `buffer_unordered` polls futures lazily (a copy is only polled once a concurrency slot frees), a naïve "sleep `interval` from when polled" would compound into a super-linear cumulative delay under low concurrency. The absolute target is a hard floor: copy `i` never launches before `i·interval`, so the run's wall-clock is `≥ (total-1)·interval` regardless of scheduling — which satisfies (and exceeds) the design's loose `≥ (ceil(total/concurrency)-1)·interval` assertion without flake. `interval == 0` is burst.
- **Load runs write ONE summary row to a SEPARATE `load_batches` table — structural, not query-filtered, non-flooding** (M7.5) — migration 3 adds `load_batches` (one row per completed/cancelled run: url, endpoint_path, total, concurrency, ok/fail/error counts, cancelled flag, min/median/p95/max ms). A load run **never** inserts into `history`, so the per-endpoint history view cannot see batch rows *by construction* — not by a query filter that a future change could forget (test `load_batches_never_appear_in_per_endpoint_history`). Mirrors the M7.4 decision that sequence steps write no history; M7.5 owns "record batch runs without flooding".
- **`cancelled` column added to `load_batches` (deviation from the design DDL)** — the design's DDL listed the count/percentile columns but not a cancelled marker, yet §1d/§2 both require persisting "a cancelled run's partial summary — mark it". A boolean `cancelled` column is the honest place to mark it (the partial counts are real, but the run did not complete). `LoadBatchSummary.cancelled` round-trips it.
- **Nearest-rank percentiles, over completed-request timings only** (M7.5) — `stats()` computes min/median/p95/max by the **nearest-rank** method (`rank = ceil(p/100 · n)`, 1-based) on a sorted copy of the timings of requests that actually completed (`Ok`/`Failed` carry a timing; `Error` does not); `median` is nearest-rank p50, mean is the arithmetic mean. Empty / all-errored input yields `None` for every latency field (counts still populate) — no divide-by-zero, no panic. Exact vectors are unit-tested (odd/even/single/empty/all-errored). Chosen over interpolated percentiles for simplicity and because nearest-rank is unambiguous and matches "the p95 is an actual observed latency".
- **Guardrail: hard-cap refuse + warn-cap loud confirm naming the URL, `[load]`-overridable** (M7.5) — `check_config(cfg, caps)` returns `Refuse` above a hard cap (`max_total` 10 000 / `max_concurrency` 200), `Warn` above a warn threshold (`warn_total` 100 / `warn_concurrency` 20), else `Ok`; refuse beats warn, and `total == 0` is not a breach (it is a harmless no-op). The runner refuses with a message (no run), or renders a loud confirm (`prompt::render_confirm`) that names the count, concurrency, and **target URL** (`Fire 500 requests at concurrency 5 against https://…?`) — `y` fires, `n`/esc aborts. Caps are overridable per-user via a `[load]` config table folded over the defaults. This makes "testing aid, not a load-cannon" real, not advisory.
- **`LoadStarted` launcher message so a running copy shows its glyph** (M7.5, minor enhancement over the design's message set) — the design's live message was completion-only (`LoadResult`), which would leave every row jumping `Pending → terminal` with the `◐` running glyph never appearing live. The launcher now also sends `AppMsg::LoadStarted { run_generation, index }` the instant a copy enters the concurrency window (right before `execute`), so its row flips to `Running` honestly. One extra bounded message per request; generation-guarded like `LoadResult`.
- **Request resolved ONCE at open, cloned for all N copies** (M7.5) — opening the runner builds the request exactly like an interactive send (endpoint clone + fold in the live body editor + `build_resolver`/`substitute_request`) and stores the resolved `Request` on `App::load_request`; every copy in every (re-)run clones *that* — no per-copy re-resolution, so the whole batch is guaranteed identical (same URL/vars/auth). `execute` stays substitution-free.
- **`tokio` + `futures` promoted to `churl-core` direct dependencies** (M7.5) — `run_load` needs `buffer_unordered` (`futures`) and, for pacing, `tokio::time::sleep`; both were already transitive (reqwest pulls tokio; `futures` was a `churl` dep). They are used *only* inside `run_load`'s body — `execute` keeps its runtime-agnostic signature (no tokio types), so the "core is UI-free and `execute` is runtime-agnostic" architecture invariant is preserved. Recorded because it moves two crates from dev-deps / transitive to direct core deps.
- **Runner state is UI-only; `App` owns execution — mirrors the M7.4 split** (M7.5) — `components/load_runner.rs` holds only display state (config, result rows, stats, focus, the config-field `LineEditor`) + the key state machine; `App` owns the client, the launcher + abort handle, the generation guard, the caps, and the resolved request, and drives the run through the shared `churl_core::load` primitives. Under snapshot-test construction (`client == None`) `start_load_run` resets the rows and returns without spawning, so the state machine is exercised by injecting `on_load_started`/`on_load_result`, and render snapshots are deterministic (no highlight worker). Modal keys are hardcoded (not `KeyMap`-remappable), like every other overlay; only the entry `Action::OpenLoadRunner` (`<leader>l` + palette) is in the keymap.
- **Results list renders O(viewport), not O(N)** (M7.5) — the list keeps a scroll offset and builds only the visible slice of rows each frame, so a 10 000-copy batch renders as cheaply as a 10-copy one. Stats are recomputed from the accumulated outcomes as each result lands (network-paced, so the recompute cost is amortised across the run).

## 2026-07-09 (PR 3a — buffer refactor, Stage 1 of tabs/buffers)

- **Per-endpoint response/scroll/in-flight; loading a new endpoint shows its OWN (Idle) response, not the previous endpoint's stale one** (PR 3a — deliberate, behaviour-changing deviation from master; accepted) — Stage 1 moves the ~15 flat per-endpoint `App` fields (editor, tabs, response + response_scroll/cursor/…, in_flight, highlight_cache/pending_highlight, url_editor/url_popup*, loaded_snapshot) INTO a `Buffer`/`EndpointBuffer` owned per loaded endpoint (`selected: Option<SelectedEndpoint>` → `buffers: Vec<Buffer>` + `active`). ## 2026-07-09 (R0 — cheap-P0 durability)

- **Atomic saves via one `persistence::atomic_write` helper** (R0) — every workspace writer (`save_endpoint`/`save_sequence`/`save_workspace_manifest`/`save_collection_meta`) funnels through `save_value`, whose sole `std::fs::write(path, output)` truncated the source-of-truth file in place — a crash or power loss mid-write destroyed the file (the sweep's second catastrophic P0). Replaced that one call with `atomic_write(path, bytes)`: write to a **sibling temp file in the same directory** (`.<file_name>.<pid>.<counter>.tmp` — a per-process `AtomicU64` counter keeps concurrent writers from colliding on one temp name; same filesystem so `rename` is atomic) → `write_all` → `sync_all` (fsync data) → `std::fs::rename` (atomic replace) → **fsync the parent directory** so the rename itself survives power loss. Directory fsync **degrades gracefully** — opening a dir for fsync is unsupported on some platforms (notably Windows), so a failure there is ignored rather than failing an otherwise-good save. On any error the temp file is removed best-effort and the original error is returned. **Error contract unchanged**: `save_value` still maps the helper's `io::Error` to `PersistenceError::Write { path, source }`, so every caller and test is untouched. The format-preserving merge (comments/ordering) is unaffected — it produces the same bytes, they just land durably. Tested by injecting a temp-create failure (read-only parent dir, unix) and asserting the pre-existing good file is byte-intact with no stray temp left; the byte-identical round-trip corpus still passes.

- **Load-runner response retention bounded to O(concurrency + K), dropped rows are not reconstructable** (R0) — `on_load_result` builds a `ResponseView` per OK copy (the body decode/line-index) and stored `ResponseState::Done { view }` on **every** row, so a high-`total` run retained `total × body` → OOM (the sweep's first catastrophic P0). Fix keeps the bloat bounded in the UI-only `LoadRunnerState`: a `live_views: VecDeque<usize>` tracks (in completion order) which rows currently hold a live `Done` view; on each new retained view, `enforce_view_window` keeps at most **K = `LIVE_VIEW_WINDOW` = 16** rows live **plus the currently-selected row** (never evicted — checked *after* the follow-selection update so the just-selected newest row is protected), downgrading the oldest non-selected retained row to a new **`ResponseState::Dropped { status, timing, size }`**. This is the accepted load-test tradeoff (you cannot inspect every one of thousands of bodies; you get status/timing/size for all of them, full bodies for a sliding window + whatever you're looking at). Per-row status/timing and the `outcomes`-driven stats are untouched, so aggregate correctness is independent of eviction. Two new `ResponseView` accessors (`status()`, `body_len()`) feed the placeholder. When the selection moves off a pinned row it becomes evictable again but is **not** proactively rebuilt (it may already be dropped — fine). New enum variant means the two exhaustive `ResponseState` matches in `response.rs` (render dispatch + `collapsed_summary`) gained a `Dropped` arm; all other sites are `if let`/`matches!` on specific variants and needed no change.

## 2026-07-10 (M7.7 Stage A — response body reformatter)

- **Reformatter v1 scope is JSON-only, via `serde_json::to_string_pretty`** (M7.7 Stage A, owner decision) — the pinned M7.7 deliverable named JSON + HTML + XML, but v1 ships **JSON only**. JSON is the dominant API content-type and `serde_json` is already a workspace dependency, so pretty-printing adds **zero new dependencies** and no new code paths beyond a parse-then-reprint (`reformat_body_if_needed` in `components/response.rs`: `serde_json::from_str::<Value>` → `to_string_pretty`). HTML/XML reformat is deferred (a later M7.7 stage or follow-up); a structured HTML/XML pretty-printer would need a new dep and its own error model, out of scope for this stage. The single content-type→syntax mapping point (`SyntaxToken::from_content_type`) still decides whether to reformat, so no per-format fork was introduced — only `SyntaxToken::Json` bodies are reformatted.
- **Pretty-by-default for json-ish content-types; malformed JSON silently falls back to raw** (M7.7 Stage A) — a `application/json` (or any json-ish content-type per `from_content_type`) body arrives **pretty** (`ResponseView::pretty` defaults `true` for `SyntaxToken::Json`, `false` otherwise). This fixes the reported bug: petstore-style minified single-line JSON previously rendered on one line (the viewer line-split on server `\n` only and never re-indented). If the body is not parseable JSON — despite the content-type — `reformat_body_if_needed` returns the original text unchanged: **never errors, never panics, never shows a diagnostic** (a viewer must always render *something*). So `pretty` can be on while the displayed text equals the raw body (a body **truncated** at the size cap is unparseable, so it renders raw and keeps its truncation marker). The pretty output **preserves the server's on-the-wire key order** — `serde_json`'s `preserve_order` feature is enabled at the workspace level (owner-approved), so `to_string_pretty` re-emits object keys in insertion order rather than sorting them alphabetically (sorted keys would misrepresent the response structure the user is inspecting). Feature unification means churl-core picks it up too: this also changes `interchange.rs` Postman export from alphabetical to insertion order (`method → header → body → auth → url` — the `.insert()` sequence is the new canonical order); accepted and expected. No test pinned the old alphabetical byte layout, so no export snapshot needed updating.
- **Raw↔pretty toggle on `p`; toggling resets folds and invalidates the highlight cache** (M7.7 Stage A) — a new `Action::TogglePretty` is bound to `p` in the `[keys.response]` overlay (`p` was free there; the global `<leader>p` switch-profile lives behind the leader, not this pane overlay) and added to the command palette (context-free, like `ToggleWrap`/`ToggleHeadersView`). `ResponseView::toggle_pretty` flips `pretty`, rebuilds the displayed `text` + `line_offsets` from the raw body, **bumps `generation`** (the syntect viewport cache is keyed on generation, so the bump invalidates stale highlights on toggle) and additionally hashes `pretty` into `viewport_hash` (belt-and-braces), and **resets folds** (`folds = None`, `folded.clear()`) — fold openers are position-based line indices and the raw↔pretty layouts differ, so a carried-over fold set would point at the wrong lines. The app handler resets cursor/scroll and clears the highlight cache (same as fold-all). Outside a JSON body view the handler no-ops with a `pretty: JSON body only` notice rather than silently doing nothing.
- **Copy (`y`/`Y`) always reads the raw on-the-wire bytes, never the reformatted text** (M7.7 Stage A) — `ResponseView` keeps a separate `raw_text` field holding the exact lossy-UTF-8 decode of the response body, never reformatted. `copy_all` (`y`, copy body) returns `raw_text` in the Body view (the headers text in Headers view), so a copy is byte-exact regardless of the pretty toggle — pasting round-trips the server's actual bytes. The displayed `text` (pretty when applicable) drives only rendering, folds, wrap, search and line navigation. This is a correctness requirement: a "copy response" that emitted reformatted-and-key-sorted JSON would silently corrupt what the user believes they copied. `copy_line` (`Y`) copies the logical line under the cursor as displayed (the visible line the user is pointing at). **Side effect on the enum-size lint**: adding `raw_text: String` pushed `ResponseState::Done` past clippy's `large_enum_variant` threshold; boxing the hot per-render view (read every frame) would add an allocation + indirection and ripple through ~20 match sites, so the lint is silenced with a scoped `#[allow(clippy::large_enum_variant)]` + rationale comment on `ResponseState`.
- **Optional A→Z key-sort toggle on `s`; default stays wire order; ephemeral per response** (M7.7 Stage A follow-up, owner-approved) — Stage A's pretty view preserves the server's on-the-wire key order (the `preserve_order` decision above). This adds an **optional** alphabetical sort on top, as an explicit opt-in — **wire order remains the default** because it faithfully mirrors the response the user is inspecting; sorting is a convenience for scanning large objects, never imposed. A dedicated `Action::ToggleSortKeys` is bound to `s` in the `[keys.response]` overlay (`s` confirmed free there — the leader `s` sequences-submenu lives behind the leader, not this pane overlay) and added to the command palette. `ResponseView::toggle_sort_keys` mirrors `toggle_pretty`: it flips a `sort_keys` flag (defaults **false**, resets per response — nothing persisted to config), rebuilds `text`/`line_offsets` through the same `reformat_body_if_needed(text, syntax, pretty, sort_keys)` path, bumps `generation`, resets folds, and clears search (all geometry-changing side effects, identical to `p`). `sort_keys` is also hashed into `viewport_hash` (belt-and-braces). The sort is **recursive** (`sort_value_keys` walks the parsed `serde_json::Value`, rebuilding every `Object`'s entries in `String::cmp` order and recursing; **arrays keep element order**, only object keys sort) and works precisely because `preserve_order` is on — the rebuilt insertion-ordered map re-emits in the sorted order we wrote. It is **independent of `pretty`**: sort only affects the displayed text while `pretty` is on and the body is JSON; the app handler guards on pretty-JSON-body and otherwise no-ops with a `sort: pretty JSON only` notice (mirrors the `pretty: JSON body only` guard). **Copy (`y`) is unaffected** — it still returns `raw_text`, byte-exact, regardless of sort/pretty state, so a sorted display never changes what gets copied.

## 2026-07-10 (M7.7 close-out — viewer control-char/ANSI sanitize, tab-width, horizontal-window slice)

- **The displayed `text` is sanitized before indexing; `raw_text` stays the untouched on-the-wire decode — copy stays byte-exact** (M7.7 close-out, the design anchor) — `ResponseView::build` gains one seam: `let text = sanitize_for_display(&reformat_body_if_needed(...))` **before** `index_lines`. The three close-out items transform the *displayed* `text`; none touch `raw_text`. The same sanitize is applied in `toggle_pretty`/`toggle_sort_keys` (which rebuild `text` from `raw_text`). This is a correctness invariant: a sanitized display must never leak into what the user believes they copied, and a hostile OSC-52 write must never reach the terminal *or* silently vanish from a copy the user pastes elsewhere.
- **Copy-line (`Y`) returns the RAW on-the-wire line for non-reformatted bodies, the displayed line only for pretty-JSON** (M7.7 close-out follow-up — P0 fix) — `copy_line` originally read `source_text()` = the **sanitized** displayed `text`, so `Y` on a `text/plain` body leaked sanitized bytes (ANSI stripped, tabs→spaces, controls→`·`, `\r\n`→`\n`) instead of the wire bytes — a D1-invariant violation (`copy_all`/`y` was always correct via `raw_text`). Fix: `build` indexes `raw_line_offsets = index_lines(&raw_text)` once (raw_text never changes after build), and `copy_line` branches on whether the body was reformatted. **Predicate:** `self.line_offsets.len() == self.raw_line_offsets.len()` in the Body view — sanitize is line-count-preserving (ANSI-strip, tab-expand, control→`·`, and `\r\n`→`\n` all keep the `\n` count), so equal line counts ⟺ sanitize-only (non-reformatted) path, where displayed line `i` maps 1:1 to raw line `i`. On that path `copy_line` slices `raw_text` via `raw_logical_line(i)` (which, unlike `logical_line`, does **not** strip a trailing `\r` — `Y` copies exactly what the wire carried, CRLF included). When pretty JSON actually changed the layout (line counts differ), it returns the displayed pretty line (serde-escaped JSON has no raw controls to mangle, and the raw index no longer corresponds). Headers view returns the headers line. Regression-locked by tests that assert `copy_line` over a mangled `text/plain` line == the exact raw bytes, and over a pretty-JSON body == the displayed pretty line.
- **Known limitation: the unwrapped horizontal window is char-bounded, not cell-bounded, for wide glyphs** (M7.7 close-out, tracked P2) — `expand_wrap` windows an unwrapped row to `[h_scroll, h_scroll + width)` in **char** columns, not display cells, so a run of 2-cell-wide glyphs (CJK/emoji) can nominally exceed `width` terminal cells. This is pre-existing behaviour of the unwrapped path (master already sliced by chars), and ratatui truncates the over-wide `Line` at the pane edge, so there is **no terminal corruption** — only a slightly-shorter-than-ideal effective window on wide-glyph lines. Left as-is (wrap mode already does cell-aware splitting via `unicode-width` for anyone who needs exact-fit); a cell-accurate unwrapped window is a possible later refinement.
- **Sanitize order = strip ANSI → expand tabs → normalize CR → replace remaining controls, in a single left-to-right scan** (M7.7 close-out) — `sanitize_for_display` runs once per response build (not per frame): (1) **ANSI stripped entirely** (zero-width) — CSI (`ESC [` … final byte `0x40..=0x7E`), OSC (`ESC ]` … terminated by BEL `0x07` or ST `ESC \`, swallowing the trailing `\`), and any other lone `ESC` (a hostile/odd server could move the cursor, recolour the pane, or emit OSC-52 clipboard writes — none may reach the terminal); (2) **tabs expanded** to the next multiple of `TAB_WIDTH` computed as a display column *within the current logical line*, the column counter reset at each `\n` (proper tab stops, not a fixed 4-space swap); (3) **`\r\n` → `\n`** (a lone `\r` is not a line break — it falls through to step 4), so no phantom blank lines; (4) every **remaining C0 (`0x00..=0x1F`), DEL (`0x7F`), and C1 (`0x80..=0x9F`)** → a single visible `·` (U+00B7). `\n` (drives `index_lines`) and the already-expanded `\t` are exempt. Correct JSON has controls escaped, so serde pretty output is already clean — the teeth are on the raw / parse-failed / `text/plain` path.
- **Tab width is a documented const (`TAB_WIDTH = 4`), NOT a config knob** (M7.7 close-out, scoping decision) — the earlier bug was tab-width-*0* (tabs collapsed to nothing); the fix is proper 4-column tab stops. A configurable tab width was explicitly out of scope for this close-out — one well-chosen constant beats a knob nobody needs, and it keeps `config.rs` unchanged.
- **The horizontal window lives as `h_scroll` on `ResponseView`, bounding unwrapped long-line render cost; row *count* is unchanged so the fold/wrap→logical mapping stays intact** (M7.7 close-out) — `reformat` fixes the common minified-JSON case, but a minified/non-JSON line has no `\n` to split on, so an unwrapped logical line is still one giant row. Rather than materialize the whole line, `expand_wrap` now windows each **unwrapped normal-line** `DisplayRow` to `[h_scroll, h_scroll + width)` (fold-headers and the `width == 0` geometry callers keep full width; wrapped mode is unchanged). Only the row's `char_start`/`char_end` change — still exactly one row per unwrapped line — so `logical_at_display_row`/`display_row_for_logical`/`total_display_rows` (which read only `visible_idx` + row position) stay correct. `h_scroll` resets to 0 on build, on any generation bump (`toggle_pretty`), and on `ToggleWrap`. The highlight job builds its lines from the *windowed* slice and `viewport_hash` keys on `h_scroll`, so a pan re-highlights the visible window and lines up 1:1 with the painted slice.
- **h_scroll clamping lives in `render_done` (mirrors `clamp_scroll`), written back via the render outcome** (M7.7 close-out) — like vertical scroll/cursor, the view is borrowed immutably during render, so `render_done` computes `h_scroll.min(max_h_scroll(width))` (where `max_h_scroll` = widest currently-*visible* line − width) and reports it as `RenderOutcome::clamped_h_scroll`; the caller writes it back with `view.set_h_scroll(...)` after the borrow split, so an over-pan self-corrects on the next frame. `scroll_h` (the `H`/`L` handler) only needs to guard the left edge (`saturating_sub`); the right edge is the render clamp.
- **Pan on `H`/`L` (aliased to Left/Right), wrap-off only, remappable via the action table** (M7.7 close-out) — new `Action::ScrollBodyLeft`/`ScrollBodyRight` are in `ACTION_TABLE` (so they are remappable and appear in `churl keymaps`) and bound in the `[keys.response]` overlay on `H`/`L` (shift-h/shift-l — free there) with Left/Right arrows as aliases (also free in the Response overlay; global arrows are unbound, only the Explorer overlay uses them). The default binding passes `KeyMap::validate` warning-clean. `scroll_h` no-ops while wrap is on (wrapped rows already fit the width). App handler pans by a fixed `H_STEP = 8` columns.
- **Search-into-view is two-axis: the match row *and* the match column** (M7.7 close-out) — the existing `response_center_on_match` already moved the cursor onto the match row (vertical scroll-into-view). It now also calls `ResponseView::ensure_column_visible(start, end, width)`, which pans `h_scroll` so an unwrapped match past the right (or left) edge falls inside `[h_scroll, h_scroll + width)` — mirroring the vertical row nudge. Without it, an off-screen-right match on a minified line would be invisible after a jump. Inert while wrap is on. Match columns come from `current_match_columns` (byte offsets mapped to char columns).
- **f-jump action label corrected to "jump to pane"** (M7.7 close-out, one-liner) — M7.10 stage B made `f`-jump pane-only (no row labels), but the `ACTION_TABLE` label still read "jump to pane / row". ## 2026-07-11 (drive-test #3 — ephemeral peek + copy for masked secret values)

- **Peek + allow copy, chosen over "never reveal"** (drive-test #3, owner-locked) — the owner needs to reveal a masked value to debug (confirm a token is correct). churl's masking (env literals, the `SESSION` group, masked auth) previously had **no** escape hatch. The locked decision is an **ephemeral peek that also allows copy**, scoped to the env & vars editor: **`p`** on a masked var-row reveals its **resolved** value in place; **`y`** while revealed copies that value. Rejected alternatives: a persistent "show secrets" toggle (a lingering-secret footgun), and reveal-without-copy (fails the owner's actual debug need — confirming/pasting a token elsewhere).
- **The reveal is ephemeral view-state ONLY — never persisted, never logged** (drive-test #3, the security anchor) — the revealed plaintext lives solely in `EnvEditorState.reveal: Option<Reveal>` (a private field holding `{ scope, row, value: String, revealed_at: Instant }`). It is **never** written to any manifest/`folder.toml`/config/state file, never logged, and never placed in a message/error/panic string (the reveal path builds no error text from the value; `set_reveal` just stows the `String`). Re-masking drops the `Reveal`, dropping its `String`. **No-secret-to-disk proof**: `save`/`save_workspace_manifest`/`save_collection_meta` serialize only `scopes` rows (never `reveal`), so a peek cannot reach a writer; asserted by `reveal_lives_only_in_view_state_never_in_saved_rows` (rows + snapshot byte-identical across a peek; clear drops the plaintext) and the app-level `env_peek_does_not_write_the_secret_to_disk` (the on-disk `churl.toml` is byte-identical after a peek+copy+close).
- **Re-mask triggers = any row/scope move, any mode change, or a 6 s timeout; one row at a time** (drive-test #3) — the single seam is at the top of `EnvEditorState::handle_key`: **any** key other than the reveal key (`p`) or the copy-while-revealed key (`y`) calls `clear_reveal()` before acting, so `j`/`k`/`h`/`l`/Tab/Enter/`i`/Esc/`q` (every move + every mode change) re-mask. `p` re-peeks (refresh) and `y` preserves the reveal so the app can read the value back for the copy. The timeout reuses churl's existing transient-message mechanism verbatim — an `Instant`-stamped struct whose `expire_reveal()` is polled on the app's 250 ms tick, exactly like `Message::is_expired()` (`REVEAL_EXPIRE_SECS = 6`, mirroring `MESSAGE_EXPIRE_SECS`). The `Reveal` is pinned to a `(scope, row)` coordinate: `revealed_value()`/render gate on it, so a stale reveal can never paint a *different* row's secret, and moving between rows guarantees at most one revealed row. Tests: `reveal_remasks_on_row_move`/`_mode_change`/`_timeout`, `only_one_row_revealed_at_a_time`.
- **The peek resolves through the normal `Resolver`, minus the per-endpoint `collection` scope** (drive-test #3) — the reveal shows the *resolved* value, not the `{{placeholder}}` text, via `App::build_env_resolver()` = `session > cli > profile > workspace > env`. It deliberately omits the `collection` scope that `build_resolver` adds, because the env editor is not tied to a loaded endpoint (a collection var resolves *per-request*, not globally — there is no single collection to consult). For the two things that are actually masked today — `SESSION` captures (literal in-memory values) and secret-named non-placeholder literals — `substitute` is a no-op, but routing through the real resolver keeps the "resolved, not literal" contract honest and future-proof.
- **The editor stays `Resolver`-free; masking predicate is shared** (drive-test #3) — the env editor is UI-only (`churl-core` stays TUI-free, and the editor holds no resolver). So `p`/`y` emit new `EnvKeyOutcome::RevealRow`/`CopyRevealed`; the app resolves (`build_env_resolver().substitute`) and copies (`enqueue_clipboard`, the same path every other copy uses) and hands the plaintext back via `set_reveal`. The "is this row masked?" test used to decide peekability (`row_is_masked`) is the **same** predicate `render_var_line` uses to decide masking, kept in one place so reveal and mask can never disagree.
- **`SESSION` stays read-only under reveal** (drive-test #3) — peek never enters edit mode and never mutates a row; the Session group's existing read-only guards (add/delete/edit keys swallowed with a message) are untouched and still fire under a live reveal. Asserted by `peek_never_makes_the_session_row_editable`.
- **Auth values in the request pane are OUT of scope** (drive-test #3, scoped honestly) — request-pane auth masking is a **separate** seam (`components/request.rs` renders `*****` inline with no shared reveal state), not trivially in the env-editor seam, so per the task's fallback this change covers the env editor only (its masked literals + the `SESSION` group). Extending peek to the request-pane auth line is a clean follow-up (it would need its own reveal state + resolver plumbing on the request pane).
