# churl — Decision Log (ADRs)

Entries are append-only; one line per decision, date-prefixed. Superseding decisions reference the one they replace.

---

## 2026-07-05

- **Name: churl** — "curl" in a TUI shell; short, memorable, not taken on crates.io at decision time.
- **Default branch: master** — matches the host platform's default; no organisational convention overrides it for this personal project.
- **HTTP client: reqwest 0.13 + rustls** — async-native, pure-Rust TLS, no OpenSSL dep. Coarse timing (connect + total) is sufficient for M0–M3; a custom hyper connector (DNS/connect/TLS split timing) is documented as the later upgrade path when fine-grained waterfall data is wanted.
- **TOML library: toml_edit** — format-preserving; users edit endpoint files by hand and expect their comments and ordering to survive a round-trip through churl. `toml` (serde-only) would silently destroy formatting.
- **File-per-endpoint + explicit `seq` field** — learned from Bruno: monolithic collection files (Slumber, Insomnia) create merge conflicts and make git history noisy. One file per endpoint scales to large collections; `seq` provides stable ordering without relying on filesystem order.
- **Templating: hand-rolled `{{var}}` substitution** — no JS runtime (ATAC carries a ~10 MB binary bloat for Deno); no Tera/Handlebars (overkill for URL/header/body variable injection). Precedence: CLI `--var` flag → named profile → process environment. Secrets are process-env only; never written to synced files.
- **Secrets hygiene: never in workspace files** — workspace = a git repo users commit and sync. Secrets go in process env or a local `.env` file the user adds to their `.gitignore`. churl never reads, writes, or suggests committing secrets.
- **Local state: rusqlite (bundled)** — history, cookies, and cached sessions are high-churn and relational. `sled` (considered) was abandoned upstream. SQLite with bundled compilation avoids a system dep and is universally available. Schema migrations run at startup.
- **Syntax highlighting: syntect + two-face, lazy, off-thread, viewport-only** — ~23 ms per viewport is the measured cold-start risk; background thread + viewport hash cache keeps the render loop smooth. Whole-document highlighting of a 1 MB response is never done.
- **Vim-modal editing: edtui** — `tui-textarea` has been unmaintained since 2024; `edtui` is actively developed and ships modal vim bindings out of the box.
- **Keybindings: crokey + semantic Key→Action map** — follows gitui/helix model; users remap actions, not raw keys. Prevents keybinding spaghetti as the feature set grows.
- **Fuzzy search: nucleo** — same engine as Helix; outperforms skim/fuzzy-matcher at scale; async-native.
- **HTTP mocking in tests: wiremock** — spins a real HTTP server in-process; tests exercise the full reqwest stack without network access.
- **Snapshot tests: insta + ratatui TestBackend** — deterministic pixel-level assertions for TUI rendering; catch accidental layout regressions.
- **No HTTP-semantic caching** — churl is a development/testing tool. Returning a cached response when the user hits "send" is a footgun. History (SQLite) covers the recall use case.
- **Async runtime: tokio** (arrives M2) — not added in M0. Plain synchronous `crossterm::event::read()` is sufficient for the placeholder TUI; the full `tokio::select!` event loop comes with request execution in M2/M3.
- **Persistence: serde reads, hand-rolled merge writes** — loads use `toml_edit::de` into serde types; saves serialize to a fresh document (normalized to `[[array-of-tables]]`) and merge it into the existing file's `DocumentMut`: unchanged values untouched, changed scalars keep the old decor, stale keys removed, equal-length arrays-of-tables merged element-wise (unequal → replaced, comments lost). Saving canonicalizes explicit `enabled = true` away.
- **SQLite migrations: `PRAGMA user_version`** — no migrations table; each migration runs in its own transaction that bumps `user_version`. Simple, idempotent across reopens, and rollback-safe per step.
- **Platform paths: `dirs` crate** — config at `<config_dir>/churl/config.toml`, state at `<data_dir>/churl/state.sqlite`; hand-rolling XDG-vs-macOS-vs-Windows resolution is not worth the bugs.
- **Secrets enforcement: name-marker heuristic at save time** — `save_workspace_manifest` refuses profile variables whose name contains a secret marker (token/secret/password/api_key/…) with a literal value; `{{...}}` template placeholders pass. Loads never refuse (it's the user's file); only churl-initiated writes are gated.
- **Fuzzy engine: `nucleo-matcher`, not `nucleo`** (refines "Fuzzy search: nucleo" above) — same scoring engine, synchronous API. The high-level `nucleo` crate's threaded streaming matcher is overkill for workspace-sized endpoint lists and would drag a worker-pool lifecycle into the TUI.
- **Key routing precedence** — (1) an open overlay (search/palette) consumes every key; (2) request pane focused with edtui in a non-Normal mode → all keys to edtui; (3) otherwise crokey `KeyMap` first, unmapped keys fall through to edtui when the request pane is focused, and mapped navigation keys forward their original event to edtui there. edtui owns insert/visual modality — the app never re-implements vim.
- **Keybinding overrides: flat `[keys]` string map in `churl-core::config`** — core carries only `"combination" = "action-name"` strings (keeps core TUI-free); the `churl` crate parses both sides at startup and fails loudly on unknown entries rather than silently skipping them. Overrides layer on top of the `key!`-built defaults.
- **Explorer laziness matches persistence laziness** — startup lists collection directories only; endpoint files parse on first expand or on search-overlay open, protecting the < 100 ms cold-start budget.
- **BackTab default binding built explicitly** — crossterm always reports Shift-Tab as `BackTab` + `SHIFT`, and crokey's `key!(backtab)` does not add the modifier (its string parser does); the default map constructs `KeyCombination::new(BackTab, SHIFT)` directly.
- **Task-level cancellation, not core-level** — `churl-core::http::execute` is a plain runtime-agnostic `async fn` with no `AbortHandle` in its signature. The TUI spawns it with `tokio::spawn` and keeps the `AbortHandle`; a monotonic `generation` counter on `App` tags each request, and a landed `AppMsg::Response` whose generation ≠ the current in-flight generation is dropped (stale result after cancel+resend). Ctrl-C cancels an in-flight request instead of quitting; `q`/`Esc` always quit. This keeps core free of any runtime dependency (refines the M0 "AbortHandle per in-flight request" wording, which lived in core in the original plan).
- **reqwest `rustls` feature, not `rustls-tls`** — reqwest 0.13 renamed the pure-rustls feature to `rustls` (with the connector selected via `ClientBuilder::tls_backend_rustls()`); the M0 decision's `rustls-tls` name was the 0.12 spelling. Same intent (pure-Rust TLS, no OpenSSL), current name.
- **No `{{var}}` templating in M3** — request execution uses the URL, headers, and body verbatim. Templating (the `{{var}}` precedence chain) is an M5 deliverable; wiring it into execution before the substitution engine exists would be dead code.
- **No response body-size cap in M3** — the full body bytes are read (`.bytes()`) and decoded lossily for the viewer; giant responses can bloat memory. Deferred to M6 (open question in MILESTONES) rather than guessed at now.
- **Response viewer: no wrap, truncate at pane width** — the virtualised viewer slices only the visible lines out of a line-offset index and lets ratatui truncate long lines at the pane edge. Line wrapping and horizontal scroll are deferred (open questions in MILESTONES); virtualisation (never materialising all lines) is what protects the 1 MB-body budget, and wrapping would complicate the offset math without buying that.
- **Viewport-stateless syntax highlighting** — the off-thread worker starts a fresh `HighlightLines` at the top of each viewport (highlighting only the visible lines, cached by a viewport hash of generation/scroll/height/syntax). This is a known imperfection: multi-line constructs (block comments/strings) that open above the viewport are mis-coloured. Accepted for the render-loop budget it buys; whole-document highlighting of a 1 MB response is never done. Foreground RGB only (background skipped so the theme doesn't fight the pane); theme is two-face's Nord.
- **History written on every terminal outcome** — success, failure, and cancel each insert a `NewHistoryEntry` synchronously in the event handler (a fast local SQLite write; acceptable on the UI thread). `status` is `Some(code)` only on success; `duration_ms` comes from response timing (`None` on cancel/failure). Insert failure warns on the statusline, never crashes; a failed history *open* disables history for the session (also non-fatal).
- **Body-size cap in M4** (supersedes "No response body-size cap in M3") — stream the response body with a default 10 MB cap, config-overridable (`max_body_bytes`); `Response` gains a `truncated` flag surfaced in the status line. Owner decision at the post-M3 plan review: it is a safety hole (accidental giant download freezes/OOMs the TUI), not polish, so it lands with M4's core work rather than waiting for release polish.
- **Auth is its own milestone (M5), not part of M4** — owner decision at the post-M3 plan review. M4 imports `-u`/`Authorization:` as plain headers; M5 adds the minimal auth model (basic/bearer/API-key) and remaps. Keeps M4 lean at the cost of one remap step. Milestones renumbered: themes/templating → M6, polish/release → M7.
- **Headers view + wrap toggle land in M7 polish** — resolves the M3 open questions. Wrap toggle chosen over horizontal scroll (wrap covers the read-the-long-line need; horizontal scroll adds offset state for little gain). Status-line header count suffices until then.
- **Insert-mode exception for Ctrl-S/Ctrl-C** (refines "Key routing precedence") — from M4, Ctrl-S (send) and Ctrl-C (cancel/quit) are intercepted *before* edtui even in insert/visual mode, making edit→send one keystroke. The only exception to "edtui owns non-Normal modes": both are control-modified non-text keys, so no editing capability is lost.
- **Request timeout configurable in M4** (`timeout_secs`, default 30) — the M3 `DEFAULT_TIMEOUT` const becomes a config knob alongside `max_body_bytes` (owner question at the post-M3 plan review: same class of safety limit, and 30 s is wrong for slow staging/report endpoints). Global config only; per-endpoint override is deliberately deferred until a real need appears.
- **curl import flag policy: strict, lossless, no file reads** — any flag outside the supported set is a hard `UnknownFlag` error (fail loudly; silently dropping a flag imports a *different* request). `-F`/`--form` multipart and `@file` data payloads are `Unsupported` errors — churl never reads files during an import. `-u` lands as a plain `Authorization: Basic <base64>` header with a warning (M5 remaps to first-class auth). The URL's query string stays in the URL — import never explodes it into `Param`s (lossless and simple). `-L`/`-s`/`-v`/`-S` are accepted silently (no request semantics; redirects are already the client default); `--compressed`/`-k` accepted with "ignored" warnings; `-o` consumes and discards its value with a warning. Body kind derives from content + headers: `{`/`[` prefix or JSON Content-Type → json, `k=v&…` shape → form, else text; `--json` forces json, implies POST, and adds `Accept: application/json` when absent.
- **curl export: shlex-quoted single line, round-trip stable** — `export_curl` emits `curl [-X METHOD] -H '…' --data '…' <url+enabled params>` with every argument through `shlex::try_quote` (paste-safe, no line continuations); only enabled headers/params are emitted. `-X` is omitted for a body-less GET but kept for a GET *with* a body so `import_curl(export_curl(e))` reproduces the same method/URL/headers/body (import infers POST from a body). Enabled params are percent-encoded and appended to the URL query, matching import's query-stays-in-URL rule.
- **`ExecuteOptions` + streamed body cap** (implements "Body-size cap in M4") — `execute(client, request, &ExecuteOptions)` streams the body via `chunk()`, accumulating at most `max_body_bytes`; a chunk that would exceed the cap is cut at the boundary, `Response.truncated` is set, and the connection is dropped. An exact-cap body is *not* truncated. The status line appends `· truncated at <size>`; history is unchanged (it never stored body size). Caller resolves the knobs from config (`Config::max_body_bytes()`, `Config::timeout()` → `build_client(timeout)`).
- **Insert-mode interception resolves through the keymap** (implements "Insert-mode exception for Ctrl-S/Ctrl-C") — in the non-Normal-edtui pass-through, a key with the CONTROL modifier is looked up in the crokey keymap and dispatched iff it resolves to `Send` or `Quit` (the Quit path already downgrades Ctrl-C to cancel while a request is in flight); everything else still goes to edtui untouched. Lookup-not-hardcode honours user remaps; the CONTROL requirement guarantees no text-input key can be stolen. This remains the single exception to "edtui owns non-Normal modes".
- **Plugin system scheduled last (M9), guardrails active from M5** (owner decision 2026-07-05) — community plugins are the final planned milestone because the plugin API freezes the shapes M5–M8 stabilise. No re-architecture needed now (core/TUI split, single `execute()` chokepoint, `Action`+palette, one content-type mapping are already the right seams), but M5–M8 must keep them: single auth dispatch point, single template resolver, data-driven palette commands, one formatter mapping, `churl-core` types as the stable plugin-facing API. Runtime tech (Lua / WASM / subprocess) is an M9 design question; the "no JS runtime" decision stands. Guardrail details in MILESTONES §M9.

## 2026-07-06

- **Auth persists as an internally-tagged `[request.auth]` table** (M5) — `Auth` is a serde enum with `#[serde(tag = "type", rename_all = "lowercase")]` (`basic` | `bearer` | `apikey`), so an endpoint file reads naturally (`type = "basic"`, `username`, `password`) and new kinds add a tag value instead of a parallel struct of optionals with a validation fn. toml_edit round-trips the tagged representation cleanly (verified first, per the M5 design's risk note — the fallback flat struct was never needed). The apikey `placement` defaults to `header` and is omitted on serialize, matching the `enabled = true` canonicalization rule. Kind changes merge safely: the format-preserving merge already drops stale keys, so `basic → bearer` removes `username`/`password`.
- **`AuthWire` single-dispatch seam in `churl-core::auth`** (implements the M9 plugin guardrail for M5) — `apply_auth(&Auth) -> AuthWire` holds the one `match` on auth kinds; every kind resolves to a wire *effect* (`Header { name, value }` or `Query { name, value }`) and `execute()`/`export_curl` only apply effects, never matching on `Auth`. A future plugin-provided auth kind slots into that single match (or its plugin-dispatch successor), not into scattered call sites.
- **Import replaces literal secret auth values with placeholders** (M5, supersedes M4's "`-u` lands as a plain base64 header") — `churl import` writes to stdout or `--out`, and a redirected stdout is a workspace file too; M4's `Authorization: Basic <base64>` path put a recoverable secret on disk. Now `-u user:pass` becomes `Auth::Basic` with the password replaced by `{{password}}` (kept verbatim if already a placeholder) and `Authorization: Bearer <literal>` becomes `Auth::Bearer` with `{{token}}`, each with a warning pointing at profiles/env (M6). Enforcement is symmetric at write time: `save_endpoint` *and* `endpoint_to_toml` refuse literal secret auth values (`PersistenceError::SecretsInAuth`), reusing the name-marker heuristic — `password`/`token` are always secret-named; an apikey `value` only when its `name` looks secret. Loads never refuse. With multiple auth sources in one command, the first takes the first-class slot; the rest stay plain headers (warning), so imports are deterministic.
- **Auth injection precedence: the user's enabled header always wins** (M5) — in `execute()`, an `AuthWire::Header` is injected only when no *enabled* user header with the same name (case-insensitive) exists; a disabled header does not block injection. This covers "user `Authorization` beats basic/bearer" and generalizes to apikey header names. Query-placed api keys are appended after enabled params with no precedence rule (both pairs are sent) — servers, not churl, define duplicate-param semantics.
- **Variable scoping: three scopes, profile wins over collection** (M6; refines the precedence in "Templating: hand-rolled `{{var}}` substitution") — owner decision 2026-07-06. Chain: CLI `--var` → active profile → collection vars → workspace vars → process env. Workspace-level `[vars]` in `churl.toml` carries shared defaults; profiles remain the *only* place environments are defined (no per-collection profiles); a collection carries a flat `[vars]` table in its `folder.toml` manifest (the filename reserved since M1 — `persistence::FOLDER_FILENAME`, already excluded from endpoint listing) — environment-independent defaults that any profile can redirect, so switching dev→prod always takes effect. `folder.toml` gets the same format-preserving merge writes and secrets name-marker enforcement as `churl.toml`. The resolver is a single function over an ordered scope list (the M9 plugin-guardrail seam); the full three-scope system ships in M6. Perf considered and negligible: one extra map probe per var at send time, one small TOML parse per (already lazy) collection open.
- **Multipart (`-F`) bodies: approved, post-release backlog** (resolves the M4 open question; refines "curl import flag policy") — owner decision 2026-07-06. Multipart/form-data becomes a model feature when picked up (multi-part body model, TUI body-type editing, reqwest multipart execution, `-F` import/export remap); until then M4's hard `Unsupported` error stands. Slot after M8 — see the post-release backlog in MILESTONES.
- **Owner drive-test review → M6.5 + M6.6, fractional numbering** (2026-07-06) — first live use surfaced 9 notes: 5 quick fixes (stacked Request/Response layout, statusline auto-expiry back to the hint, profile-switch message dedup, in-flight visibility, active-profile marker in the picker) become **M6.5**; the request-editing UX (URL bar with method chip + indicators, Params/Headers/Auth/Body tabs, in-app endpoint/collection CRUD) becomes **M6.6**, scheduled *before* release — it had been deferred in M2's notes ("full editing UX matures in later milestones") but never assigned a milestone, a planning gap the review caught. Fractional numbers because `M7`–`M9` references (esp. the M9 plugin guardrails) are baked into code comments and docs; renumbering would churn the codebase for cosmetics. Response-body search stays in M7.
- **Distribution: prebuilt binaries + `curl | sh` installer, zero end-user prerequisites** (M7, owner requirement 2026-07-06) — the release action ships per-platform binaries (macOS arm64/x86_64, Linux x86_64 musl static + aarch64, Windows); rustls (no OpenSSL) and bundled SQLite were chosen day-one precisely so the binary is self-contained — Rust is a build-time dependency only. `install.sh` detects OS/arch and installs to `~/.local/bin`. Onboarding: `?` help overlay (the effective keymap, via M6's `KeyMap::iter`) and a `churl tutorial` demo-workspace scaffold.
- **Resolver = ordered scopes + env fallback, the single `{{var}}` seam** (M6, implements the M9 template guardrail) — `churl-core::template::Resolver` holds a `Vec<Scope>` (each a named flat map, highest precedence first) and `resolve(name)` returns the first scope that defines it, falling through to `std::env::var` last (process env is never snapshotted). `substitute` hand-rolls a `{{name}}` delimiter scan (no regex crate) matching `config::is_template_placeholder`'s shape (name chars `[A-Za-z0-9_.-]`, inner whitespace trimmed); an unresolved or malformed placeholder is left **verbatim** (M5's send-verbatim rule — no error). `substitute_request` is the one place a `Request` is templated (url, header/param *values*, body, all auth string fields; header/param *names* never). It runs in the TUI at send time on the already-cloned request, before `http::execute` — `execute()` and `export_curl` stay substitution-free, and resolved values never touch disk. Plugin template *functions* (M9) extend the lookup here, not scattered call sites.
- **Collection metadata in `folder.toml` via `CollectionMeta`** (M6) — the M1-reserved `folder.toml` (`persistence::FOLDER_FILENAME`, already excluded from endpoint listing) now parses into `CollectionMeta { vars }` (serde default; missing file ⇒ `Ok(default)`). `load_collection_meta`/`save_collection_meta` reuse the existing format-preserving `save_value` merge machinery (not a fork) and `save_collection_meta` refuses literal secret-named vars via `collection_secret_violations`, mirroring the manifest's `secret_violations` (which M6 extends to also cover the workspace-level `[vars]`, prefixed `vars.`). The explorer caches each collection's vars lazily alongside its endpoints; the app asks for the selected endpoint's collection vars at send time.
- **Theme = named style slots parsed from core strings, mirroring `[keys]`** (M6) — `churl-core::config` carries only strings (`theme` selects a built-in, `theme_colors` maps slot name → colour); the TUI's `Theme` parses them into ratatui `Style`s at startup and **fails loudly** on an unknown built-in, unknown slot, or bad colour (named ANSI or `#rrggbb`), naming the offender — no silent fallback, exactly like the keymap. Slots are data-driven (one `slot_mut` match, one field each: `border_focused`, `border_unfocused`, `title`, `selection`, `statusline`, `status_error`, `auth_mask`, `response_status`, `jump_label`) so a new slot is one line, per the M9 palette-guardrail spirit. Built-ins `dark` (default, keeps M5 rendering — text snapshots unchanged) / `light`; `[theme_colors]` overrides individual slots over the selected built-in. The syntect embedded theme follows the UI theme (Nord for dark, `EmbeddedThemeName::InspiredGithub` for light) through the single `SyntaxToken` mapping point. **Deviation from the M6 design's `[theme.colors]`**: `theme` is a scalar key (`theme = "dark"`), so a `[theme.colors]` sub-table would collide with it in TOML — the table is `[theme_colors]` instead (flat, top-level). Same data, same slot names.
- **Jump-mode: overlay-level label navigation** (M6) — a new `Mode::Jump` + `Action::Jump` (default `f`, config-overridable). Entering assigns home-row-first single-char labels (`a s d f g …`) to the three panes first, then explorer rows starting at the scroll offset (`first_visible` — labelling from row 0 would let offscreen rows eat the alphabet in a scrolled tree; main-session review fix), capping at the label alphabet. A label focuses that pane, or focuses the explorer + moves its cursor to that row and selects it (endpoint rows load, same as Enter). It is routing precedence slot 0 (alongside search/palette): it consumes every key — a label jumps, `Esc` always cancels, the Jump key cancels only when it labels no target (an assigned label wins, or the default `f`'s own row would be unreachable), everything else is ignored. Labels render themed (`jump_label`): at the start of each explorer row and in each pane title (`[a]`). `SwitchProfile` is palette-only (no default key); the statusline shows the active profile name when one is set.
- **`--var`/`--profile` global CLI flags + `churl keymaps`** (M6) — `--var key=value` is repeatable (missing `=` is a hard error) and populates the highest-precedence `cli` resolver scope; `--profile <name>` selects the active profile at startup and errors on an unknown name, listing the available ones (a typo'd profile would otherwise silently send the wrong environment). `churl keymaps` prints the effective map (defaults + `[keys]` overrides) sorted by action config-name, as plain aligned text — `action-name  combo[, combo…]  (default | overridden)  label` — where *overridden* means the action's bindings differ from `KeyMap::default()`; unbound actions show `(unbound)`.
- **Contextual keymap overlays** (M6.6) — `KeyMap` gains `overlays: BTreeMap<PaneCtx, HashMap<KeyCombination, Action>>` (`PaneCtx ∈ {Explorer, UrlBar, Request, Response}`); `lookup_ctx(key, ctx)` searches the focused pane's overlay first, then the global map. This is the vehicle for keys that must mean one thing pane-locally and another globally — the flat crokey table can't host both. Config gains optional `[keys.explorer]` / `[keys.urlbar]` / `[keys.request]` / `[keys.response]` sub-tables under `[keys]`, parsed with the same fail-loud unknown-table/combo/action errors; a custom `Config` split (`KeyEntry` untagged enum + `split_key_overlays`) partitions the raw `[keys]` table into flat scalars vs nested overlays so `Config::keys` stays a flat string→string map. `churl keymaps` prints overlays under per-pane headers after the global section.
- **`1`–`4` shadow global pane-focus inside the Request pane** (M6.6) — in the Request overlay, `1`–`4` select tabs (Params/Headers/Auth/Body); globally `1`–`3` still focus panes. Rationale: the keystroke north star (direct tab jump is the common loop) and pane focus stays reachable everywhere via Tab/Shift-Tab and jump-mode. Recorded per the design's explicit ask.
- **Derived dirty tracking, not a flag** (M6.6) — `App` keeps `loaded_snapshot: Option<Endpoint>` (a pristine clone at load); dirty is *computed* by comparing the live request (incl. the edtui body folded in) against the snapshot on each render tick. No boolean flag to forget to set/clear on the dozen mutation paths (url/method/params/headers/auth/body); comparison at this size is negligible. The `●` indicator and the discard-changes guard both read `is_dirty()`. Save (`w`) refreshes the snapshot; discard drops it.
- **`LineEditor` over an editor dependency** (M6.6) — a ~90-line hand-rolled single-line editor (`Vec<char>` buffer + char cursor; insert/backspace/delete, ←/→, Home/End, Ctrl-A/Ctrl-E) shared by the URL bar, request-row field edits, and CRUD prompts. edtui is a full vim-modal editor — overkill and wrong-feeling for a one-line input — and no new dep is worth one line of text. Char-indexed so multibyte input behaves. The M4 Ctrl-S/Ctrl-C interception generalises to it: while a `LineEditor` holds the keyboard, a CONTROL key resolving to Send/Quit is dispatched instead of typed.
- **In-app CRUD flows through `churl-core::persistence` seams; the TUI never touches the filesystem** (M6.6) — new core fns `create_endpoint`/`rename_endpoint`/`delete_endpoint`/`create_collection`/`rename_collection`/`delete_collection`, all typed-error (`PersistenceError`, gaining `EmptyName` + `AlreadyExists`) and all save paths keep the secrets refusal. `create_endpoint`'s `seq` is `max(existing seq) + 1` (plain +1 — the fixture corpus uses no fixed step) and its filename is a kebab-case slug of the name, collision-suffixed `-2`/`-3` (matching the corpus). `rename_endpoint` writes the name change first (running the secrets gate) *then* moves the file, so a refusal aborts before any move (atomic: both or neither). `create_collection` makes only the directory — `folder.toml` stays lazy until vars exist, matching the existing missing-is-empty behaviour.
- **Delete confirm is risk-proportional** (M6.6, owner decision) — deleting an endpoint is a `y/n` confirm (one file, recoverable by not saving elsewhere); deleting a collection requires typing its exact name (recursive directory removal is destructive). Switching endpoints while dirty raises a three-way `Confirm(DiscardChanges)`: `s` save-then-switch / `d` discard / `Esc` stay — never an auto-write (owner requirement: churl never writes on a keystroke, only on explicit `w`).
- **Command palette is a curated allowlist, not `Action::all()`** (M6.6, owner addition mid-flight) — the palette used to build over every `Action`, surfacing pane-contextual motions (Up/Down/Collapse/Tab*/Row*/Method*/EditUrl…) that no-op or misbehave when dispatched from an overlay (owner drive-test: "most commands don't work"). It is now an explicit `const COMMANDS: &[(&str, Action)]` of context-free commands only (Send, Cancel, Save, New endpoint/collection, Rename, Delete, Switch profile, Focus explorer/URL bar/request/response, Quit). New Actions do **not** auto-appear — inclusion is a deliberate per-command decision. Palette-dispatched CRUD acts on the explorer selection exactly like the keymap path; a missing selection surfaces a statusline error, never a silent no-op. A unit test guards that no context-only action is in the list.
- **Single guarded endpoint-switch seam (`guarded_load`)** (M6.6 review round) — the review found dirty edits were silently discarded on three switch paths that bypassed the explorer-Enter guard: the search overlay, jump-mode row labels, and CRUD reselects (new endpoint / rename). Every path that replaces the loaded endpoint now funnels through one seam: `App::guarded_load(PendingLoad)` (`PendingLoad ∈ {Row(index), File(path)}`). - ~~**Clipboard via OSC 52, no native dep** (M7 wave 1)~~ **— REVERSED by M7.5.3 (see below).** Original: copy (`y` view / `Y` line) emits `ESC ] 52 ; c ; <base64> BEL` to ratatui's terminal backend writer (`tui/clipboard.rs`), flushed after the key is handled (dispatch has no terminal handle, so a `pending_clipboard` payload is drained in the run loop). Payload cap (1 MB, `MAX_COPY_BYTES`, char-boundary) and the `copied first 1.0 MB of 4.2 MB` note carry over.
- **Clipboard: native (arboard) primary + OSC-52-with-passthrough fallback** (M7.5.3, reverses the M7-wave-1 decision above) — the owner needs copy to work across almost all terminals, multiplexers, **and** operating systems. `tui/clipboard::copy(payload, out) -> CopyOutcome` now tries the **native OS clipboard first** via `arboard` (macOS NSPasteboard, Windows, Linux **X11** — the only path that reliably reaches the system clipboard regardless of terminal; native Wayland is NOT enabled — arboard's `wayland-data-control` feature is off — so a pure-Wayland session uses the OSC 52 fallback, a follow-up if native Wayland is wanted), and only if that is unavailable/errors falls back to **OSC 52 wrapped for the active multiplexer**: tmux (`$TMUX`) → DCS passthrough `ESC P tmux ; <osc52-with-each-ESC-doubled> ESC \` (the payload's leading ESC doubled → two ESCs, not three); GNU screen (`$STY` or `$TERM` starts `screen`) → `ESC P <osc52> ESC \`; else raw. **Honesty**: the silent `let _ =` is gone — the run loop reports the real `CopyOutcome` (native/osc52 = "copied…", total failure = "copy failed"); the resolved-vars secret warning stays. `arboard = { version = "3.6", default-features = false }` on the **churl bin only** (never churl-core) — it was already a transitive dep via edtui, so it adds no new packages and its Linux backend (`x11rb` → pure-Rust `rustix`, no libxcb-dev) already compiled on ubuntu CI (no `ci.yml` change needed). Linux caveat: the X11 selection clears when the owning process exits; churl is a long-lived TUI so a copy persists for the session — acceptable, and we do **not** spawn a persistence daemon. **Testing seam**: native copy is never called from `cargo test` (headless CI has no display) — unit tests cover only the pure logic (OSC 52 framing, tmux/screen passthrough wrapping, multiplexer detection over injected env values, payload capping, `CopyOutcome`); the copy message is now built as a deferred *success* message and asserted via `App::pending_copy_message`, since the real copy runs in the terminal-owning run loop.
- **Wrap index built eagerly per render, highlighting deferred under wrap** (M7 wave 1) — soft-wrap expands the visible map into display rows with a `unicode-width`-aware greedy split (O(total lines), same order as the existing line-offset build; viewport rendering stays virtualised). **Fallback taken**: under wrap, response bodies render as *unhighlighted* plain text — slicing highlighted syntect spans at wrap boundaries (split spans by display width) was judged not worth the complexity for this milestone. Unwrapped mode keeps full off-thread highlighting. Recorded as a deviation from the pinned design's preferred path.
- **JSON fold regions scanned once per response by a string-aware scanner** (M7 wave 1) — `components::fold::scan_regions` does a single O(n) pass tracking in-string/escape state, pushing `{`/`[` openers onto a stack; a closer forms a region only with a *matching* opener kind (`}`↔`{`, `]`↔`[` — a mismatch drops the opener without a region) and only when opener and closer lines differ. Braces inside string literals are inert; unbalanced/truncated JSON never panics (unclosed openers at EOF yield no region). Scanned lazily on the first fold action, cached on the view. Only `SyntaxToken::Json` bodies fold (the single content-type mapping point stays authoritative — no per-format forks); non-JSON `o`/`O` no-op with a `folding: JSON responses only` notice (the Headers view of a JSON response gets its own `folding: body view only`).
- **Search feedback uses two mechanisms deliberately** (M7 wave 1, accepted deviation — the pinned design asked for one) — while a search is live the stats title carries `k/N matches` (persistent state, and it survives the message row being occupied by the `/query` input), and each `n`/`N` navigation additionally posts `match k/N` to the message row (transient event, confirming the jump just taken). The two answer different questions and neither position can cover both; keeping both is deliberate.
- **Case-insensitive search maps offsets through a per-char lowering table** (M7 wave 1 fix round) — `lowercase_with_offsets` builds the lowered haystack alongside one `(lowered_offset, original_offset)` entry per original char (+ sentinel); every lowered match position maps exactly back to original byte offsets (`map_lowered_offset`; offsets inside a multi-byte lowercase expansion round outward so the mapped range always covers the match). No length heuristics, no whole-line fallback — per-char length shifts whose totals cancel (e.g. `İ` 2→3 with `ẞ` 3→2 on one line) match correctly.

## 2026-07-08 (M7.1 — collection interchange)

- **JSON interchange hand-rolled against `serde_json::Value`, no Postman schema crate** (M7.1) — `churl_core::interchange` maps Postman Collection v2.1 to/from the churl model by walking untyped `Value`s. Rationale: real-world exports are messy (missing/renamed/extra fields, auth stored as arrays *or* objects, bare-string URLs), and a strict typed `#[derive(Deserialize)]` schema would reject valid files. Import is deliberately *liberal* — unknown methods default to GET with a warning, unsupported constructs degrade to warnings, never hard errors. `serde_json` is the one new dependency (workspace + churl-core).
- **Selectable export dialect: Postman v2.1 (round-trip) OR churl-native JSON** (M7.1, resolves the design-session open question) — `JsonDialect {Postman, Native}` chosen per export action. Postman v2.1 is the interoperable, lossy-but-round-tripping format (`import_postman_v21(export_postman(x))` reproduces method/url/headers/body/auth — round-trip test enforced); churl-native is a thin lossless envelope (`{ churl_version, name, collections: [{ name, endpoints: [Endpoint...] }] }`) reusing the model's serde derives, for full-fidelity backup/transfer between churl workspaces. No native round-trip importer yet (native import is not a stated requirement) — the native shape is for backup and is trivially re-parsable.
- **Nested Postman folders flatten into a single collection name on import** (M7.1) — churl collections are one directory deep (no nested collection dirs), so a request nested under `A > B` imports into a collection named `A / B` (the `folder_path` joined with `" / "`, then slugified by persistence). Root-level requests land in a collection named after the collection's `info.name`. Nested collection directories are a deliberate post-release backlog item — flattening keeps the imported layout navigable today without a model change.
- **Query stays in the URL on Postman import** (M7.1, mirrors the M4 curl-import decision) — `request.url.raw` is imported verbatim including its query string; import does **not** explode the query into `Param`s. Keeps import lossless and simple, and matches the existing curl-import behaviour (`query_string_stays_in_url`). Postman `url.query[]` (when present) is ignored in favour of `raw`.
- **Postman body-mode + auth-secret handling on import** (M7.1) — body modes: `raw` → Text/Json (by `options.raw.language == "json"`), `urlencoded` → Form (`k=v&…`, raw join — symmetric with export's raw split so round-trips are lossless), and `formdata`/`file`/`graphql` drop the body with a warning. Auth `basic`/`bearer`/`apikey` map to the churl first-class kinds; literal secret values are placeholder-ized (`{{password}}`/`{{token}}`/`{{api_key}}`) exactly like the M5 curl-import policy so `save_endpoint` never refuses them, and export refuses any endpoint still carrying a literal secret auth value (reuses `config::auth_secret_violations`). Collection-level `variable[]` is not imported (warned — define them as workspace/profile vars instead).
- **Import shared between CLI and TUI via `interchange::write_import`** (M7.1) — both `churl --import-collection <file>` (writes into the cwd workspace, prints a summary, then launches the TUI — fails loudly with a nonzero exit on a bad file rather than launching half-imported) and the in-TUI palette import call the one core writer, which creates one collection per flattened folder and writes each endpoint via `create_endpoint` + `save_endpoint`. The bin's `import_collection_into` is the thin testable seam over it.
- **`C` (shift-c) is the only new default keybind; interchange import/export stays palette-only** (M7.1) — copy-as-curl earns a Request-pane overlay binding (`C` is free globally and in the overlay; verified against the `?` help guard test, which renders it automatically); the resolved-vars copy variant and all import/export actions are rare and need a path/curl prompt anyway, so they live in the command palette only. Export writes are guarded to stay inside the workspace root via a lexical-normalize + `starts_with(canonical_root)` check (`export_target`); imports read any user-typed path (their own file). Copy-as-curl has two palette variants: default emits `{{var}}` verbatim, the "resolved vars" variant substitutes through the M6 resolver first (explicit secrets-caution opt-in that may place a real secret on the clipboard).

### M7.1 review-round fixes (2026-07-08, main-session adversarial review)

- **`write_import` bootstraps a `churl.toml` when the target dir has none** (M7.1 review, fix F1) — `churl --import-collection` into a bare directory previously wrote collection dirs + endpoint files but no manifest, so the launched TUI called `open_workspace` → `None` and showed an empty explorer (the import was invisible, contradicting the spec's "opens the TUI with the collection loaded"). `write_import` now writes a minimal manifest (`name` = `info.name`, or `imported`) when `churl.toml` is absent, and leaves an existing manifest untouched (in-TUI imports already have one). - **Slug-collision on import warns instead of silently merging** (M7.1 review, fix F4/F5) — distinct flattened collection names that slugify to the same directory (`"A B"` vs `"a-b"`, a nested path colliding with a literal folder, empty folder names) were silently merged into one collection with no signal. `write_import` now detects when two distinct names resolve to the same dir and pushes a warning into the summary; the reported collection count is the number of distinct directories. Empty Postman folder names default to `folder` (previously only *absent* names did). Covered by `write_import_warns_on_collection_slug_collision`.
- **Known limitation — the secret gate covers only `auth.*`, not literal secrets in headers/URL/body** (M7.1 review, F3, deferred by design) — import placeholder-izes and export refuses only first-class auth secrets (`Basic.password`, `Bearer.token`, secret-named `ApiKey.value`), matching the M5 persistence policy (headers were never gated). A literal token living in an `Authorization` header or URL query is therefore imported to disk and exported to JSON verbatim. This is symmetric with existing behaviour and not a regression, but a shared JSON export is a new, highly shareable surface — extending secret detection to header values is a policy decision left to the owner rather than changed asymmetrically here. Export still refuses first-class auth secrets in both dialects.
- **Structured Postman `url` without `raw` imports empty, with a warning** (M7.1 review, F6) — a `url` object using only the `{host, path, query}` arrays (no `raw`) now imports with an empty URL *and* a warning, rather than silently. Real v2.1 exports include `raw`; this keeps the "liberal import" promise honest. Form-body round-trip (F7) remains lossy for values containing literal `&`/`=` (raw join/split, no percent-coding) — documented, low impact.

## 2026-07-08 (M7.3 — environments & vars editor + crash fix)

- **Manifest-in-collection crash fix + lenient endpoint load** (M7.3) — `Collection::endpoints()` skipped `folder.toml` but not `churl.toml`, so a nested-workspace layout (a collection dir that is itself a workspace root — e.g. `churl --import-collection` from the demo root) had its `churl.toml` parsed as an `Endpoint` (`missing field 'request'`), and the `?` propagated to abort the whole TUI load. Fix is two-layer: (1) `endpoint_files()` (shared by both load methods) now skips `MANIFEST_FILENAME` as well as `FOLDER_FILENAME`; (2) a new `Collection::endpoints_lenient() -> CollectionLoad { endpoints, warnings }` degrades a per-file parse failure to a `"skipped <file>: <reason>"` warning instead of aborting — only the directory `read_dir` itself stays a hard error (a vanished collection is a real failure, one bad file is not). `endpoints()` stays strict for callers that want it (existing `malformed_endpoint_error_names_the_file` test unchanged); the TUI explorer/search load paths use the lenient variant and the app drains `ExplorerState::take_warnings()` into the M6.7 message row after expand/search-open/reload (fail loudly, never silent). Chosen over "don't treat a workspace-subdir as a collection" because the lenient path also covers the general "one hand-corrupted endpoint must not nuke the workspace" robustness the milestone asked for.
- **Save-prune verified, no save-path change needed** (M7.3, the CRITICAL correctness gate) — the design flagged that an editor "delete" is broken if the format-preserving save only overlays. Investigation: `merge_tables` **already prunes** — it removes keys present on disk but absent from the saved struct (its first loop), and an unequal-length `[[profiles]]` array is wholly replaced (comments on survivors lost, the entry gone). So deleting a var key, deleting a profile, renaming a profile/var, and emptying a scope's `[vars]` all correctly disappear from disk. No production change; instead the gate is locked down by round-trip tests that re-read the file and assert the removal (`deleting_a_workspace_var_key_removes_it_from_disk`, `deleting_a_profile_removes_its_table_from_disk`, `renaming_a_profile_leaves_no_old_name_on_disk`, `renaming_a_var_key_leaves_no_old_name_on_disk`, `deleting_a_collection_var_key_removes_it_from_disk`, `deleting_the_last_var_removes_the_vars_table`), with surviving keys' comments asserted intact.
- **Editor is a self-contained modal in the `churl` crate; profile CRUD lives in the editor's working state, not core** (M7.3) — all new UI state (`EnvEditorState` and friends) lives in `components/env_editor.rs`; `churl-core` gained nothing UI. Profiles are created/renamed/deleted purely in that working state and persisted by rebuilding `Workspace { name, vars, profiles }` and calling `save_workspace_manifest` — no profile-CRUD helpers were added to core (the design's permitted-but-not-required option), keeping the resolver/persistence seams untouched (M9 guardrail).

Trigger: `<leader>v` (`v` was free) + palette command; both go through one `Action::OpenEnvEditor`.

This stays inside the editor rather than threading through `Mode::Confirm`/`pending_load` (that seam is endpoint-load specific), per the design.
- **Precedence display is computed relative to the selected scope's context** (M7.3, the headline feature) — the resolver's collection layer is per-endpoint, so a single global "winner" for a var name is ambiguous across collections. Resolution rule for the annotation: build the chain from cli vars, the **active** profile only, the **selected collection** (only when a collection scope is being viewed), and workspace, in Resolver precedence order (cli > profile > collection > workspace). A row is tagged ` ✓` when the selected scope is the highest-precedence definer, else ` → <winner>`; the footer shows the full chain (`base_url: profile dev — · workspace ✓`, where `—` means the active profile is shown but does not define the name). Rows of an **inactive** profile are tagged ` (inactive)` (that environment never resolves). The asterisk is workspace-only: a profile/cli winner outranks collections, so it stays a precise `✓`, and the collection-scope view is already exact.

- **Active profile applied on save** (M7.3, minor deviation from the design's "on save/close") — `x` changes the editor's `active_profile` mirror for the precedence display; the app's real active profile is updated only when the edits are **saved** (a discard/close leaves it untouched). Active profile is session-only state, but coupling its persistence to save keeps "nothing takes effect without `w`" consistent and avoids applying a profile the user then discarded.
- **Editor modal keys are hardcoded (not `KeyMap`-remappable), like the other overlays** (M7.3, accepted deviation) — the design asked the footer hint to be built "from the actual bindings, not a stale string". The env-editor keys (`j/k/a/d/r/n/x/w/q`, edit-mode `enter/tab/esc`) are modal-internal, exactly like Search/Palette/MethodMenu keys, and are not in the user-remappable `KeyMap`. ## 2026-07-08 (M7.4 — request sequences / API E2E testing)

- **Sequences persist as `sequences/<slug>.toml`, a reserved workspace dir (not a collection)** (M7.4) — one file per sequence, `seq`-ordered, mirroring the file-per-endpoint philosophy. `persistence::SEQUENCES_DIRNAME = "sequences"` is excluded from `OpenWorkspace::collections()` (same spirit as skipping `folder.toml`/`churl.toml`), so a sequences dir never surfaces as a fake collection (regression test `sequences_dir_is_not_a_collection`). `Sequence { seq, name, on_error, steps: Vec<SequenceStep> }` serializes steps as `[[step]]` with a nested `[step.extract]` map; `SequenceStep { seq, endpoint, extract }` where `endpoint` is a workspace-relative path. CRUD seams mirror the endpoint ones (`load/save/create/rename/delete_sequence`, slug + `next_seq`, empty-name refusal); no secrets gate is needed — a sequence file holds endpoint refs and extraction *expressions* (var names → expressions), never secret values. `OpenWorkspace::sequences() -> SequenceLoad { sequences, warnings }` uses the M7.3 lenient posture (one unparseable file degrades to a warning, never aborts).
- **Extraction is a documented, dependency-free subset over `serde_json` — no JSONPath crate** (M7.4) — `sequence::extract_value(&Response, expr)` supports exactly: `status` (numeric status as string), `header:<Name>` (case-insensitive header lookup, error if absent), and a JSON path `$.a.b[0].c` / `a.b[2]` (leading `$.` optional) parsed by a hand-rolled byte scanner (breaks only on ASCII `.`/`[`/`]`, so unicode keys are safe and it never panics). Leaf coercion: JSON string → its value, number/bool → display, object/array → compact JSON; a `null` leaf, missing key, out-of-range index, type mismatch, non-JSON body, or malformed expression are each a distinct `ExtractError`. Rationale: a JSONPath crate is a heavyweight dep for a testing tool's needs; the subset covers the real chaining cases (tokens, ids, headers) and stays inside the "no new heavyweight deps" rule. Any single extraction rule failing fails the whole step (subject to `on_error`).
- **The run engine lives in `churl-core`; the resolver stays the single seam via a prepended "extracted" scope** (M7.4, the M9 plugin guardrail) — `sequence::prepare_step` builds the resolver in canonical order with one extra scope **prepended at highest precedence**: `extracted > cli > profile > collection > workspace` (then process env). Resolution is never forked — it is the same `Resolver`/`substitute_request` seam every other send uses, with one ephemeral scope. **Extracted wins** so a chained value is never shadowed by a same-named ambient var (wiremock-proven by `extracted_var_beats_workspace_var`); documented precedence choice. The collection scope is loaded fresh per step from *that step's* endpoint's collection, so a multi-collection sequence resolves each step against its own collection vars.
- **Shared run seams — what's identical vs what the TUI mirrors** (M7.4; wording corrected in the fix round) — the core exposes three shared seams both `run_sequence` and the live TUI driver call: `classify_response(&Response, &SequenceStep) -> (StepResult, extracted)` (status ≥ 400 → `Failed`; else run extraction → `Ok`/`ExtractError`), `should_halt(&StepResult, OnError) -> bool` (the halt/continue decision), and `prepare_step`/`extract_step`/`ordered_steps`. **What the seam does *not* cover, and the TUI therefore mirrors, is exactly two trivial lines**: the transport-error → `StepResult::HttpError(string)` mapping (the sequence `AppMsg` carries `Result<Response, String>`, the error stringified at the task boundary like the normal `AppMsg::Response`, so the TUI can't hand core an `HttpError` value) and the skipped-tail marking on halt. `run_sequence`'s convenience `classify_step(&Result<Response, HttpError>, …)` wraps `classify_response` for the value-typed error it *does* have. Drift is guarded by the cross-check test `sequence_transition_matches_core`: four scripted scenarios (all-ok+extract / halt-on-500 / continue-past-500 / extract-error-halts) run through the real `run_sequence` (wiremock HTTP) **and** the App's `on_sequence_step` driver (injected outcomes) and must produce the same per-step `StepResult` sequence + same extracted map. Earlier wording ("one classify seam") overstated the sharing — corrected here.
- **Step endpoints cannot escape the workspace root — lexical *and* canonical** (M7.4; hardened in the fix round) — `resolve_step_path` rejects `..`/absolute/drive-prefix components lexically (fast pre-check), **then** canonicalizes the deepest existing ancestor of the resolved path (following symlinks) and requires it to stay under the canonical workspace root. This closes an in-workspace **symlink escape** (`endpoint = "link/secret.toml"` where `link/` is a symlink to an outside dir) that the purely lexical check missed — the earlier "a step can never escape the workspace root" claim was false for symlinks and is now true (test `resolve_step_path_rejects_symlink_escape`, unix). A *missing* endpoint is deliberately **not** a traversal: its deepest existing ancestor is a real in-workspace directory, so it resolves inside the root and the load simply fails not-found — keeping traversal and not-found distinct failure kinds. Batch/sequence history is an M7.5 concern (concurrent-runs milestone already owns "record batch runs without flooding per-endpoint history").

## 2026-07-08 (M7.5 — concurrent requests / load testing) — closes the M7.1–M7.5 block

- **Bounded concurrency via `futures`' `buffer_unordered`, not a manual `FuturesUnordered` + `Semaphore`** (M7.5, deviation from the design's literal wording) — the design specified "a `FuturesUnordered` bounded by a `tokio::sync::Semaphore`". A semaphore-guarded `FuturesUnordered` needs the launch loop to *also* drain the set while waiting for a permit (otherwise the permit-acquire await deadlocks, since permits only free as in-flight futures complete). `futures::stream::iter(0..total).map(execute-copy).buffer_unordered(concurrency)` gives the **identical** guarantee — at most `concurrency` request futures are polled at once — with no manual permit bookkeeping and no deadlock foot-gun, and it *is* a `FuturesUnordered` internally. The non-negotiable ("bounded concurrency must actually bound") is met and **proven** by a wiremock test that records each request's arrival timestamp and computes the peak count of overlapping `[arrival, arrival+hold]` intervals, asserting it never exceeds the cap (and is ≥ 2, so the bound is meaningfully exercised). Both `run_load` and the live TUI launcher use this same construct, so they cannot drift.
- **Cancel = abort the ONE launcher task; no detached per-request `tokio::spawn`** (M7.5, the batch-cancel non-negotiable) — the TUI fires the whole batch from a *single* `tokio::spawn`ed launcher that owns the `buffer_unordered` stream; its `AbortHandle` is `App::load_abort`. Aborting it drops the stream, which drops every in-flight reqwest future (reqwest cancels on drop) — that is the entire batch-cancel mechanism. There is deliberately no per-request spawn that could outlive the abort. Proven live by `load_cancel_aborts_the_batch_live` (multi-thread wiremock, concurrency 2, slow server): a cancel right after launch means the server sees only a handful of the 20 copies — the un-launched ones never fire. A per-run `run_generation` (bumped on cancel/re-run/close) additionally drops any straggler result that lands after a cancel.
- **Absolute-target pacing (not sleep-since-previous)** (M7.5) — `interval > 0` paces launches by making copy `i` sleep until `start + i·interval` before calling `execute`. Because `buffer_unordered` polls futures lazily (a copy is only polled once a concurrency slot frees), a naïve "sleep `interval` from when polled" would compound into a super-linear cumulative delay under low concurrency. The absolute target is a hard floor: copy `i` never launches before `i·interval`, so the run's wall-clock is `≥ (total-1)·interval` regardless of scheduling — which satisfies (and exceeds) the design's loose `≥ (ceil(total/concurrency)-1)·interval` assertion without flake. `interval == 0` is burst.
- **Load runs write ONE summary row to a SEPARATE `load_batches` table — structural, not query-filtered, non-flooding** (M7.5) — migration 3 adds `load_batches` (one row per completed/cancelled run: url, endpoint_path, total, concurrency, ok/fail/error counts, cancelled flag, min/median/p95/max ms). A load run **never** inserts into `history`, so the per-endpoint history view cannot see batch rows *by construction* — not by a query filter that a future change could forget (test `load_batches_never_appear_in_per_endpoint_history`). Mirrors the M7.4 decision that sequence steps write no history; M7.5 owns "record batch runs without flooding".
- **`cancelled` column added to `load_batches` (deviation from the design DDL)** — the design's DDL listed the count/percentile columns but not a cancelled marker, yet §1d/§2 both require persisting "a cancelled run's partial summary — mark it". A boolean `cancelled` column is the honest place to mark it (the partial counts are real, but the run did not complete). `LoadBatchSummary.cancelled` round-trips it.
- **Nearest-rank percentiles, over completed-request timings only** (M7.5) — `stats()` computes min/median/p95/max by the **nearest-rank** method (`rank = ceil(p/100 · n)`, 1-based) on a sorted copy of the timings of requests that actually completed (`Ok`/`Failed` carry a timing; `Error` does not); `median` is nearest-rank p50, mean is the arithmetic mean. Empty / all-errored input yields `None` for every latency field (counts still populate) — no divide-by-zero, no panic. Exact vectors are unit-tested (odd/even/single/empty/all-errored). Chosen over interpolated percentiles for simplicity and because nearest-rank is unambiguous and matches "the p95 is an actual observed latency".
- **Guardrail: hard-cap refuse + warn-cap loud confirm naming the URL, `[load]`-overridable** (M7.5) — `check_config(cfg, caps)` returns `Refuse` above a hard cap (`max_total` 10 000 / `max_concurrency` 200), `Warn` above a warn threshold (`warn_total` 100 / `warn_concurrency` 20), else `Ok`; refuse beats warn, and `total == 0` is not a breach (it is a harmless no-op). The runner refuses with a message (no run), or renders a loud confirm (`prompt::render_confirm`) that names the count, concurrency, and **target URL** (`Fire 500 requests at concurrency 5 against https://…?`) — `y` fires, `n`/esc aborts. Caps are overridable per-user via a `[load]` config table folded over the defaults. This makes "testing aid, not a load-cannon" real, not advisory.
- **`LoadStarted` launcher message so a running copy shows its glyph** (M7.5, minor enhancement over the design's message set) — the design's live message was completion-only (`LoadResult`), which would leave every row jumping `Pending → terminal` with the `◐` running glyph never appearing live. The launcher now also sends `AppMsg::LoadStarted { run_generation, index }` the instant a copy enters the concurrency window (right before `execute`), so its row flips to `Running` honestly. One extra bounded message per request; generation-guarded like `LoadResult`.
- **Request resolved ONCE at open, cloned for all N copies** (M7.5) — opening the runner builds the request exactly like an interactive send (endpoint clone + fold in the live body editor + `build_resolver`/`substitute_request`) and stores the resolved `Request` on `App::load_request`; every copy in every (re-)run clones *that* — no per-copy re-resolution, so the whole batch is guaranteed identical (same URL/vars/auth). `execute` stays substitution-free.
- **`tokio` + `futures` promoted to `churl-core` direct dependencies** (M7.5) — `run_load` needs `buffer_unordered` (`futures`) and, for pacing, `tokio::time::sleep`; both were already transitive (reqwest pulls tokio; `futures` was a `churl` dep). They are used *only* inside `run_load`'s body — `execute` keeps its runtime-agnostic signature (no tokio types), so the "core is UI-free and `execute` is runtime-agnostic" architecture invariant is preserved. Recorded because it moves two crates from dev-deps / transitive to direct core deps.
- **Runner state is UI-only; `App` owns execution — mirrors the M7.4 split** (M7.5) — `components/load_runner.rs` holds only display state (config, result rows, stats, focus, the config-field `LineEditor`) + the key state machine; `App` owns the client, the launcher + abort handle, the generation guard, the caps, and the resolved request, and drives the run through the shared `churl_core::load` primitives. Under snapshot-test construction (`client == None`) `start_load_run` resets the rows and returns without spawning, so the state machine is exercised by injecting `on_load_started`/`on_load_result`, and render snapshots are deterministic (no highlight worker). Modal keys are hardcoded (not `KeyMap`-remappable), like every other overlay; only the entry `Action::OpenLoadRunner` (`<leader>l` + palette) is in the keymap.
- **Results list renders O(viewport), not O(N)** (M7.5) — the list keeps a scroll offset and builds only the visible slice of rows each frame, so a 10 000-copy batch renders as cheaply as a 10-copy one. Stats are recomputed from the accumulated outcomes as each result lands (network-paced, so the recompute cost is amortised across the run).

## 2026-07-09 (R0 — cheap-P0 durability)

- **Atomic saves via one `persistence::atomic_write` helper** (R0) — every workspace writer (`save_endpoint`/`save_sequence`/`save_workspace_manifest`/`save_collection_meta`) funnels through `save_value`, whose sole `std::fs::write(path, output)` truncated the source-of-truth file in place — a crash or power loss mid-write destroyed the file (the sweep's second catastrophic P0). Replaced that one call with `atomic_write(path, bytes)`: write to a **sibling temp file in the same directory** (`.<file_name>.<pid>.<counter>.tmp` — a per-process `AtomicU64` counter keeps concurrent writers from colliding on one temp name; same filesystem so `rename` is atomic) → `write_all` → `sync_all` (fsync data) → `std::fs::rename` (atomic replace) → **fsync the parent directory** so the rename itself survives power loss. Directory fsync **degrades gracefully** — opening a dir for fsync is unsupported on some platforms (notably Windows), so a failure there is ignored rather than failing an otherwise-good save. On any error the temp file is removed best-effort and the original error is returned. **Error contract unchanged**: `save_value` still maps the helper's `io::Error` to `PersistenceError::Write { path, source }`, so every caller and test is untouched. The format-preserving merge (comments/ordering) is unaffected — it produces the same bytes, they just land durably. Tested by injecting a temp-create failure (read-only parent dir, unix) and asserting the pre-existing good file is byte-intact with no stray temp left; the byte-identical round-trip corpus still passes.

It flags **only** genuine defects — documented intentional single-pane overlay shadows (Response `h`=ToggleHeadersView shadowing global Collapse, Response `/`, Request `1`–`4`) produce **zero** warnings: (a) the leader key also bound as a normal global/pane action (the dead `space=` case — the normal bind can never fire); (b) a `"+name"` descent whose `[keys.leader.name]` table does not exist (dangling menu); (c) a `[keys.leader.name]` table no descent points to (orphan/unreachable menu); (d) two combo *strings* normalizing to the same key within one scope (silent last-wins); (e) a global bind shadowed in **every** pane overlay (globally unreachable — only trips when all four overlays rebind the same combo to something else, so a single-pane shadow never fires it). Surfaced in **three non-blocking channels**: `eprintln!` at startup **before** raw mode (`tui::run`, visible in the launching shell), a **first-frame statusline toast** `⚠ N keymap issue(s) — run churl keymaps` (stored on `App::keymap_warnings`, fired once by `App::run` via `arm_keymap_warning_toast`), and a trailing `⚠ Conflicts` section in `churl keymaps`. - **Load-runner response retention bounded to O(concurrency + K), dropped rows are not reconstructable** (R0) — `on_load_result` builds a `ResponseView` per OK copy (the body decode/line-index) and stored `ResponseState::Done { view }` on **every** row, so a high-`total` run retained `total × body` → OOM (the sweep's first catastrophic P0). Fix keeps the bloat bounded in the UI-only `LoadRunnerState`: a `live_views: VecDeque<usize>` tracks (in completion order) which rows currently hold a live `Done` view; on each new retained view, `enforce_view_window` keeps at most **K = `LIVE_VIEW_WINDOW` = 16** rows live **plus the currently-selected row** (never evicted — checked *after* the follow-selection update so the just-selected newest row is protected), downgrading the oldest non-selected retained row to a new **`ResponseState::Dropped { status, timing, size }`**. The response viewer renders `Dropped` honestly (status line `{status} · {timing} · {size} · not retained`, body "response body not retained (memory-bounded)") — it is a status-only placeholder, **not** a lazily-rebuilt view: we deliberately keep **no body bytes** for evicted rows, so a dropped row's body is gone and cannot be shown again. This is the accepted load-test tradeoff (you cannot inspect every one of thousands of bodies; you get status/timing/size for all of them, full bodies for a sliding window + whatever you're looking at). Per-row status/timing and the `outcomes`-driven stats are untouched, so aggregate correctness is independent of eviction. Two new `ResponseView` accessors (`status()`, `body_len()`) feed the placeholder. When the selection moves off a pinned row it becomes evictable again but is **not** proactively rebuilt (it may already be dropped — fine). ## 2026-07-10 (M7.7 Stage A — response body reformatter)

- **Reformatter v1 scope is JSON-only, via `serde_json::to_string_pretty`** (M7.7 Stage A, owner decision) — the pinned M7.7 deliverable named JSON + HTML + XML, but v1 ships **JSON only**. JSON is the dominant API content-type and `serde_json` is already a workspace dependency, so pretty-printing adds **zero new dependencies** and no new code paths beyond a parse-then-reprint (`reformat_body_if_needed` in `components/response.rs`: `serde_json::from_str::<Value>` → `to_string_pretty`). HTML/XML reformat is deferred (a later M7.7 stage or follow-up); a structured HTML/XML pretty-printer would need a new dep and its own error model, out of scope for this stage. The single content-type→syntax mapping point (`SyntaxToken::from_content_type`) still decides whether to reformat, so no per-format fork was introduced — only `SyntaxToken::Json` bodies are reformatted.
- **Pretty-by-default for json-ish content-types; malformed JSON silently falls back to raw** (M7.7 Stage A) — a `application/json` (or any json-ish content-type per `from_content_type`) body arrives **pretty** (`ResponseView::pretty` defaults `true` for `SyntaxToken::Json`, `false` otherwise). This fixes the reported bug: petstore-style minified single-line JSON previously rendered on one line (the viewer line-split on server `\n` only and never re-indented). If the body is not parseable JSON — despite the content-type — `reformat_body_if_needed` returns the original text unchanged: **never errors, never panics, never shows a diagnostic** (a viewer must always render *something*). So `pretty` can be on while the displayed text equals the raw body (a body **truncated** at the size cap is unparseable, so it renders raw and keeps its truncation marker). The pretty output **preserves the server's on-the-wire key order** — `serde_json`'s `preserve_order` feature is enabled at the workspace level (owner-approved), so `to_string_pretty` re-emits object keys in insertion order rather than sorting them alphabetically (sorted keys would misrepresent the response structure the user is inspecting). Feature unification means churl-core picks it up too: this also changes `interchange.rs` Postman export from alphabetical to insertion order (`method → header → body → auth → url` — the `.insert()` sequence is the new canonical order); accepted and expected. No test pinned the old alphabetical byte layout, so no export snapshot needed updating.
- **Raw↔pretty toggle on `p`; toggling resets folds and invalidates the highlight cache** (M7.7 Stage A) — a new `Action::TogglePretty` is bound to `p` in the `[keys.response]` overlay (`p` was free there; the global `<leader>p` switch-profile lives behind the leader, not this pane overlay) and added to the command palette (context-free, like `ToggleWrap`/`ToggleHeadersView`). `ResponseView::toggle_pretty` flips `pretty`, rebuilds the displayed `text` + `line_offsets` from the raw body, **bumps `generation`** (the syntect viewport cache is keyed on generation, so the bump invalidates stale highlights on toggle) and additionally hashes `pretty` into `viewport_hash` (belt-and-braces), and **resets folds** (`folds = None`, `folded.clear()`) — fold openers are position-based line indices and the raw↔pretty layouts differ, so a carried-over fold set would point at the wrong lines. The app handler resets cursor/scroll and clears the highlight cache (same as fold-all). Outside a JSON body view the handler no-ops with a `pretty: JSON body only` notice rather than silently doing nothing.
- **Copy (`y`/`Y`) always reads the raw on-the-wire bytes, never the reformatted text** (M7.7 Stage A) — `ResponseView` keeps a separate `raw_text` field holding the exact lossy-UTF-8 decode of the response body, never reformatted. `copy_all` (`y`, copy body) returns `raw_text` in the Body view (the headers text in Headers view), so a copy is byte-exact regardless of the pretty toggle — pasting round-trips the server's actual bytes. The displayed `text` (pretty when applicable) drives only rendering, folds, wrap, search and line navigation. This is a correctness requirement: a "copy response" that emitted reformatted-and-key-sorted JSON would silently corrupt what the user believes they copied. `copy_line` (`Y`) copies the logical line under the cursor as displayed (the visible line the user is pointing at). **Side effect on the enum-size lint**: adding `raw_text: String` pushed `ResponseState::Done` past clippy's `large_enum_variant` threshold; boxing the hot per-render view (read every frame) would add an allocation + indirection and ripple through ~20 match sites, so the lint is silenced with a scoped `#[allow(clippy::large_enum_variant)]` + rationale comment on `ResponseState`.
- **Optional A→Z key-sort toggle on `s`; default stays wire order; ephemeral per response** (M7.7 Stage A follow-up, owner-approved) — Stage A's pretty view preserves the server's on-the-wire key order (the `preserve_order` decision above). This adds an **optional** alphabetical sort on top, as an explicit opt-in — **wire order remains the default** because it faithfully mirrors the response the user is inspecting; sorting is a convenience for scanning large objects, never imposed. A dedicated `Action::ToggleSortKeys` is bound to `s` in the `[keys.response]` overlay (`s` confirmed free there — the leader `s` sequences-submenu lives behind the leader, not this pane overlay) and added to the command palette. `ResponseView::toggle_sort_keys` mirrors `toggle_pretty`: it flips a `sort_keys` flag (defaults **false**, resets per response — nothing persisted to config), rebuilds `text`/`line_offsets` through the same `reformat_body_if_needed(text, syntax, pretty, sort_keys)` path, bumps `generation`, resets folds, and clears search (all geometry-changing side effects, identical to `p`). `sort_keys` is also hashed into `viewport_hash` (belt-and-braces). The sort is **recursive** (`sort_value_keys` walks the parsed `serde_json::Value`, rebuilding every `Object`'s entries in `String::cmp` order and recursing; **arrays keep element order**, only object keys sort) and works precisely because `preserve_order` is on — the rebuilt insertion-ordered map re-emits in the sorted order we wrote. It is **independent of `pretty`**: sort only affects the displayed text while `pretty` is on and the body is JSON; the app handler guards on pretty-JSON-body and otherwise no-ops with a `sort: pretty JSON only` notice (mirrors the `pretty: JSON body only` guard). **Copy (`y`) is unaffected** — it still returns `raw_text`, byte-exact, regardless of sort/pretty state, so a sorted display never changes what gets copied.

## 2026-07-10 (M7.7 Stage B — help-overlay `/` search)

- **Help `/` is highlight-and-jump, reusing the response body-search engine — rows are never hidden** (M7.7 Stage B, owner decision 5) — `/` inside the `?` help overlay opens an incremental search that **highlights matches in place and jumps to the current one**, exactly like the response viewer's `/`; it is **not** a filter that hides non-matching rows. The full gesture set mirrors body search: `/` opens an input, typing recomputes matches live and scrolls the current one into view, `n`/`N` cycle matches (wrapping), Esc cancels (clears the search, overlay stays open), Enter commits (matches retained for `n`/`N`, input closed). Matching is **smart-case** (all-lowercase query = case-insensitive; any uppercase char = case-sensitive). The overlay title carries the live query + a `k/N` counter; match spans are painted with the same styling as the body search (current match reversed, other matches dim+underlined). Rationale for highlight-and-jump over filtering: consistency with the one search UX the user already knows, and a filtered help list would hide the surrounding section context that makes a binding legible.
- **One shared smart-case matcher + one shared span painter — the two searches cannot fork** (M7.7 Stage B) — rather than replicate the smart-case algorithm, the body-search matching logic was **extracted** from `ResponseView::compute_matches` into a free `pub(crate) fn smart_case_matches(lines, query) -> Vec<(line, start, end)>` in `components/response.rs`; `compute_matches` now delegates to it (feeding its own logical lines), and the help search calls it over the help overlay's flattened **plain-text** lines. The match-span painter `build_search_spans` (current=reversed, others=dim+underlined) was likewise made `pub(crate)` and reused by `help::render`. This guarantees byte-identical matching + highlight semantics across both surfaces — a change to smart-case behaviour is a one-place edit, and the parity is asserted by a test that drives the *shared* helper directly. The unicode case-fold offset mapping (`lowercase_with_offsets`/`map_lowered_offset`) rides along unchanged inside the shared helper.
- **Help search lives on `App`, analogous to body search but self-contained to the overlay** (M7.7 Stage B) — three fields on `App`: `help_search: Option<help::HelpSearch>` (query + matches + current index — the help analogue of `response::SearchState`), `help_search_editor: LineEditor` (the incremental input), and `help_search_input: bool` (input-open flag; Enter clears it but keeps `help_search`). Key routing stays inside the existing `handle_help_key` (the overlay intercepts keys ahead of the mode machine, so no new `Mode` variant is needed): while `help_search_input`, keys delegate to `handle_help_search_key` (mirrors `handle_body_search_key`); otherwise `/` opens the search, and `n`/`N` step when a committed search exists. `HelpSearch::set_query` rebuilds the help lines from the live keymap on each keystroke (help content is cheap and keymap-derived, so there is no cached line buffer to invalidate) and recomputes matches; `help_center_on_match` nudges `help_scroll` to keep the current match within the last-rendered viewport. Closing the overlay (`?`/Esc/`q`) clears all help-search state.
