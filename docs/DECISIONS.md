# churl — Decision Log (ADRs)

Entries are append-only; one line per decision, date-prefixed. Superseding decisions reference the one they replace.

---

## 2026-07-05

- **Name: churl** — "curl" in a TUI shell; short, memorable, not taken on crates.io at decision time.
- **Default branch: master** — matches the host platform's default; no organisational convention overrides it for this personal project.
- **HTTP client: reqwest 0.13 + rustls** — async-native, pure-Rust TLS, no OpenSSL dep. Coarse timing (connect + total) is sufficient for M0–M3; a custom hyper connector (DNS/connect/TLS split timing) is documented as the later upgrade path when fine-grained waterfall data is wanted.
- **TOML library: toml_edit** — format-preserving; users edit endpoint files by hand and expect their comments and ordering to survive a round-trip through churl. `toml` (serde-only) would silently destroy formatting.
- **File-per-endpoint + explicit `seq` field** — learned from Bruno: monolithic collection files (Slumber, Insomnia) create merge conflicts and make git history noisy. One file per endpoint scales to large collections; `seq` provides stable ordering without relying on filesystem order.
- **Templating: hand-rolled `{{var}}` substitution** — no JS runtime (ATAC carries a ~10 MB binary bloat for Deno); no Tera/Handlebars (overkill for URL/header/body variable injection). Precedence: CLI `--var` flag → named profile → process environment. Secrets are process-env only; never written to synced files.
- **Secrets hygiene: never in workspace files** — workspace = a git repo users commit and sync. Secrets go in process env or a local `.env` file the user adds to their `.gitignore`. churl never reads, writes, or suggests committing secrets.
- **Local state: rusqlite (bundled)** — history, cookies, and cached sessions are high-churn and relational. `sled` (considered) was abandoned upstream. SQLite with bundled compilation avoids a system dep and is universally available. Schema migrations run at startup.
- **Syntax highlighting: syntect + two-face, lazy, off-thread, viewport-only** — ~23 ms per viewport is the measured cold-start risk; background thread + viewport hash cache keeps the render loop smooth. Whole-document highlighting of a 1 MB response is never done.
- **Vim-modal editing: edtui** — `tui-textarea` has been unmaintained since 2024; `edtui` is actively developed and ships modal vim bindings out of the box.
- **Keybindings: crokey + semantic Key→Action map** — follows gitui/helix model; users remap actions, not raw keys. Prevents keybinding spaghetti as the feature set grows.
- **Fuzzy search: nucleo** — same engine as Helix; outperforms skim/fuzzy-matcher at scale; async-native.
- **HTTP mocking in tests: wiremock** — spins a real HTTP server in-process; tests exercise the full reqwest stack without network access.
- **Snapshot tests: insta + ratatui TestBackend** — deterministic pixel-level assertions for TUI rendering; catch accidental layout regressions.
- **No HTTP-semantic caching** — churl is a development/testing tool. Returning a cached response when the user hits "send" is a footgun. History (SQLite) covers the recall use case.
- **Async runtime: tokio** (arrives M2) — not added in M0. Plain synchronous `crossterm::event::read()` is sufficient for the placeholder TUI; the full `tokio::select!` event loop comes with request execution in M2/M3.
- **Persistence: serde reads, hand-rolled merge writes** — loads use `toml_edit::de` into serde types; saves serialize to a fresh document (normalized to `[[array-of-tables]]`) and merge it into the existing file's `DocumentMut`: unchanged values untouched, changed scalars keep the old decor, stale keys removed, equal-length arrays-of-tables merged element-wise (unequal → replaced, comments lost). Saving canonicalizes explicit `enabled = true` away.
- **SQLite migrations: `PRAGMA user_version`** — no migrations table; each migration runs in its own transaction that bumps `user_version`. Simple, idempotent across reopens, and rollback-safe per step.
- **Platform paths: `dirs` crate** — config at `<config_dir>/churl/config.toml`, state at `<data_dir>/churl/state.sqlite`; hand-rolling XDG-vs-macOS-vs-Windows resolution is not worth the bugs.
- **Secrets enforcement: name-marker heuristic at save time** — `save_workspace_manifest` refuses profile variables whose name contains a secret marker (token/secret/password/api_key/…) with a literal value; `{{...}}` template placeholders pass. Loads never refuse (it's the user's file); only churl-initiated writes are gated.
- **Fuzzy engine: `nucleo-matcher`, not `nucleo`** (refines "Fuzzy search: nucleo" above) — same scoring engine, synchronous API. The high-level `nucleo` crate's threaded streaming matcher is overkill for workspace-sized endpoint lists and would drag a worker-pool lifecycle into the TUI.
- **Key routing precedence** — (1) an open overlay (search/palette) consumes every key; (2) request pane focused with edtui in a non-Normal mode → all keys to edtui; (3) otherwise crokey `KeyMap` first, unmapped keys fall through to edtui when the request pane is focused, and mapped navigation keys forward their original event to edtui there. edtui owns insert/visual modality — the app never re-implements vim.
- **Keybinding overrides: flat `[keys]` string map in `churl-core::config`** — core carries only `"combination" = "action-name"` strings (keeps core TUI-free); the `churl` crate parses both sides at startup and fails loudly on unknown entries rather than silently skipping them. Overrides layer on top of the `key!`-built defaults.
- **Explorer laziness matches persistence laziness** — startup lists collection directories only; endpoint files parse on first expand or on search-overlay open, protecting the < 100 ms cold-start budget.
- **BackTab default binding built explicitly** — crossterm always reports Shift-Tab as `BackTab` + `SHIFT`, and crokey's `key!(backtab)` does not add the modifier (its string parser does); the default map constructs `KeyCombination::new(BackTab, SHIFT)` directly.
- **Task-level cancellation, not core-level** — `churl-core::http::execute` is a plain runtime-agnostic `async fn` with no `AbortHandle` in its signature. The TUI spawns it with `tokio::spawn` and keeps the `AbortHandle`; a monotonic `generation` counter on `App` tags each request, and a landed `AppMsg::Response` whose generation ≠ the current in-flight generation is dropped (stale result after cancel+resend). Ctrl-C cancels an in-flight request instead of quitting; `q`/`Esc` always quit. This keeps core free of any runtime dependency (refines the M0 "AbortHandle per in-flight request" wording, which lived in core in the original plan).
- **reqwest `rustls` feature, not `rustls-tls`** — reqwest 0.13 renamed the pure-rustls feature to `rustls` (with the connector selected via `ClientBuilder::tls_backend_rustls()`); the M0 decision's `rustls-tls` name was the 0.12 spelling. Same intent (pure-Rust TLS, no OpenSSL), current name.
- **No `{{var}}` templating in M3** — request execution uses the URL, headers, and body verbatim. Templating (the `{{var}}` precedence chain) is an M5 deliverable; wiring it into execution before the substitution engine exists would be dead code.
- **No response body-size cap in M3** — the full body bytes are read (`.bytes()`) and decoded lossily for the viewer; giant responses can bloat memory. Deferred to M6 (open question in MILESTONES) rather than guessed at now.
- **Response viewer: no wrap, truncate at pane width** — the virtualised viewer slices only the visible lines out of a line-offset index and lets ratatui truncate long lines at the pane edge. Line wrapping and horizontal scroll are deferred (open questions in MILESTONES); virtualisation (never materialising all lines) is what protects the 1 MB-body budget, and wrapping would complicate the offset math without buying that.
- **Viewport-stateless syntax highlighting** — the off-thread worker starts a fresh `HighlightLines` at the top of each viewport (highlighting only the visible lines, cached by a viewport hash of generation/scroll/height/syntax). This is a known imperfection: multi-line constructs (block comments/strings) that open above the viewport are mis-coloured. Accepted for the render-loop budget it buys; whole-document highlighting of a 1 MB response is never done. Foreground RGB only (background skipped so the theme doesn't fight the pane); theme is two-face's Nord.
- **History written on every terminal outcome** — success, failure, and cancel each insert a `NewHistoryEntry` synchronously in the event handler (a fast local SQLite write; acceptable on the UI thread). `status` is `Some(code)` only on success; `duration_ms` comes from response timing (`None` on cancel/failure). Insert failure warns on the statusline, never crashes; a failed history *open* disables history for the session (also non-fatal).
- **Body-size cap in M4** (supersedes "No response body-size cap in M3") — stream the response body with a default 10 MB cap, config-overridable (`max_body_bytes`); `Response` gains a `truncated` flag surfaced in the status line. Owner decision at the post-M3 plan review: it is a safety hole (accidental giant download freezes/OOMs the TUI), not polish, so it lands with M4's core work rather than waiting for release polish.
- **Auth is its own milestone (M5), not part of M4** — owner decision at the post-M3 plan review. M4 imports `-u`/`Authorization:` as plain headers; M5 adds the minimal auth model (basic/bearer/API-key) and remaps. Keeps M4 lean at the cost of one remap step. Milestones renumbered: themes/templating → M6, polish/release → M7.
- **Headers view + wrap toggle land in M7 polish** — resolves the M3 open questions. Wrap toggle chosen over horizontal scroll (wrap covers the read-the-long-line need; horizontal scroll adds offset state for little gain). Status-line header count suffices until then.
- **Insert-mode exception for Ctrl-S/Ctrl-C** (refines "Key routing precedence") — from M4, Ctrl-S (send) and Ctrl-C (cancel/quit) are intercepted *before* edtui even in insert/visual mode, making edit→send one keystroke. The only exception to "edtui owns non-Normal modes": both are control-modified non-text keys, so no editing capability is lost.
