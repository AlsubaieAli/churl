# churl — Decision Log (ADRs)

Durable architectural decisions (why the code is shaped this way). Process archaeology (drive-test rounds, maintainer notes, bug root-causes) lives in git history, not here.

Organized by area; each bullet is one durable decision. A decision that reverses or supersedes an earlier one says so inline.

## State model & module structure

- **Illegal states unrepresentable — mode-owned state lives IN its `Mode` variant.** Each modal overlay's owned state lives in its `Mode` variant, not in a parallel `Option<T>` field: `Mode::EnvEditor(EnvEditorState)`, `Mode::LoadRunner(LoadRunnerState)`, `Mode::Sequence { view, editor, runner }`. The `(Mode::X, None)` illegal state ("active editor with no data") is unconstructable; opening a mode is a single transition and closing (`Mode::Normal`) drops the state; the defensive `is_none()→Normal` guards and their `.expect()`s are gone. Cost: `Mode` is non-`Copy`/non-`PartialEq` (a discriminant-only `PartialEq` would silently equate two different runner states), so comparisons use `matches!` and copies match by reference. `body_search_return: Mode` parks the whole mode (payload included) via `mem::replace` so body-search opened over a runner Response restores the runner, not `Normal`.
- **One shared picker overlay is a data-carrying `enum Picker`.** `picker: Option<Picker>` replaces the old `Option<PickerState>` + parallel item `Vec`s + one-shot `bool`s. Each variant carries its own items + intent alongside the shared `PickerState` finder: `Search { state, targets, after_pick }` / `Palette { state, actions }` / `Profile` / `Workspace` / `Sequence { state, paths, runs }` / `Auth`. The index can only address its own variant's list and a stale intent can't leak across kinds — the wrong-`Vec` / stale-index out-of-bounds hazards are unrepresentable. `PickerState` stays the shared inner field so render + key routing stay kind-agnostic.
- **Extractions target a *child* module of the owning type's module — never a sibling.** Rust privacy is "visible to the defining module and its descendants", so a helper moved into a child of `app` (`app/render.rs`, `app/pure.rs`) keeps full access to `App`'s private fields with zero visibility changes. Extracting to a *sibling* (top-level `tui/render.rs`) would force `pub(crate)` onto ~36 `App` fields — a behaviour/visibility change, not a pure move. Rule for every modularization: if a move seems to need a `pub(crate)` field, it was extracted to the wrong place. `app.rs` became the directory module `app/` with `mod.rs` as the spine (via `git mv` for reviewable rename detection); `render` re-exported so the public `churl::tui::app::render` path is unchanged.
- **`#[non_exhaustive]` on churl-core's error enums and growable-data structs — NOT on behavioral domain enums.** APPLY to every public `thiserror` error type (`ExtractError`, `SequenceError`, `ConfigError`, `HttpError`, `HistoryError`, `PersistenceError`, `ImportError`, `InterchangeError`, `ParseMethodError`) plus the growable `LoadStats` — they grow additively and downstream match-with-catch-all is idiomatic and non-breaking. DO NOT APPLY to behavioral domain enums the TUI matches exhaustively for correctness (`Method`, `Auth`, `BodyKind`, `ApiKeyPlacement`, `OnError`, `LoadCheck`, `ReqOutcome`, `StepResult`, `KeyEntry`) — exhaustive matching there is a safety feature: adding `Method::Connect` should force a compile error at every dispatch site, not fall into a wildcard. Default-to-skip for any ambiguous type.
- **Published result/output/summary structs are `#[non_exhaustive]`; serde model DTOs stay constructible.** A struct the crate *produces and callers only read* — import results, interchange summaries, run outcomes, history rows, secret findings/decisions (`ImportResult`, `ImportSummary`, `ImportedRequest`, `CollectionImport`, `StepOutcome`, `SequenceRun`, `HistoryEntry`, `SecretFinding`, `SecretDecision`) — carries `#[non_exhaustive]` so a future additive field is a patch, not a breaking, change (an additive public field to an exhaustive struct is breaking under semver, which is why a mere field addition once forced a surprise minor bump). The gate is downstream *construction*: apply it only when nothing outside `churl-core` builds the struct via a `Foo { .. }` literal or matches it exhaustively without `..`. Serde model DTOs the binary constructs field-by-field (`Endpoint`, `Header`, `Param`, `Body`, `Request`, `Profile`, `Workspace`, `Timing`, `Response`, config/builder structs) stay exhaustive — as do output types the binary or its tests still literal-construct (`LoadBatchSummary`, written by the load runner; `CookieView`, built in a render test). If adding the attribute forces any downstream construction or match to change, the struct is the wrong target.

## Core architecture & purity

- **`churl-core` is UI-free; `execute` is runtime-agnostic.** Model, persistence, and HTTP live in core; no TUI deps (ratatui/crossterm) ever. `churl-core::http::execute` is a plain `async fn` with no `AbortHandle` in its signature — the TUI spawns it with `tokio::spawn`, keeps the `AbortHandle`, and a monotonic `generation` counter on `App` drops stale results after cancel+resend. `tokio`/`futures` are direct core deps (used only inside `run_load`'s body for `buffer_unordered` + pacing), but `execute` keeps its runtime-agnostic signature, preserving the invariant.
- **Error handling split: `thiserror` in libraries, `color-eyre` in the binary.** `churl-core` (and future `churl-*` libs) use typed `thiserror` errors; the `churl` crate / `main.rs` use `color-eyre` for context-rich reporting.
- **`AuthWire` single-dispatch seam in `churl-core::auth`.** `apply_auth(&Auth) -> AuthWire` holds the one `match` on auth kinds; every kind resolves to a wire *effect* (`Header`/`Query`) and `execute()`/`export_curl` only apply effects, never matching on `Auth`. A future plugin-provided auth kind slots into that single match, not scattered call sites. Injection precedence: an `AuthWire::Header` is injected only when no *enabled* user header with the same (case-insensitive) name exists.
- **Template resolver = ordered scopes + env fallback, the single `{{var}}` seam.** `Resolver` holds a `Vec<Scope>` (named flat maps, highest-precedence first) and falls through to process env last (never snapshotted). `substitute_request` is the one place a `Request` is templated (url, header/param *values*, body, all auth string fields; names never), run at send time on the already-cloned request before `http::execute` — so `execute`/`export_curl` stay substitution-free and resolved values never touch disk. An unresolved placeholder **fails loud** (`unresolved_placeholders` names any `{{var}}` still present) so the three send paths refuse rather than shipping a literal. Plugin template functions extend the lookup here.
- **Variable scoping: `extracted > session > cli > profile > [leaf-collection … root-collection]`, then process env (M7.9).** Profiles are the only place environments are defined; the collection `[vars]` in each `folder.toml` and the root collection's `[vars]` in `churl.toml` carry environment-independent defaults any profile can redirect. **M7.9 collapsed the workspace into the root collection**, so there is no longer a separate `workspace` scope — the single `collection` layer became the endpoint's **ancestor chain**, walked innermost→outermost (leaf collection, its parents, ending at the root collection, whose vars are the old `churl.toml [vars]`). Resolution is **inherit-and-override**: a child collection's var shadows a same-named ancestor's, and a root-level endpoint sees only the root collection's vars. Sequence runs prepend a highest-precedence ephemeral `extracted` scope (chained values never shadowed by an ambient var); the in-memory Session store sits just below it, and `prepare_step` walks the same ancestor chain for its step's endpoint. Resolution is never forked — one `Resolver`/`Scope` seam for both standalone-send and sequence-run (the ancestor scopes all share the `"collection"` diagnostic name and are resolved in order).
- **Plugins ship after 1.0; their guardrail seams are held now.** The plugin API freezes the shapes the feature stream stabilises, so it ships after the 1.0 quality/security sweep — but the seams are kept meanwhile: single auth dispatch (`AuthWire`), single template resolver, data-driven palette commands, one content-type→syntax mapping, and `churl-core` types as the stable plugin-facing API. Runtime tech (Lua/WASM/subprocess) is deferred; the "no JS runtime" decision stands.

## Persistence & durability

- **TOML via `toml_edit`, format-preserving; serde reads, hand-rolled merge writes.** Users edit endpoint files by hand and expect comments/ordering to survive a round-trip; `toml` (serde-only) would destroy formatting. Loads use `toml_edit::de` into serde types; saves serialize to a fresh document and merge into the existing `DocumentMut` — unchanged values untouched, changed scalars keep decor, stale keys removed. Saving canonicalizes explicit `enabled = true` away.
- **File-per-endpoint + explicit `seq` field.** Monolithic collection files create merge conflicts and noisy git history; one file per endpoint scales, and `seq` gives stable ordering without relying on filesystem order. Sub-collection metadata lives in `folder.toml` (`CollectionMeta { vars }`); the **root collection's** meta is `churl.toml` (`Workspace { name, vars, profiles }`) — M7.9 folded the workspace into the root collection, so there is no separate workspace tier. Endpoints attach to any collection node **including the root** (a root-level endpoint is an endpoint file directly under the workspace root). Collections nest to arbitrary depth: `OpenWorkspace::collections()` lists the root's immediate children (excluding the reserved `sequences/`); `Collection::sub_collections()` recurses at any deeper level. Sequences stay `sequences/<slug>.toml` in a reserved dir at the **root only** — a `sequences` directory nested under a sub-collection is an ordinary collection (`sequences` is a global, root-owned role, like profiles). `folder.toml`/`churl.toml` are skipped as endpoint files at **every** level.
- **Atomic saves via one `persistence::atomic_write` helper.** Every workspace writer funnels through `save_value`, whose sole `std::fs::write` truncated the source-of-truth file in place (a crash mid-write destroyed it). Replaced with: sibling temp file in the same dir (per-process `AtomicU64`-suffixed name) → `write_all` → `sync_all` → atomic `rename` → parent-dir fsync (degrades gracefully where dir-fsync is unsupported, e.g. Windows) → best-effort temp cleanup on error. Error contract unchanged (`PersistenceError::Write`); the format-preserving merge produces the same bytes, they just land durably.
- **Element-wise array-of-tables merge** (supersedes an earlier "unequal length → replaced" clause). `merge_items` merges `[[array-of-tables]]` element-wise *regardless of length*: recurse the overlapping prefix (survivors keep decor), append new tables, truncate removed trailing ones — replacing a wholesale `*old = new.clone()` that discarded all comments/whitespace/order on surviving siblings when the array changed length. This is what makes the format-preserving promise hold across a header/param add or remove.
- **Reserved-name disambiguation, not rejection.** A create/rename whose slug equals a reserved churl stem is *bumped* (`churl` → `churl-2.toml`) like any filename clash. `RESERVED_FILE_STEMS = [churl, folder]` guards endpoint/sequence **files** (a `churl.toml` endpoint would parse as the manifest and vanish); `RESERVED_DIR_NAMES = [sequences]` guards collection **directories**. The display `name` is untouched; the disambiguated on-disk stem is surfaced in the TUI confirmation (no silent no-op). Reserved sets are kept in sync with the filename constants by a test.
- **New endpoint/collection paths are claimed atomically, not probed-then-created.** Picking a free `<slug>-N` with `.exists()` and creating the file later leaves a window where a second concurrent "new endpoint" save can slip into the same name and one write silently clobbers the other. The free-name search and the claim are now the SAME step: each candidate is opened with `OpenOptions::create_new(true)` (files) / `fs::create_dir` (collection dirs) — both fail atomically if the name is already taken — and an `AlreadyExists` advances the loop to the next `-N` instead of trusting a stale probe. The first successful create wins; two simultaneous saves land on distinct files with no lost write (a 12-thread test asserts this). `create_endpoint`/`create_sequence` fill the reserved zero-byte placeholder in place via the existing format-preserving merge (byte-identical output); `rename_*` claim the destination name before the `fs::rename` onto it. Collision filenames, reserved-stem bumping, and the surfaced-on-bump behaviour are all unchanged — only the clobber window is closed. Rename's directory *move* (not a create) keeps the name-only `collection_dir_name` picker, since `fs::rename` can't fold a claim in without clobbering; the create site is where the documented race lived.
- **Tree CRUD (move / copy / duplicate / reorder) reuses the atomic-claim discipline; ordering is per-node `seq`, filesystem-authoritative.** `CollectionMeta` gains `seq: u32` (skip-if-`0`) and the collection loaders sort `(seq, name)` — an all-default (pre-M7.12) corpus keeps its byte-identical alphabetical order, so back-compat holds by construction with **no migration** (a var-less collection only materializes a `folder.toml` on its first reorder). Ordering stays per-collection rather than a parent-held index so the filesystem remains the source of truth (a parent index would drift when dirs change outside churl). `move_endpoint`/`copy_endpoint`/`duplicate_endpoint`/`move_collection`/`copy_collection`/`duplicate_collection` mirror the create/rename claim + `-N` suffixing; a relocated endpoint appends at its destination (`seq = next_seq(dest)`), a relocated subtree appends among its new siblings. Reorder densely renumbers the sibling group `0..N` before swapping the two neighbours (a bare swap of two default-`0`s would be a no-op) and reports edge hits ("already first/last"). Reorder is **group-internal and parent-preserving** — re-parenting is exclusively move-to. No vars are copied on move/copy (a relocated node inherits its new ancestor chain; unresolved `{{var}}`s fail loud at send).
- **A single fail-loud helper rewrites sequence-step references on any path change.** `retarget_sequence_steps(root, old, new)` repoints every step whose workspace-relative `endpoint` equals `old` (a moved/renamed endpoint) or lies under `old` as a directory prefix (a moved/renamed collection), rebuilt with `/` separators and OS-independent. Copy/duplicate never call it (the original stays valid; the copy is new and unreferenced); rename/move do (also closing the pre-existing latent bug where a plain rename silently broke referencing sequences). The count of rewritten steps surfaces on the status line.
- **One destination picker for create + move-to + copy-to; root is always the first option.** A stateless fuzzy picker over every collection node incl. the root (chosen over a yank/paste clipboard-mode that would add drift-prone pending-cut state, fighting the illegal-states posture). It reuses the existing picker infra under the `Mode::Palette` routing tier (like the profile/auth pickers), carrying its target `dirs`, a `DestPurpose`, and the relocation `source` inside the variant. `<leader>n`/`<leader>N` always go destination-picker → name prompt (they ask *where* explicitly); the explorer `n`/`N` keep the M7.9 cursor-context fast path.
- **The shared name prompt auto-detects a pasted curl.** A buffer starting with the `curl` word (word-boundary) flips the prompt label to "Import from curl" and, on submit, runs the M4 `import_curl` parser (which already auto-names, e.g. `/health` → `health`) at the chosen/cursor target; a parse failure is fail-loud (nothing created, prompt stays open). This subsumes the standalone paste-curl action, retired to a palette alias. Root render order was unified to **collections-then-endpoints** (matching nested collections).
- **The env-editor Session group is writable, app-level only.** The in-memory `session` scope (process-lifetime, never on disk — a security property) is edited in place via `a` set / `d` delete / `c` clear, value masked as typed; overwrite is last-write-wins, an empty name is rejected, an empty value is allowed. Writes go through app-level `SetSessionVar`/`DeleteSessionVar` into the same map the extraction path feeds — **no churl-core, resolver, or persistence change**, and no reserved names (the store is a flat machine-local convenience, not a filesystem node).
- **SQLite: WAL + `busy_timeout(5000ms)` + `BEGIN IMMEDIATE` migration lock.** `HistoryStore::open` sets `busy_timeout` then `journal_mode=WAL`, failing loud (`HistoryError::WalUnavailable`) if the FS silently fell back (file-backed DBs only; in-memory legitimately reports `memory`). The whole migration loop runs inside one `BEGIN IMMEDIATE` transaction: write lock up front, `user_version` re-read under the lock, only still-pending migrations apply, commit once — all-or-nothing, so a concurrent opener blocks then no-ops. Migrations are append-only and immutable; a new column is always a new migration (`ALTER TABLE ADD COLUMN`), never an edit to an existing migration's DDL (editing it never lands on already-migrated DBs).
- **Local state in `rusqlite` (bundled); high-churn/relational data only.** History, workspace recency, and load-batch summaries are machine-local and never belong in synced `churl.toml` files. Bundled SQLite avoids a system dep. Recency (`workspaces` table, UPSERT by canonical/absolute path) can't span workspaces in files anyway.
- **Three unbounded-growth bounds (config-free consts).** (1) App channel `mpsc::channel(1024)` — all `AppMsg` senders are spawned async tasks (`.send().await` = backpressure); the highlight worker is a dedicated OS thread (`blocking_send`), so no sender awaits on the render thread. (2) History pruning caps `history`/`load_batches` at the newest `HISTORY_ROW_CAP`=10 000 rows each via an indexed `DELETE … WHERE id <= max_id - CAP`. (3) Highlight cache is evicted from the previously-active buffer on every buffer switch, bounding cached lines to (active-buffers × 64). Open-buffer LRU capping was deliberately not added (buffers are user-opened and bounded by intent).
- **Secrets hygiene: never in workspace files; name-marker heuristic at save time.** A workspace is a git repo users commit; secrets go in process env or a gitignored `.env`. churl-initiated writes refuse profile/collection/workspace/auth vars whose name looks secret with a literal value (`{{...}}` placeholders pass); loads never refuse (it's the user's file). Enforcement is symmetric across `save_endpoint`/`endpoint_to_toml`. Import placeholder-izes literal secret auth values (`{{password}}`/`{{token}}`); export refuses any endpoint still carrying one. The `auth.*`-only scope of the original gate is widened by the three decisions below (coverage, grandfathering, policy); `endpoint_to_toml` keeps the baseline-free auth-only hard block (`SecretsInAuth`) because a redirected stdout has no on-disk baseline.
- **Placeholder gating is a single well-formed token; secret markers are an audited substring list.** `is_template_placeholder` accepts only one `{{ name }}` run whose inner token is a non-empty `[A-Za-z0-9_.:-]` with no internal whitespace and no stray braces (`{{env:FOO}}` passes for future env refs). The old loose "starts `{{`, ends `}}`" accepted `{{}}`/`{{a b}}`/`{{a}} {{b}}`, letting a literal secret masquerade as a placeholder and skip the gate. `SECRET_NAME_MARKERS` gains audited low-false-positive markers (`passphrase`, `privatekey`, `access_key`, `secret_key`, `client_secret`, `signature`); `token` already subsumes `apitoken`/`access_token`, and noisy roots (`auth`, `session`, `cookie`, `key`, `pat`) are deliberately excluded — `Authorization`/`Cookie` are caught by header name instead. A *whole-value* placeholder check is right for raw credential fields (auth value, env var); embedded values (`Authorization: Bearer {{token}}`) use `template::contains_placeholder` (does the value embed any placeholder) so a templated credential is not flagged.
- **Grandfather pre-existing secrets; block only newly-authored ones — symmetric across env-editor and endpoint saves.** The gate is now baseline-aware: a save scans both the content being written and the file's current on-disk content, keyed by finding **location** (field path / var name). A violation whose location already existed on disk is *pre-existing* → grandfathered (save proceeds, warning emitted); a location clean/absent in the baseline is *new*. churl must never *author* a fresh literal secret, but it must not hold an unrelated save hostage to a pre-existing one the user never touched (the friction the old always-refuse gate caused — e.g. renaming an endpoint that carried a hand-written literal token now succeeds). Both write paths (`save_endpoint`/env-editor manifest+collection saves) funnel through `secrets::decide` in churl-core; detection returns data, the TUI renders the `!` marker / warning text. Sending is never gated — the gate is save-time only.
- **Save-gate covers the whole request, split by severity; `secret_policy = strict | warn`, strict default.** Coverage extends past `auth.*` to header values, URL (query keys + `user:pass@` userinfo), query params, env-var values, and bodies. Findings carry a `Severity`: **name-anchored** (`Block`) — a secret-named field/header/query-key/env-var, or URL userinfo, carrying a literal — is reliable enough to refuse a *new* write under strict; **value-only** (`Warn`) — a secret-*shaped* value under an innocent name (`looks_like_secret_value`: `sk-`/`ghp_`/`xox`/`AKIA` prefixes, JWTs, ≥32-char high-entropy runs) and any request body — is too noisy to block and only warns. `Config::secret_policy()` resolves `strict` (default: block new name-anchored, warn on value-only + all grandfathered) or `warn` (everything warns, nothing blocks), failing loud on an unknown value like every other knob (`SecretPolicy::{Strict,Warn}`).
- **Sequence steps and load batches write NO per-endpoint history.** Running a sequence or load batch fires real requests through the one `execute()` chokepoint but records nothing in the per-endpoint `history` table (avoids flooding the history view). A load run writes ONE summary row to a SEPARATE `load_batches` table — non-flooding *by construction*, not by a query filter a future change could forget.

## HTTP & networking

- **HTTP client: reqwest 0.13 + rustls** (feature named `rustls`, selected via `tls_backend_rustls()`). Async-native, pure-Rust TLS, no OpenSSL — chosen day-one so release binaries are self-contained. Coarse timing is sufficient; a custom hyper connector for waterfall timing is the documented later upgrade path.
- **Body-size cap + configurable timeout, streamed.** `execute(client, request, &ExecuteOptions)` streams the body via `chunk()`, accumulating at most `max_body_bytes` (default 10 MB, config-overridable); a chunk exceeding the cap is cut at the boundary and `Response.truncated` is set. This is a safety hole (an accidental giant download OOMs the TUI), not polish. `timeout_secs` (default 30) is the sibling config knob; per-endpoint override deferred until a real need appears.
- **Default `User-Agent: churl/<version>`, user header wins.** reqwest sends no UA by default and real services reject UA-less requests (httpbingo answers 402); an enabled user `User-Agent` header overrides it per-request.
- **No HTTP-semantic caching.** churl is a dev/testing tool; returning a cached response on "send" is a footgun. History (SQLite) covers recall.
- **Cross-origin redirect policy = `redirect: strict | strip | follow-all`, default `strip`; redirects followed MANUALLY in core.** Origin = scheme + host + port (an `http→https` upgrade, host change, or port change all cross it). `strip` (default) follows redirects but drops every auth-bearing header before a cross-origin hop (curl/browser behaviour), via **two anchors** — a header is dropped if EITHER its **name** is auth-bearing (`Authorization`, `Cookie`, any secret-named header via `looks_like_secret_name`, or the churl-injected auth header) OR its **value** looks like a secret (`secrets::looks_like_secret_value` — `sk-`/`ghp_`/`AKIA`/other vendor prefixes, JWTs, long high-entropy runs; a `{{placeholder}}` is never secret-shaped). The value anchor closes the gap where an opaque-named header carries a real token (`X-Custom-Auth: sk-live-…`), which a name-only set leaks. Stripping is the safe direction: a value-shape false positive only means a non-secret header doesn't reach a *different* origin. Same-origin hops keep all headers. **Irreducible residual:** a custom-named header whose value is neither secret-named nor secret-shaped (e.g. a short low-entropy bespoke token) still survives a cross-origin hop — closing that fully needs value-*provenance* stripping (tracking which header values were resolved from a `{{secret}}`), which `execute` can't see because it receives an already-resolved `Request`; filed as an R3 follow-up. Users following untrusted redirects should prefer `strict`. `strict` follows same-origin hops only and surfaces a cross-origin 3xx (status + `Location`) instead of following it. `follow-all` keeps every header across origins (the foot-gun) and warns once. The client is built with `redirect::Policy::none()` and `execute` runs its own hop loop (cap 10) because reqwest's built-in cross-origin stripping covers only a fixed five headers (`Authorization`, `Cookie`, `cookie2`, `Proxy-Authorization`, `Www-Authenticate`) and — verified against reqwest 0.13.4 `src/redirect.rs` — would miss a secret-named custom header like `X-Api-Key`, and its `Policy::custom` closure can only decide follow/stop/error, never mutate outgoing headers. Manual following is the only honest way to guarantee "drop *any* auth-bearing header on origin change." Method/body rewriting mirrors reqwest/tower-http and RFC 9110: 303 → GET (unless HEAD) + drop body; 301/302 → POST becomes GET + drop body (other methods untouched); 307/308 → method + body preserved. Workspace-level knob (not per-endpoint) — the simpler default. The policy decision lives in core; the one-time `follow-all` warning is emitted from core (stderr) and observable via `http::follow_all_warned()`.

## curl import/export

- **curl import flag policy: strict, lossless, no file reads.** Any flag outside the supported set is a hard `UnknownFlag` error (silently dropping a flag imports a *different* request); `-F` multipart and `@file` data are `Unsupported` errors (churl never reads files during import). The URL's query string stays in the URL — never exploded into `Param`s. Body kind derives from content + headers. `-u` lands as first-class `Auth::Basic` with the password placeholder-ized.
- **curl export: shlex-quoted single line, round-trip stable.** Every argument goes through `shlex::try_quote` (paste-safe, no line continuations); only enabled headers/params emit. `-X` is omitted for a body-less GET but kept for a GET *with* a body so `import(export(e))` reproduces method/URL/headers/body.

## Collection interchange (Postman JSON)

- **JSON interchange hand-rolled against `serde_json::Value`, no Postman schema crate.** Real-world exports are messy (missing/renamed fields, auth as arrays *or* objects); a strict typed schema would reject valid files. Import is deliberately *liberal* — unknown methods default to GET with a warning, unsupported constructs degrade to warnings, never hard errors.
- **Selectable export dialect: Postman v2.1 (round-trip) OR churl-native JSON.** Postman v2.1 is the interoperable lossy-but-round-tripping format (enforced by a round-trip test); churl-native is a thin lossless envelope reusing the model's serde derives for full-fidelity backup between churl workspaces. Nested Postman folders flatten into one collection name (churl collections are one directory deep). Query stays in the URL on import, mirroring curl import. Export path writes are guarded to stay inside the workspace root via lexical-normalize + canonical `starts_with` (blocks `..`/absolute *and* symlinked-component escapes).
- **Import auto-detects the dialect by envelope sniff — no `--format` flag.** `import_json` parses once and routes on the top-level shape: a `churl_version` key → churl-native, else `info`/`item` → Postman v2.1, else a clear `UnsupportedSchema` error. This makes native import "just work" symmetrically with how Postman import already does — the user never picks a format. Native import is enforced by a round-trip corpus (export→import→write→reload) symmetric with the Postman and curl round-trip tests. A native file whose `churl_version` exceeds this build's `CHURL_NATIVE_VERSION` is rejected with a clear "upgrade churl" error, never best-effort-parsed (silently dropping a newer producer's added fields would corrupt the backup); a single unparseable endpoint still degrades to a warning like the Postman path, but a malformed envelope is a hard error.

## TUI: navigation, keymap, editing

- **Keybindings: crokey + semantic Key→Action map, fully data-driven.** Users remap actions, not raw keys (gitui/helix model). churl's entire binding surface — leader key + submenu keys/membership + pane overlays + global — is remappable via `[keys.*]`; core carries only `"combo" = "action"` strings (stays TUI-free), the `churl` crate parses both sides at startup and fails loud on unknown entries. Leader submenus are data-driven (`submenus: HashMap<String, Submenu>` + `LeaderEntry::Submenu(name)`, built-ins seeded as defaults); config can create arbitrary submenus. This is the plugin-command foundation.
- **Load-time keymap conflict/shadow validator — loud, non-blocking, genuine-defects-only.** `KeyMap::validate` runs on the final keymap over the *raw* config tables (to spot silent last-wins) and flags only genuine defects: the leader key also bound as a normal action (dead bind); dangling/orphan submenu tables; two combos normalizing to the same key in one scope; a global bind shadowed in *every* pane overlay. Documented intentional single-pane overlay shadows produce zero warnings. Surfaced in three non-blocking channels: `eprintln!` before raw mode, a first-frame statusline toast, and a `⚠ Conflicts` section in `churl keymaps`.
- **Key routing precedence.** (1) an open overlay consumes every key; (2) request pane focused with edtui in a non-Normal mode → all keys to edtui; (3) otherwise crokey `KeyMap` (via `lookup_ctx`, focused-pane overlay first then global), unmapped keys fall through to edtui in the request pane. edtui owns insert/visual modality — the app never re-implements vim. **One documented exception:** a CONTROL-modified key resolving to Send/Quit is intercepted *before* edtui even in insert/visual mode (edit→send is one keystroke; no text-input key can be stolen).
- **Contextual keymap overlays (`PaneCtx`).** `KeyMap` carries `overlays: BTreeMap<PaneCtx, …>`; `lookup_ctx(key, ctx)` searches the focused pane's overlay first, then global. This is the vehicle for a key meaning one thing pane-locally and another globally (Response `h`/`/`/`p`/`s`, Request `1`–`4`) — a flat crokey table can't host both.
- **4-region Tab model, not 5 panes.** Tab stops map to *distinct visible regions*, not every focusable pane: `Left column → URL bar → Request → Response`. The right column shows 3 rows simultaneously (3 stops); the left column shows one sub-pane at a time (1 stop, restoring the last-active sub-pane + zoom on return). A collapsed explorer is skipped, not reopened, by the Tab path; explicit focus still auto-reopens.
- **`f`-jump is pane-only — five regions, zero row labels; mnemonics e/s/u/r/p.** Labels the five pane regions (Endpoints, Sequences sub-pane, UrlBar, Request, response), not endpoint rows. Pressing the Jump key twice cancels. The Sequences sub-pane is modelled off the `Pane` axis (it lives inside `Pane::Explorer`), not as a fifth `Pane`.
- **`cycle-region-fwd`/`back` shipped UNBOUND; Ctrl-Tab deliberately not hardcoded.** A region-aware cycling action (left column cycles sub-panes, right cycles buffers) exists but is bound to no key by default — users bind a portable key via `[keys]`. Ctrl-Tab is terminal-unreliable (indistinguishable from Tab without the Kitty keyboard protocol; dead on Terminal.app / Windows conhost / tmux-without-passthrough, and claimed by iTerm2/browsers/tmux). Terminal-level key decisions are reasoned against the weakest terminal, not the dev's. Enhanced-keyboard-protocol support (which also unlocks reliable Ctrl-J) is a tracked follow-up layered over portable defaults.
- **Derived dirty tracking, not a flag.** `App` keeps `loaded_snapshot: Option<Endpoint>` (a pristine clone at load); dirty is *computed* by comparing the live request against it each render — no boolean to forget to set/clear across the dozen mutation paths. churl never writes on a keystroke; only explicit `w` saves, and a dirty endpoint-switch routes through one guarded seam (`guarded_load`) that raises a save/discard/stay confirm.
- **`LineEditor` over an editor dependency.** A ~90-line hand-rolled single-line editor (char-indexed for multibyte) shared by the URL bar, request-row field edits, and every CRUD prompt — edtui is a full vim-modal editor, overkill for one-line input, and no new dep is worth one line of text. `NewEndpoint` defaults here too; it only escalates to the multi-line edtui vim editor (`CurlPrompt`, `tui/app/mod.rs`) when there's real curl/multi-line content, per the two-stage design below.
- **`NewEndpoint` is a two-stage prompt: light by default, heavy only for real curl content.** Every other CRUD prompt purpose is a one-line name; `NewEndpoint` used to open straight into the multi-line vim editor (`CurlPrompt`) even for the common case of typing a short endpoint name, which is needless overhead for the fast path. Now `<leader>n`/`n` open the single-line `LineEditor` like any other prompt (`curl_prompt: None`), and only escalate to the multi-line editor (`curl_prompt: Some(..)`) on real multi-line/curl content — the `Option<CurlPrompt>` field is the sole discriminator between the two sub-states (illegal-states-unrepresentable: both can never be active at once). Two entry points reach the heavy editor: (1) pasting text that starts with `curl` or contains a newline switches to it in Normal mode, seeded with any text already typed plus the paste, ready to review/edit/submit; (2) `Ctrl-e` explicitly expands it in Insert mode, seeded with whatever was typed so far, for composing a curl by hand. `Ctrl-e` shadows `LineEditor`'s own end-of-line binding, so the interception is scoped to the `NewEndpoint` single-line sub-state only — every other prompt purpose keeps `Ctrl-e` = end-of-line. A curl parse failure on submit re-opens directly into the multi-line editor (Normal mode, buffer intact), never the single-line default, since there's curl content to fix.
- **In-app CRUD flows through `churl-core::persistence` seams; the TUI never touches the filesystem.** Typed-error core fns (`create/rename/delete_endpoint` + `_collection` + `_sequence`), all keeping the secrets refusal. Delete is risk-proportional (endpoint = `y/n`; collection = type-the-name, since it's recursive). Missing vim motions edtui lacks (`W`/`B`/`^`/`f`/`F`/`t`/`T`) live in `components/vim_ext.rs` as pure Normal-mode cursor mutations, applied to every edtui editor (Body, URL popup, and the `NewEndpoint` curl prompt) — forking edtui's key register was out of scope.
- **Command palette is a curated allowlist, not `Action::all()`.** An explicit `const COMMANDS` of context-free commands only — pane-contextual motions that no-op or misbehave when dispatched from an overlay are excluded. New actions do not auto-appear; inclusion is a deliberate per-command decision (guarded by a test).
- **Leader key = Space; find/open pickers are leader continuations.** `<leader><leader>` opens the endpoint picker, `<leader>s <leader>` the sequence picker, `<leader>l <leader>` the load-runner picker — one gesture per job, freeing `f` for jump-mode. A leader *continuation* of Space is expected and never flagged by the validator.
- **`<leader>r` reloads the workspace from disk, dirty-guarded.** A first-class reload re-reads `churl.toml` (`OpenWorkspace::open` on the current root, reusing the env-editor live-refresh idiom) and rebuilds the explorer, so external edits to `churl.toml`/`folder.toml` (another editor, a second churl instance) are picked up without a restart — `reload_explorer` alone only rebuilt the tree from the in-memory manifest. The dirty guard reuses `any_buffer_dirty`: a reload with any dirty buffer DEFERS (refuses with a status message asking to save first) rather than routing through the discard-changes confirm, whose only resolution is to *destroy* buffers — the wrong outcome for a refresh, since re-reading + remapping can shift collection indices under an open editor. On a failed manifest re-open the current workspace is left intact (fail loud, never half-swap). The action lives in the `churl` bin (keymap/dispatch/status); churl-core stays TUI-free.

## TUI: response viewer

- **Display pipeline: fold → wrap → viewport, all pure.** The viewer composes three pure transforms over logical lines in a fixed order (JSON fold filter → optional wrap expansion → viewport slice). Cursor/scroll are display-row indices (post-fold, post-wrap); search matches are stored against logical lines and mapped through the pipeline. Per-view UI state lives on `ResponseView` (resets together on a new response); cursor/scroll/geometry live on `App`. All stages are pure fns, keeping snapshot tests runtime-free.
- **Virtualised render + viewport-only, viewport-stateless highlighting.** The line-offset index materialises only visible lines (protects the 1 MB-body budget). Syntax highlighting (syntect + two-face) is off-thread, viewport-only, cached by a viewport hash. It starts a fresh `HighlightLines` per viewport — a known imperfection (multi-line constructs opening above the viewport are mis-coloured), accepted for the render-loop budget; whole-document highlighting of a 1 MB response is never done. Highlight-worker thread-spawn failure degrades to plain rendering (`spawn` returns `Option`), never panics.
- **Reformatter is JSON-only via `serde_json`, pretty-by-default, wire-order preserved.** JSON is the dominant API content-type and `serde_json` is already a dep, so pretty-printing adds zero deps (parse → `to_string_pretty`). Malformed JSON silently falls back to raw (a viewer must always render *something*). `preserve_order` is enabled workspace-wide so keys re-emit in on-the-wire insertion order (sorting would misrepresent the response); an optional `s` A→Z sort layers on top, ephemeral per response. HTML/XML reformat is deferred (would need a new dep + error model).
- **Copy reads raw on-the-wire bytes, never the reformatted/sanitized display.** `ResponseView` keeps a separate `raw_text` (exact lossy-UTF-8 decode). `copy_all` (`y`) returns it byte-exact regardless of pretty/sort/gutter/sanitize state — a copy that emitted reformatted-and-sorted JSON would silently corrupt what the user believes they copied. `copy_line` (`Y`) returns the raw wire line (CRLF included) for non-reformatted bodies, the displayed pretty line only when pretty-JSON actually changed the layout.
- **Displayed text is sanitized before indexing; `raw_text` stays untouched.** `sanitize_for_display` runs once per build: strip all ANSI (CSI/OSC/lone ESC — a hostile server must never reach the terminal or emit OSC-52 clipboard writes) → expand tabs to `TAB_WIDTH=4` display-column stops → `\r\n`→`\n` → remaining C0/DEL/C1 controls → a visible `·`. It is line-count-preserving. `TAB_WIDTH` is a documented const, not a config knob.
- **`h_scroll` horizontal window bounds unwrapped long-line render cost.** A minified/non-JSON line has no `\n` to split, so `expand_wrap` windows each unwrapped row to `[h_scroll, h_scroll+width)` (still exactly one row per line, so the fold/wrap→logical mapping stays intact). Panned on `H`/`L` (wrap-off only); clamped in `render_done` and written back via the render outcome; search-into-view pans both the match row and column. Wrap mode does cell-aware splitting via `unicode-width`; the unwrapped window is char-bounded (a known wide-glyph limitation, no terminal corruption since ratatui truncates at the edge).
- **Response-view render geometry cache, keyed on its inputs.** The `render_done` path recomputed FULL-body geometry every frame including idle ticks (O(body) per tick on a multi-MB body). A single-slot memo per product (fold-filtered visibles, wrap-expanded rows, widest-line width), each keyed on a signature capturing exactly its inputs, serves an unchanged-signature frame verbatim — a signature match guarantees a byte-identical recompute. Two keys because the widest-line product omits `h_scroll` (independent of horizontal pan). Lives on `ResponseView` in a `RefCell`, dropped and rebuilt when a new response replaces the view.
- **Line-number gutter, ON by default, toggled by `#`.** Numbers the 1-based *displayed* logical line (continuation rows blank, fold-headers carry the opener's number); gutter width is the digit count of the total line count (stable while scrolling). The gutter is subtracted from the body width ONCE in `render_done` and that single reduced width drives every geometry consumer (`max_h_scroll`, `expand_wrap`, cursor↔logical mapping). Render-only — never enters `raw_text`, copy, or the line index.
- **Runner viewers route Response keys through the SAME dispatch + `response_*` handlers as the main pane.** The load/sequence runners' hand-maintained response key tables (which had drifted twice) are deleted; `active_response_surface()` resolves `Main | LoadRunner | Sequence` from mode+focus, and a shared `ResponseGeometry` on the buffer AND both runner states makes the `response_*` handlers mode-aware with no per-handler branching. One code path, so the runner viewer can never drift again. Body-search opened over a runner returns to the runner (`body_search_return`), not `Normal`.
- **Sticky, error-styled status-bar flag once history writes start failing.** A consecutive-failure counter (`history_write_failures`) arms `⚠ history not recording` in the statusline at the first persistent failure and clears on the next success — replacing a scroll-past toast that let history silently stop recording. On quit, every in-flight request records an interrupted history row (`write_history(None, None)`, the cancel path) across ALL buffers before the runtime drops, so a request that may have completed on the wire never vanishes untraced.

## TUI: sequences, load runner, env editor

- **Extraction is a documented, dependency-free subset over `serde_json` — no JSONPath crate.** `extract_value` supports `status`, `header:<Name>`, and a JSON path (`$.a.b[0].c`), parsed by a hand-rolled byte scanner (unicode-safe, never panics). A JSONPath crate is heavyweight for a testing tool's needs; the subset covers the real chaining cases. Any single rule failure fails the step (subject to `on_error`).
- **The run engine lives in `churl-core`; runner state is UI-only; the App drives.** Core exposes shared seams (`classify_response`, `should_halt`, `prepare_step`/`extract_step`/`ordered_steps`) that both `run_sequence` and the live TUI driver call — drift guarded by a cross-check test running both through identical scenarios. The TUI mirrors exactly two trivial lines the seam can't cover (transport-error stringification at the task boundary + skipped-tail marking). Bounded concurrency uses `futures`' `buffer_unordered` (identical guarantee to a semaphore-guarded `FuturesUnordered`, no deadlock foot-gun); cancel aborts the ONE launcher task, dropping every in-flight future (no detached per-request spawn). Absolute-target pacing (copy `i` never launches before `i·interval`) avoids super-linear cumulative delay under low concurrency.
- **Step endpoints cannot escape the workspace root — lexical *and* canonical.** `resolve_step_path` rejects `..`/absolute/drive-prefix lexically, then canonicalizes the deepest existing ancestor and requires it to stay under the canonical root (closing an in-workspace symlink escape). A *missing* endpoint is not a traversal (its ancestor resolves inside the root → not-found, a distinct failure kind).
- **Load-runner response retention bounded to O(concurrency + K); dropped rows are not reconstructable.** A high-`total` run building a `ResponseView` per copy retained `total × body` → OOM. The runner keeps a full `Done` view for at most K=16 rows + the selected row (never evicted); the oldest non-selected retained row downgrades to `ResponseState::Dropped { status, timing, size }` with NO body bytes — the viewer renders it honestly ("response body not retained"). Per-row status/timing and the outcomes-driven stats are untouched, so aggregate correctness is independent of eviction. Nearest-rank percentiles over completed-request timings only; empty/all-errored input yields `None` (no panic).
- **Load-runner guardrail: hard-cap refuse + warn-cap loud confirm naming the URL, `[load]`-overridable.** `check_config` returns `Refuse` above a hard cap (10 000 total / 200 concurrency), `Warn` above a threshold, else `Ok`. The runner refuses with a message or renders a confirm naming count, concurrency, and target URL — making "testing aid, not a load-cannon" real, not advisory. All interrupt paths (cancel / re-run / close-mid-run) share one seam that records the partial (cancelled-marked) summary before aborting.
- **Env editor: self-contained modal in the `churl` crate; profile CRUD in working state, core UI-free.** All editor state lives in `components/env_editor/`; core gained nothing UI. Profiles are created/renamed/deleted in working state and persisted by rebuilding `Workspace` and calling `save_workspace_manifest`. The discard guard is editor-local, not the global `guarded_load`. Save validates everything up front (duplicate-name + secret gate across all scopes) so no secret leaks and no partial write is silently wrong; the multi-file write itself is sequential and loud-on-failure (temp-write+rename hardening deferred). Precedence display is computed relative to the selected scope's context (the collection layer is per-endpoint, so a single global "winner" is ambiguous).
- **Env-editor secret gate = grandfather + warn** (reverses a pure validate-all-before-write dead-end). churl never *writes* a new literal secret (hard block stays), but a pre-existing hand-authored literal no longer blocks an unrelated save — it saves with a warning + an inline `!` marker on the offending row. The gate's intent (don't let churl persist secrets) is preserved; only the enforcement point moves from all-or-nothing to new-literals-only.
- **Extraction rule → optional in-memory Session variable.** A step's rule can feed its captured value into a Session variable that outlives the run (so a standalone request later resolves `{{token}}`). Modelled as a separate `persist: Vec<String>` name list (additive, format-preserving, backward-compatible) — never overloading the `extract` map. The Session store is IN-MEMORY, process-lifetime, keyed by canonical workspace root (cross-workspace isolation), NEVER written to disk (a deliberate security invariant, asserted by a test that walks the tree post-run). A failed extraction never clobbers a prior value. The env editor lists Session captures as a read-only, always-masked scope with a clear action.

## Clipboard

- **Native Wayland clipboard on** (`wayland-data-control`) — reverses the reversal that left it off. arboard was originally added with the Wayland feature off, so a pure-Wayland session (no XWayland) fell through to OSC 52; enabling the feature makes arboard try the native wl-data-control protocol when `WAYLAND_DISPLAY` is set, falling back to X11 only if that fails. Verified against arboard 3.6's own manifest/source (not assumed): the feature's `wl-clipboard-rs` dep is gated to non-macOS Unix, so it is a compile-time no-op on macOS/Windows (macOS build confirmed green). The OSC 52 fallback is unchanged and still covers displayless terminals where no native clipboard is reachable. Runtime Wayland correctness can't be proven in headless CI (`copy_native` isn't exercised by `cargo test`); the gating is verified by reading arboard's primary source.
- **Native (arboard) primary + OSC-52-with-passthrough fallback** (reverses an earlier "OSC 52, no native dep" decision). Raw OSC 52 does not reach the system clipboard through tmux/screen without passthrough-wrapping, and macOS Terminal.app ignores it entirely — yet the old code reported success unconditionally, so copy silently failed for those users. `copy` now tries the native OS clipboard first via `arboard` (the only path that reliably reaches the system clipboard regardless of terminal; native Wayland support is covered by the note above), falling back to OSC 52 wrapped for the active multiplexer (tmux DCS passthrough / screen / raw). The real `CopyOutcome` is reported honestly. `arboard` is on the churl bin only (never core) and was already a transitive dep via edtui. Payload capped at 1 MB (`MAX_COPY_BYTES`, char-boundary).
- **edtui editors (curl prompt, URL popup, Body editor) hold an in-memory clipboard, not the OS clipboard.** Every `EditorState` churl constructs goes through a shared `new_editor_state` helper that swaps edtui's arboard-backed default for `edtui::clipboard::InternalClipboard`. Vim's own yank/paste (`y`/`p`/`d`) in those editors is therefore editor-local — it targets vim's unnamed register in memory, not the system clipboard, matching vim's own default behaviour and stopping a stray in-editor yank from clobbering whatever the user has on their real clipboard. It also sidesteps a concurrency hazard: arboard's `Default` opens a single shared OS clipboard handle, and more than one editor touching it at once (observed under the test suite's parallel threads) segfaults the OS clipboard backend. Pasting *into* churl still works exactly as before — Cmd/Ctrl+V lands via terminal bracketed paste, which writes into and reads back out of this same in-memory clipboard, so nothing user-visible changes there.

## UI: theming & rendering

- **Theme = named style slots parsed from core strings, mirroring `[keys]`.** Core carries only strings (`theme` selects a built-in, `[theme_colors]` maps slot→colour); the TUI parses them into ratatui `Style`s at startup and **fails loudly** on an unknown built-in/slot/colour, naming the offender — no silent fallback, exactly like the keymap. Slots are data-driven (one `slot_mut` match) so a new slot is one line. Built-ins `dark`/`light`; the syntect embedded theme follows the UI theme through the single content-type→syntax mapping point.
- **Both tab bars render as filled chips.** The top buffer strip and the request Params/Headers/Auth/Body bar render each tab as a `▐`…`▌` chip with a gap; a shared `chip_width`/`chip_window` centralises width accounting so the keep-active-visible window can never drift from what the render loop paints (the request bar reuses the strip's window-scroll so the active chip is always fully visible). A `theme.tab_inactive` slot gives inactive chips a real background fill.

## Tooling, CI & distribution

- **Supply-chain gate = `cargo-deny` (advisories + licenses + bans + sources), replacing the standalone `cargo audit` job.** deny's advisories check reads the same RustSec DB as audit, so it subsumes it, and adds three more gates in one config-driven pass (`deny.toml`): an explicit license allow-list (no blanket allow — every license the resolved tree carries is named, including `BSL-1.0` from arboard's Windows clipboard backend), a bans check (wildcard deps denied, duplicate versions warned), and a sources check (crates.io only; a git/alt-registry source must be allow-listed). `unmaintained` only **warns** — matching audit, which never failed on it — because the two hits (`bincode 1.3.3`, `yaml-rust 0.4.5`) are transitive under `syntect` with no safe upgrade; vulnerabilities and yanked crates still hard-fail. Installed via `taiki-e/install-action` (prebuilt, no node24 issue), same as the audit job it replaces.
- **CI = independent parallel jobs in one `ci.yml`: Format, Lint, Test, supply-chain (cargo-deny).** Each check is its own job so it can be required individually in branch protection and gives an isolated signal. No `needs:` between them — they are mutually independent, so chaining would only serialize with no benefit (`needs:` is reserved for genuine pipelines like the release flow). Splitting costs runner-*minutes* but churl is a PUBLIC repo where Actions minutes are free, so the only currency is granularity + isolation — the split delivers it at no cost. (This calculus flips for a private repo.)
- **Test and Lint fan out across an OS matrix (`ubuntu-latest`, `macos-latest`, `windows-latest`; Lint on Linux + Windows), `fail-fast: false`.** Cross-platform path behaviour and the cfg-gated clipboard backends only get exercised if the suite actually runs on each OS. Each matrix leg is a distinct check (`Test (windows-latest)` etc.), so the "each check independently required in branch protection" design holds per-leg — the required-checks list must be updated to name the matrixed checks. Lint runs on Windows (not macOS) so clippy sees the Windows-only clipboard/path code; macOS clippy adds little over the macOS Test leg. `fail-fast: false` keeps every OS reporting so one platform's failure never masks another's.
- **Branch naming: `<type>/<milestone>-<slug>`.** Conventional-commit type prefix + optional milestone id + kebab slug (`feat/d1-demo-stabilize`, `refactor/r0-atomic-saves`); non-milestone work drops the milestone. The `<type>` matches the PR-title conventional-commit type so branch, PR title, and squash commit stay consistent.
- **Distribution: prebuilt binaries + `curl | sh` installer, zero end-user prerequisites** (via `taiki-e/upload-rust-binary-action`). rustls (no OpenSSL) and bundled SQLite were chosen day-one so the binary is self-contained (Rust is a build-time dep only). Targets: macOS arm64/x86_64, Linux x86_64 + aarch64 musl (static), Windows x86_64. `install.sh` (POSIX sh) detects OS/arch and installs to `~/.local/bin`; `install.ps1` (PowerShell) mirrors it for Windows — same option surface (`-To`/`-Tag`/`-Force`/`-DryRun`), same stem-based `.sha256` verification, installs to `%LOCALAPPDATA%\Programs\churl`. Both are attached to every release so `releases/latest/download/install.{sh,ps1}` resolve, and both are proven end-to-end by the `smoke-test-install` matrix (ubuntu/macos run install.sh, windows runs install.ps1).

## Lifecycle: version pinning, self-update, uninstall

- **Version pinning is advisory (warn-only), via a standalone `.churl-version` file.** A workspace may carry a `.churl-version` file at its root naming one exact version (nvmrc-style). When present and the running binary differs, churl **warns on stderr and runs anyway** — it never refuses and never switches/acquires a version (out of scope, and refusing would strand a user mid-task over a cosmetic mismatch). A standalone dotfile (not a `churl.toml` key) was chosen so the pin is greppable, tool-agnostic, and editable without parsing the manifest — matching `.nvmrc`/`.ruby-version` convention; absent ⇒ no-op. The pure half (discovery at the workspace root, first-non-empty-line parse, and the compare) lives in `churl-core::pin` (unit-tested: matches/differs/absent/malformed); the warning *display* is surfaced once at workspace load in the `churl` bin. `.churl-version`'s slug (`churl-version`) is added to `RESERVED_FILE_STEMS` so no create/rename slugs an endpoint into the pin marker.
- **Pin/update compare strictness: semver-aware when both parse, exact-string fallback.** `check_pin`/`compare_versions` parse both sides as `semver::Version` (the crate was already reachable in the resolved tree, now a direct dep) and compare the precedence fields — `major.minor.patch` + pre-release — **ignoring build metadata** (`+meta` carries no precedence per the spec, so `1.2.3+ci` satisfies a `1.2.3` pin). When either side is not valid semver (a nightly/tag string), it falls back to exact string equality (nvmrc-style) so it still compares sensibly and never panics. This buys numeric comparison (`0.10.0` > `0.9.0`) over the naive string compare an nvmrc would do.
- **Self-update is native self-replace from GitHub releases, fail-closed + reversible.** `churl update` queries `releases/latest` (GitHub API, `User-Agent` set — GitHub requires it), compares to the running version, and — already-current ⇒ reports and exits with **no download**. `--check` is a dry-run (report + exit); otherwise it prompts `[y/N]` (`-y` bypasses). It selects the asset for the compiled target triple with names matching the install scripts/`release.yml` exactly (`churl-<target>.tar.gz`/`.zip` + `churl-<target>.sha256`), **verifies the SHA-256 before touching anything** (mismatch ⇒ abort, nothing changed), backs the current binary up to `churl.bak`, then atomically replaces the running binary via the **`self_replace`** crate (never hand-rolling the Windows atomic-rename-over-running-binary path). All update logic lives in the **bin** (network + self_replace + release-artifact concerns are not core); the pure pieces (target→asset mapping, version compare, checksum parse/verify, backup+stage) are unit-tested against a fake binary in a tempdir — the network + `self_replace` swap itself are proven by the reviewer + a real release, not by `cargo test`.
- **Uninstall removes the binary by default; user data only behind `--purge`.** `churl uninstall` removes just the running executable and **prints the config dir + `state.sqlite` paths it deliberately left behind** — non-destructive by default. `--purge` additionally deletes churl's own config directory (parent of `global_config_path()`, honouring the `CHURL_CONFIG` override) and the state db (`history::default_state_path()`), but only after printing exactly what will go and a `[y/N]` confirm (`-y` bypasses). It **never** touches user workspace files (`churl.toml`, collections, `sequences/`) — those are the user's data, not churl's; `state.sqlite` is app-managed and fair game under `--purge` only. The path-selection is a pure `removal_plan(exe, config_dir, state_db, purge)` returning `{remove, kept}`, unit-tested against tempdir-scoped paths — including an assertion that workspace files are never in the removal set.

## Session request controls: cookies, proxy, insecure-TLS (M8)

- **All three are session state applied by rebuilding the single client.** Proxy, insecure-TLS, and the cookie jar are held on `App` (`session_proxy`/`session_insecure`/`cookies_enabled` + `Arc<ChurlCookieJar>`), never per-endpoint or per-request. `http::build_client_with(&ClientConfig { timeout, proxy, insecure, cookies })` is the one seam; `build_client(timeout)` delegates to it with safe defaults so the ~50 existing test call sites stay untouched. Every live change (`<leader>k`, the Options overlay) routes through `App::rebuild_client` — there is exactly one client, always current. No dual-client machinery, no per-request client selection.
- **Proxy precedence CLI `--proxy` > workspace `churl.toml` > global config > env; credentials never persisted.** An unset proxy calls no `.proxy()` at all, so reqwest honors `HTTP(S)_PROXY`/`NO_PROXY` for free. A proxy URL with userinfo (`user:pass@`) is allowed from CLI/env/the overlay (masked in every UI display) but **refused on save** to `churl.toml`/`config.toml` (`proxy_has_credentials` → `PersistenceError::ProxyCredentialsRefused`) — never silently stripped, because a stripped credential would surprise the user into thinking it was saved. HTTP/HTTPS only (`Proxy::all`); SOCKS is M8.1.
- **Insecure-TLS is `danger_accept_invalid_certs(true)` — and with rustls that is enough.** The rustls verifier the flag installs ignores **both** an invalid/self-signed cert and a hostname mismatch, so `danger_accept_invalid_hostnames` (a native-tls-oriented knob) is deliberately not added — a real self-signed + wrong-host TLS test proves the behaviour (secure client rejects both, insecure accepts both). It began as a session-scoped launch default (`-k`/config); a durable per-endpoint opt-in shipped later, scoping a disk-level TLS downgrade to exactly the endpoint that needs it while siblings stay verified (per-*workspace* persistence stays out of scope). It always flies a loud RED "⚠ TLS OFF" statusline flag — an insecure session must never look secure.
- **Cookie jar: an owned `cookie_store::CookieStore` wrapper, opt-in, per-workspace, persistent.** `ChurlCookieJar(RwLock<CookieStore>)` mirrors reqwest's own `Jar` but over a store *we* own, so we can serialize it — this sidesteps the `reqwest_cookie_store` bridge crate's separate version pin. Cross-origin non-leak is a property of the store's RFC 6265 domain/path/`Secure` matching, not extra code, and layers under R3's manual `Cookie`-header strip (belt and suspenders, proven by a host-A-to-host-B test). Persistence is SQLite migration 5 (`cookies` table keyed by canonicalized workspace root), **persistent cookies only** (session cookies stay in RAM and evaporate on exit — standard). The `Arc` lives on `App` and outlives client rebuilds, so toggling cookies off→on keeps the accumulated jar. Not encrypted at rest — parity with the local-only, uncommitted `state.sqlite`.
- **In-TUI Options overlay + `<leader>o`/`<leader>k`; both real keymap actions.** The overlay (`components/options/`, a `Mode::Options(OptionsState)` variant per the illegal-states-unrepresentable rule) is UI-only: it emits an `OptionsOutcome` the app applies, then refreshes the overlay's mirror. `o`/`k` were collision-checked free at the leader root (mnemonics: options / insecure) and registered as `Action::{OpenOptions,ToggleInsecure}`, so they auto-appear in `?` help + the command palette. `churl cookies list|clear` gives the same jar headless access.

## Per-endpoint insecure-TLS (M8.1)

- **Insecure-TLS gains a durable per-endpoint dimension without displacing the session override.** `Request.insecure` (serde-default, omitted when false → back-compatible, no numbered SQLite migration since endpoint files are serde TOML) turns off certificate *and* hostname verification for that one endpoint. The session `<leader>k` / Options toggle stays an **ephemeral whole-session override**; there is deliberately **no per-workspace** tier (a workspace-wide disk-level TLS downgrade is too blunt). The effective insecure for a send is `endpoint.request.insecure || session_insecure` — so a sibling secure endpoint in the same session still verifies.
- **The single-client model survives; divergence is served by a cached insecure variant, not per-request clients.** `App::rebuild_client` still builds one shared client from the session knobs. Only the one divergent case — a *verifying* session sending an endpoint that individually opted in — needs a different client, so `App::client_for(request)` returns the shared client for the common case and, for an opted-in endpoint, a lazily-built `insecure_client` that mirrors the shared client's timeout/proxy/cookies with verify forced off. Every `rebuild_client` invalidates that variant (`None`), so it can never drift from the session's other settings. When the session is already insecure the variant is never built (the shared client already matches). This keeps the M8 "exactly one client for the common path" property while making per-endpoint TLS real, and it is proven by a real self-signed-server test where one session sends a secure endpoint (rejected) and an opted-in endpoint (accepted).
- **The per-endpoint gesture is `<leader>K`, distinct from the session `<leader>k`, and loud.** Shift-k toggles the *selected* endpoint's flag and persists it immediately via the format-preserving `save_endpoint`; it is a separate `Action::ToggleEndpointInsecure` so it auto-surfaces in `?` help + the palette. The effective flag drives the red "⚠ TLS OFF" statusline badge (`insecure_active` = `session || selected.request.insecure`), and the URL bar carries a steady error-styled "⚠ insecure" marker so a saved verify-off endpoint is visible at rest. curl `-k`/`--insecure` now **bakes** `insecure = true` onto the imported endpoint (a security property of the request) with a loud warning, and curl **export re-emits `-k`** so import↔export round-trips (corpus-tested), while `-x`/`--proxy` stays session-scoped (a connection setting, not an endpoint property). The flag rides serde through churl-native import/export for free (round-trip tested); Postman export drops it (hand-mapped) — acceptable.
- **Known limitation — shared cookie jar spans the insecure/secure boundary.** The per-endpoint insecure client and the shared verifying client use the **same** `Arc<ChurlCookieJar>`, so a cookie learned over an *unverified* hop to `a.example.com` (scoped `Domain=example.com`) can later ride a *verified* request to `api.example.com` — an unverified origin can seed a cookie that a trusted sibling then sends. This is the accepted M8.1 trade-off (one jar keeps the single-client model and cross-toggle continuity); the real fix — quarantining cookies learned over an insecure hop — is bigger than M8.1 and is filed in ROADMAP's known-limitations backlog. Cross-*origin* leak is still prevented by the store's RFC 6265 scoping; this caveat is specifically the cross-*subdomain*, insecure→secure seam.

## CLI & headless (M8.2)

- **The `--json` envelope is a versioned compatibility surface, frozen before the CLI shape shipped.** `{ schema_version, ok, command, data, error }` on stdout only — every log/warning/diagnostic goes to stderr, `--json` disables color/spinners/prompts/bracketed-paste. `schema_version` bumps only on a breaking change; a new optional field or a new `error.kind` slug is additive. Full contract: [`docs/CLI.md`](CLI.md).
- **`run`/`send` drive the exact same `churl_core::http::execute`/`Resolver`/persistence seams the TUI uses — no forked request logic.** `run <endpoint>` resolves a `collection/.../endpoint name` display path directly against `churl_core::persistence` (`crates/churl/src/resolve.rs`) rather than spinning up the TUI's lazily-cached `ExplorerState` tree — there is no live multi-request session in a one-shot process, so the "session" resolver scope is simply omitted (cli → profile → collection ancestor chain leaf→root, matching `App::build_resolver` minus that one scope).
- **Exit codes are a frozen 6-band table, and band 2 (usage errors) is owned entirely by clap.** A missing/conflicting/unknown flag to `churl` itself prints clap's own text and exits 2 — no JSON envelope is constructed for it even under `--json`, deliberately: inventing an app-level envelope for what clap already handles would create two divergent "usage error" shapes. `send`'s URL requirement (`<URL>` positional XOR `--url`) and its `-X`/`-H` value shapes are therefore validated via clap `ArgGroup`/`value_parser`, not runtime checks, so a bad invocation stays in band 2.
- **`error.kind` is a small closed enum, one variant per concrete failure mode, banded 1:1 to an exit code** (`crates/churl/src/output.rs::ErrorKind`) — `no-workspace`/`endpoint-not-found`/`unresolved-var`/`unknown-profile`/`config-error` (band 3), `invalid-url`/`timeout`/`transport-error` (band 4, mapped from `HttpError`), `not-a-curl-command`/`import-write-failed` (band 5). Every `churl_core::import::ImportError` variant collapses to the single `not-a-curl-command` slug — from the caller's point of view it's one bucket ("the input wasn't an importable curl command"), and the free-text `message` still carries the specific reason.
- **Pre-flight failures (global config, current directory) route through the envelope, never a `?`-bubble.** A malformed global config (unreadable, bad TOML, or an invalid knob such as `redirect`) or an undeterminable cwd is a *resolution* failure, not a usage error — mapped to the closed `config-error` slug in **band 3** and emitted as a proper envelope. This closes a real gap: routing these through `main() -> Result<()>` with `?` would let Rust's `Termination` exit **1** (which is FROZEN as RESERVED for assertion failure) and print a Debug blob to stderr with **no** envelope under `--json`. Every non-zero, non-clap exit now carries `ok:false` + a populated `error.kind`. (A malformed `--var` is the one pre-flight case that is genuinely usage — a clap `value_parser` keeps it in band 2, clap-owned/envelope-exempt.)
- **`unresolved-var` is a new hard failure mode headless: a literal `{{name}}` left after substitution refuses the send.** Mirrors `App::send_request`'s existing fail-loud check (`churl_core::template::unresolved_placeholders`) — a headless agent silently shipping a broken `{{token}}` header would be far worse than an interactive user seeing a statusline message.
- **The echoed `request` masks secrets on BOTH `headers` and `url`, on the success surface (JSON envelope + `-v` trace) AND the failure surface (`invalid-url` message + detail).** Headers use a dual-anchor mask (name via `looks_like_secret_name`/`authorization`/`cookie`, value via `looks_like_secret_value`). URLs use `churl_core::secrets::mask_url` — the redaction twin of the existing `secrets::scan_url` scanner, masking the `user:pass@` userinfo password and secret-named/secret-shaped query values. This reuses machinery churl-core already had (an earlier draft wrongly claimed URL masking needed value-*provenance* tracking "bigger than M8.2" — it does not; `scan_url` already pinpoints exactly these spans). The real outgoing request always uses the unmasked values; only the echo is redacted. **Known limitation (inherited, not M8.2's to widen):** an opaque header/query *name* carrying a *low-entropy* value that trips neither the name nor value anchor still echoes — this is the same irreducible residual R3's redirect-`strip` heuristic documents (closing it fully needs value-provenance tracking, a codebase-wide change). Best-effort request-echo masking, not a guarantee.
- **`churl import`'s default flips to writing into the cwd workspace; `--stdout` keeps the old default; `--out FILE` is unchanged.** A missing workspace is `no-workspace` (band 3) with a hint to run `churl init` first — deliberately *not* auto-vivifying a workspace as a side effect of an unrelated command (unlike `--import-collection`, which does bootstrap a bare `churl.toml` — that flag's whole purpose is "import into whatever's here," while `import`'s default assumes an already-open workspace, matching `run`'s assumption).
- **The workspace-write import path is atomic and keeps every endpoint addressable, via `churl_core::persistence::create_endpoint_with`.** It claims `slug(name)`, saves the *real* endpoint under a single claim guard, and commits only after the save succeeds — so a refused literal secret (`SecretsRefused`/`SecretsInAuth`) or a disk error leaves the workspace unchanged (the naïve `create_endpoint` + overwriting `save_endpoint` would commit a placeholder first and orphan it on the second failure). On a filename collision the claim bumps to `<slug>-N.toml` **and** rewrites the saved endpoint's `name` to that stem, so two same-URL imports never share a `name` (which would make the second unreachable via `run`, since resolution is first-by-name); the rename is surfaced as a loud warning, never silent.
- **`churl tutorial` is hard-removed, no alias, no deprecation shim.** Its scaffold logic moved to `churl init --demo`; plain `churl init` scaffolds a blank `churl.toml` only. `init`'s target defaults to the **current directory** (git-init-like) rather than tutorial's old always-a-fresh-subdirectory default, and refuses only when a `churl.toml` already exists at the target — not "the whole directory must be empty" (tutorial's rule only made sense because it always created a dedicated fresh subdirectory).
- **`-X`/`--method` on `send` is validated against churl's closed `Method` enum, not curl's free-form string.** curl accepts any `-X` value verbatim; churl's `Request.method` is a 7-member enum the whole codebase (persistence, TUI method-cycle, execute) already assumes closed. An unsupported method is therefore a clap usage error (band 2), not a runtime failure — a small, deliberate divergence from curl parity in exchange for keeping `Method` closed everywhere else.
- **No cookie jar for headless `run`/`send`.** A one-shot process builds exactly one client via `build_client_with` and exits; `ClientConfig.cookies` stays `None`. The persistent per-workspace jar remains reachable headlessly via the pre-existing `churl cookies list|clear`.
- **Shell completions (`clap_complete`) and a man page (`clap_mangen`) are runtime subcommands (`churl completions <shell>`, `churl man`), not a build-time step.** Keeps them versioned with the binary (no packaging step can go stale) at the cost of generating them on demand rather than shipping a pre-rendered file.

## Assertions (M8.4)

- **An assertion's `target` IS an extraction expression — the M7.4 sequence grammar, reused verbatim rather than forked.** `churl_core::assert::Assertion { target, op, value }` runs `target` through the exact same `sequence::extract_value` every sequence-step `[step.extract]` rule uses (`status` / `header:<Name>` / a dependency-free JSON path). Two parsers for "get a value out of a response" would drift; one grammar, two consumers (extraction-into-a-variable, extraction-into-a-comparison) is the only new surface. `AssertOp` is a closed, exhaustively-matched enum (`Eq`/`Ne`/`Contains`/`Exists`/`Absent`/`Lt`/`Gt`/`Le`/`Ge`) serialized as its canonical string (`==`, `contains`, …) — mirroring `Method`'s own string (de)serialization — never the Rust variant name, so the persisted TOML and the JSON report both read as the human `--assert` syntax.
- **Exit 1 is a *success*-with-failed-assertions path, structurally distinct from `ErrorKind` — never shoehorned into the closed error enum.** A request whose assertions failed still executed and got a response; `output::emit`'s `Ok` branch prints the normal success envelope (`ok: true`, `data` populated, `error: null`) and only the **process exit code** signals the failure, via a new `SuccessExitCode` trait (`crates/churl/src/output.rs`) every `emit`-able payload implements (default `0`; `headless::ExecData` overrides to `1` when `assertions.passed == false`). This is a **deliberate, narrow exception** to the frozen "`ok` mirrors the exit code" rule stated in `docs/CLI.md` — callers must branch on `data.assertions.passed`, never on `ok`, to detect an assertion failure. `ErrorKind::InvalidAssertion` (band 5) is the *separate*, genuine error path for a **malformed** `--assert` flag (a usage/input mistake caught before any request runs) — that one **does** get a normal `ok:false` envelope, because parsing happens pre-flight, not post-response.
- **Assertions populate the M8.2-reserved `data.assertions` key without a schema-version bump.** `null` stays the shape for an assertion-free call (unchanged M8.2 behaviour, still checked by a back-compat test); a non-empty assertion set replaces it with the `AssertionReport` object (`passed`/`total`/`failed`/`results[]`). Per `docs/CLI.md`'s own additive rule, populating a previously-reserved-`null` key with a concrete shape is exactly the case the reservation was built for — `SCHEMA_VERSION` stays `1`.
- **`assertions` lives on `Endpoint`, not `Request` — a deliberate placement choice, not the letter of the original design brief.** `Request` is deliberately kept an *exhaustive*, field-by-field-constructed serde DTO (see "Published result/output/summary structs are `#[non_exhaustive]`; serde model DTOs stay constructible" above) with ~45 literal-construction call sites across the codebase, most of them bare HTTP-execution test fixtures (`tests/http.rs`, `load.rs`, `tls.rs`, the URL-bar/send-command/in-flight-request builders) that have nothing to do with a *persisted, saved* endpoint. `Endpoint` — a 3-field struct with roughly two-thirds as many construction sites, all of them already endpoint-shaped — is the cleaner seam: assertions are a property of a *saved invocation* (like `SequenceStep.extract`/`.persist`), not of the wire-shaping `Request` data `execute()`/`export_curl` consume. Consequence: the on-disk shape is a **top-level** `[[assertions]]` array-of-tables (sibling of `[request]`), not the originally-sketched `[[request.assertions]]` — documented here since an early draft of this milestone's brief assumed the latter.
- **`churl run`'s effective assertion set is the endpoint's persisted `[[assertions]]` THEN its CLI `--assert` flags, in that order (append, never override).** Matches the general "CLI is the highest-precedence *addition*, not a replacement" posture the rest of the CLI surface uses for `--var`. `churl send` has no persisted endpoint, so its CLI flags are the entire set. Evaluation happens in `run_cmd`/`send_cmd` right after `run_execution` returns (not inside the shared executor itself) — `run_execution` stays a pure "resolve → execute → shape" seam or-else `assertions` couples the executor to two different call sites' merge policies, so parsing+merging lives with each command's own args instead.
- **A failed extraction is a value-op failure, not a panic or a silently-passing default.** `Eq`/`Ne`/`Contains`/`Lt`/`Gt`/`Le`/`Ge` against a target that fails to extract (missing header, bad JSON, out-of-range index, …) is `pass: false` with the extractor's own `ExtractError` message surfaced as the reason — never a fabricated "0" or empty-string comparison. `Exists`/`Absent` instead branch on *which kind* of extraction failure occurred (`assert::is_not_found`, an exhaustive match over `ExtractError` deliberately not marked to skip `#[non_exhaustive]`'s external-crate leniency, since `assert` lives inside `churl-core` itself): a missing header/key/index or a JSON `null` leaf reads as "not found" (the only case `Absent` accepts), while a malformed expression or non-JSON body is a genuine error neither op treats as a clean pass.
- **Regex `matches` is deferred to the backlog (below), not shipped in M8.4.** The op set covers exact/substring/existence/numeric comparison — the common assertion shapes for a curl-replacement TUI — without pulling in a `regex` dependency or its match-semantics surface (anchoring, flags, capture groups) on day one. `AssertOp`'s design keeps the parser/evaluator/canonical-string plumbing centralized in three functions (`canonical`/`from_token`/`evaluate_value_op`), so adding a `matches` variant later is a small, additive change — not a structural one.
- **A value-less op (`exists`/`absent`) with trailing text is a parse error, not silently ignored.** `id exists == 200` is almost always a user who meant equality and mistyped the operator word; dropping the `== 200` would hand them a check that passes on *every* run — the worst failure mode for a testing tool (a false green). `Assertion::parse` rejects it with `UnexpectedValue`, symmetric with the `MissingValue` rejection for value-ops. **Assertions persist only through native TOML; foreign interchange (OpenAPI/Postman/.http) has no assertion concept, so import seeds an empty set and export omits the field** — a deliberate lossy boundary, not a round-trip bug.

## Headless sequence runs (M8.4.1)

- **The per-step envelope is the frozen single-request `data`, not a new shape — one masking surface, not two.** `run-seq` streams NDJSON where each step line's `data` is the *identical* `ExecData` a standalone `run`/`send` emits. This is enforced structurally: the request-echo masking + response projection + assertion-report shaping was extracted from `run_execution` into one pure `crate::headless::shape_exec_data(request, response, assertions, trace)` that *both* the single-request path and the per-step runner call. Duplicating that projection per caller would have forked a **security-sensitive** surface (secret masking of the request echo) — the exact drift risk the extraction removes. A byte-identical test (`cli_run_seq::per_step_data_is_byte_identical_to_a_single_run`, modulo the volatile `Date`/timing) pins it.
- **Each NDJSON line is a self-contained, envelope-shaped object with a `type` discriminator; the run ends with one `type:"summary"` line.** Step lines carry the full `{ schema_version, ok, command, data, error }` plus `type`/`seq`/`endpoint` (and `skipped`/`extract_error` when set), so a consumer parsing the single-request envelope already understands a step line. `schema_version` repeats per line deliberately — the NDJSON ideal is that any line is independently parseable (a tail-follower that missed the header still knows the schema). The **summary line is the freeze-once design asset**: overall `ok`, `steps{total,ran,skipped,failed}`, and aggregate `assertions{total,passed,failed}`.
- **A step's persisted `[[assertions]]` ride through `PreparedStep`, loaded once.** `prepare_step` already loads each step's endpoint to resolve its request; it now also returns `endpoint.assertions` on the (newly `#[non_exhaustive]`) `PreparedStep`, so the headless runner gates each step without a second disk read. The live TUI runner ignores the field (it does not yet evaluate sequence assertions — a deliberate headless-first scope).
- **A broken extraction chain fails the run (exit 1), surfaced via `extract_error` — closing a real CI footgun the milestone's literal exit rule left open.** Under the default `on_error: halt`, a step that returns 200 but whose `extract` rule captures nothing (`$.token` missing) halts the run and skips the tail. A literal reading of "exit 1 only on a failed assertion" would exit **0** there — a green CI run for a sequence whose chain silently broke. A broken chain is unambiguously a run failure (a downstream `{{captured}}` can never resolve), so it exits 1, with the reason on the step line's `extract_error`. This is a deliberate *completion* of the underspecified exit model, not a contradiction of it — flagged for owner review.
- **An unasserted HTTP error status (≥400) is NOT an exit trigger, keeping `run-seq` consistent with single-request `run`.** A completed 4xx/5xx is `ok:true`, exit-neutral unless an assertion (`status < 400`) catches it — the same rule single-request `run` follows, preserving the "one assertion model, three scopes" unification. It *is* visible: it counts in `steps.failed`, shows its status in `data.response.status`, and (under the default halt) skips the tail (`skipped:true`). Exit precedence: a transport/resolution band (3/4/5) wins over an assertion/extraction failure (1) — "request errors still win."
- **Halt/continue and value-chaining reuse the core engine's exact seams (`ordered_steps`/`prepare_step`/`classify_step`/`should_halt`); only assertion evaluation + the output stream are new.** Flow control therefore cannot drift from the TUI runner or the wiremock-tested `run_sequence`. Assertion evaluation is layered *on top* (it never halts the chain — `on_error` governs flow via extraction/status/transport results; assertions are observations that affect only the exit code), so an interactively-authored sequence runs byte-identically headless.
- **No cookie jar; chaining is via extracted values.** Consistent with the M8.2 "no jar for one-shot headless" decision — a sequence's session state flows as extracted tokens (a login body's `$.token` → a later `Authorization: Bearer {{token}}`), the documented mechanism, not implicit cookies. One client is built **per step** (honouring that step's endpoint's durable `insecure` flag, parity with `run`), not one shared client — correctness over connection reuse for a short test flow.
- **Addressed by file stem, resolved by enumerating real `sequences/*.toml` files — traversal-safe by construction.** `run-seq ../etc/passwd` is a plain `sequence-not-found`, never a path escape, because resolution matches the arg against the stems of files actually in `sequences/`, rather than joining it into a path. The lenient loader's warnings (a malformed *sibling* sequence) surface on stderr, never silently swallowed.
- **`run-seq` owns its stdout/stderr/exit directly rather than funnelling through `output::emit`.** `emit` prints exactly one envelope; a stream needs many. So `run-seq` prints each line itself and returns the exit code, while **pre-flight** failures (no workspace, sequence not found, bad `--profile`) — which happen before the stream starts — reuse the extracted `output::emit_error` to emit the same single error envelope every other headless command does.
- **No per-step CLI `--assert` override in M8.4.1 — deferred, not dropped.** The endpoint-persisted `[[assertions]]` path fully serves the motivating CI gate (login→cart→order→cancel, each endpoint self-gated). A step-qualified CLI flag (`--assert 'login: $.token exists'`) needs a new step-addressing grammar (by `seq`? by endpoint path? ambiguous when a sequence repeats an endpoint) whose wrong freeze would be costly — so it is recorded here rather than shipped half-designed.

## Headless load runs (M8.4.2)

- **A load run is one aggregate result → a single envelope, not per-request NDJSON.** `churl load` reuses the exact `output::emit` seam `run`/`send` use (`command:"load"`, `data:{config,stats,assertions}`), NOT `run-seq`'s NDJSON stream. A load run's meaningful output is the *aggregate* (counts, percentiles, throughput); streaming thousands of per-request lines would flood a consumer with data it has to re-aggregate anyway, and the percentiles can only be computed once every request is in. The per-request outcomes stay internal to `run_load`; only the rollup crosses the envelope boundary. (`SuccessExitCode` carries the exit-1-on-failed-assertion path, unchanged from M8.4.)
- **The success/error rates are methods on `LoadStats` (in `churl-core`), the single source of truth both the TUI and the headless JSON call.** `LoadStats::attempted()` (`ok+failed+errored`), `success_rate()` (`ok/attempted`), `error_rate()` (`(failed+errored)/attempted`), each `Option` (`None` over a zero-attempted run). Defining "success rate" once in the model — rather than open-coding `ok as f64 / total` at each presentation site — means the TUI stat line and the `stats.success_rate` assertion/JSON can never disagree on the denominator (attempted, not the *configured* `total` that includes not-yet-completed copies). Latency stats stay computed over *completed* requests only, unchanged. (The current TUI stat line shows raw counts, not a rate, so there is nothing to redirect yet; the methods are the canonical definition any future rate display must use.)
- **`stats.*` is a new target-resolver arm on the existing assertion grammar — the op grammar is reused verbatim.** `parse_stats_assertion` calls the same `Assertion::parse` (`<target> <op> <value>`, same `AssertOp` set) response assertions use, then validates the target against a closed `StatTarget` enum (`churl_core::load`). Numeric comparison reuses the response evaluator (`assert::evaluate_value_op`) unchanged — only *resolution* differs (a `stats.<field>` against `LoadStats`+`rps`, vs. an extraction expression against a `Response`). This is the "one assertion model, three scopes" story's third scope (response → flow → **aggregate**): a single grammar, three target resolvers, never three parsers. A `StatAssertion` is constructed only via `parse_stats_assertion`, so an unresolvable `stats.foo` (or a response target like `status`) cannot reach evaluation — it is a pre-flight band-5 `invalid-assertion`, illegal-states-unrepresentable.
- **Stat units are a freeze-once contract: latencies in integer milliseconds, rates in `0..1`, `rps` a float, counts integers.** Pinned in `docs/CLI.md`. Latencies resolve via `Duration::as_millis` — the **same** rounding the `*_ms` envelope fields use (a shared `StatTarget::resolve` derivation) — so an asserted value equals the reported one and the two can't drift (a `stats.p95 < 500` sees the same integer the JSON's `p95_ms` shows). An undefined stat (a latency over an all-errored run, a rate over a zero-attempted run) resolves to `None`: `exists`/`absent` test that definedness; a value-op against it fails loudly rather than comparing a fabricated zero.
- **`rps` is measured in the bin caller by wrapping `run_load` in an `Instant`, not by widening the core signature.** `run_load` uses `start.elapsed()` internally for pacing but returns no elapsed; rather than change its (frozen, TUI-shared) return type to thread a wall-clock out, `load_cmd` measures `Instant::now()` around the call. `rps = attempted / elapsed_secs` — **attempted**, not completed, so it reads as the offered load rate ("we fired N requests in T seconds"); `None` when nothing was attempted or the run took no measurable time (guards a divide-by-zero `inf`). The same `rps` value is handed to both the envelope and the assertions, so reported and asserted throughput are literally one number.
- **Per-request transport failures are NOT a command error — they land in `stats.errored`.** A load run whose requests all time out is still a *successful command* (the aggregate is the result: `errored == count`, exit 0 absent an assertion). Only a failure to *build the client* is a band-4 error. This is deliberate: a QA/CI load test wants the error *rate* as data (`--assert 'stats.error_rate <= 0.01'`), not a hard exit that hides the number.
- **Guardrail caps: the hard-cap `Refuse` tier IS enforced headless (added during the adversarial gate); the soft `Warn` tier degrades to a stderr note.** The TUI load runner gates large runs behind `check_config`'s three-tier `Ok`/`Warn`/`Refuse`. The `Refuse` tier is an absolute ceiling ("a testing aid, not a load-cannon") — dropping it on the headless path was a real safety regression (a CI typo `--total 100000` could DoS a target), flagged by the builder and both adversarial reviewers, so `load` now calls `check_config` pre-flight and maps `Refuse` onto a new band-5 `load-cap-exceeded` before any request fires (raise `[load] max_total`/`max_concurrency` to lift it — the message says so). The `Warn` tier is a *soft confirm* with no TTY to confirm at headlessly, so it prints a loud `warning:` line to stderr and proceeds — the caller chose an explicit, below-ceiling number. (An earlier draft deferred the whole thing pending an owner call; the gate closed the hard-cap half since it mirrors existing TUI behaviour and reuses `check_config` verbatim, and left the ceiling owner-tunable via `[load]`.)
- **`load` gates on its CLI `stats.*` `--assert` flags only — an endpoint's persisted `[[assertions]]` are not applied.** Those check a single *response*; a load run has no single response, only the aggregate. Documented in `docs/CLI.md` so it is not a surprise (unlike `run`, which appends CLI asserts after the endpoint's persisted set).

## Settings panel: config-writer & persistence semantics

- **Persistence is session mirror + explicit Save-as-default, never implicit — and Save writes only knobs the user NET-CHANGED in the panel, never a CLI/workspace/global-config value the panel merely displays.** A settings-panel edit applies to the live session immediately (client rebuild where a seam exists, exactly like the session-scoped `<leader>k`/proxy/cookies controls always have); nothing touches `config.toml` until the user explicitly saves, and even then a knob is written only if its value differs from a net-change baseline (a per-knob touched-set driven by a value baseline captured at panel open and re-captured after each save). Net change, not interaction: a toggle round-trip back to origin, or an Enter-commit of a seeded value that didn't change it, un-marks the knob — so merely opening the proxy editor on a forced proxy and pressing Enter never persists it. A CLI flag or workspace override the panel simply resolved into the session and displayed is invisible to the writer, however much it differs from what's on disk — the dirty dot follows the same touched-set (touched AND differs from disk, never a bare value comparison), so the indicator and Save always agree. This closes the footgun where `churl -k` (or any other forced value) followed by an unrelated edit — or even just interacting with the forced knob — used to silently freeze it into the global default.
- **The net-change baseline is SESSION-LIFETIME, not re-captured on every panel open (M8.5.3 fix).** The baseline is seeded once, on the panel's first-ever open this session, and re-seeded only by a successful Save — a bare close/reopen leaves it untouched. The earlier per-open capture re-baselined from the LIVE working copy on every open, which was silently wrong across a close-without-save: edit a knob (touched, baseline stale), close, reopen (baseline WRONGLY re-seeded from the now-dirty live value, since it equals the live value by construction), then a net-zero-looking adjust (e.g. quick-adjust up then back down) compared against that wrong baseline and un-marked the knob — Save then wrote nothing, so the edit was never persisted despite never having been explicitly discarded. `App::settings_baseline_established` (a `bool`, false only before the first open) is the guard; FIX-1's net-zero semantics (a knob re-committed to its startup/CLI/workspace-forced value stays untouched) are unchanged, since the baseline is still exactly the session-start value in the common case.
- **The config-writer is a `toml_edit` DOM patch over a sparse edit set, not a `Config` re-serialize.** `save_defaults` takes a struct with an `Option` per managed knob (`Some` = touched, persist it; `None` = untouched, its on-disk key is never even read) and reads the existing file into a `DocumentMut` (an empty document when the file is missing — a missing file is not an error, mirroring the loader), sets only the TOUCHED keys directly on the DOM (skipping the write entirely when the value wouldn't actually change, so a no-op never churns a key's decor), and writes back atomically (temp file + rename). A plain `toml::to_string(&Config)` would nuke the user's comments, key order, and any key churl doesn't model; the edit struct additionally carries no fields at all for `[keys]`/`[theme_colors]`/`raw_keys`, so those tables are structurally unreachable from the writer regardless of what future code does with it.
- **Setting a touched value back to its built-in default removes the key rather than writing it explicitly.** A touched field is compared against the same default its own resolver method falls back to before deciding to set-or-remove; an all-default sub-table (`[load]`, `[advanced]`) is pruned entirely once empty — but only for the sub-fields actually touched, never a table the user never edited this session. A config.toml a user never touched stays byte-quiet; "reset then save" cleans up a stale override instead of freezing it in place.
- **A credential-bearing proxy is skipped, not blocked, on write.** `proxy_has_credentials` gates the proxy key alone: a panel-typed credentialed proxy is never written to disk (matching the existing workspace-manifest rule), but it does not fail the rest of the save — every other touched knob in the same call still persists, and a toast names what was skipped and why. Refusing the whole save over one knob was worse than skipping just that one; a CLI/workspace-forced credentialed proxy the user never touched in the panel never reaches the writer at all, so it was never at risk of being written either way.
- **A quick-adjust floor clamps to the step, not to the bare positive-whole-number floor `1` (M8.5.3 fix).** `step_u64`/`step_usize` (the `J`/`K` quick-adjust in `edit.rs`) used to clamp a decrement at `1` regardless of the knob's step — fine for a step-1 knob (concurrency/timeout), but a step-10 (Total) or step-`1 MB` (body cap) knob could be walked off its grid down to a single unit (e.g. 1 byte for a body cap), which `commit_edit`'s own typed-entry floor (`> 0`) would never itself produce. The floor is now `step` (the smallest positive grid value), so every quick-adjusted value stays step-aligned regardless of direction.
- **`<leader>r` (`reload_workspace`) doubles as the leader key's live-apply gate (M8.5.3), reversing the earlier "no cheap re-parse seam" premise.** `KeyMap::set_leader(&str) -> Result<()>` already existed and is cheap (crokey parse + normalize); the panel's `ApplyLeaderKey` outcome still only updates the working copy (a leader edit must never change the meaning of a key mid-combo), but `<leader>r` now also calls `set_leader` on the working copy — parse-guarded, so a bad combo keeps the old leader and reports the error rather than bricking input — and its confirmation toast always names the leader-key outcome, independent of whether the workspace-reload portion of the same keypress proceeded. The panel's toast on `ApplyLeaderKey` was reworded from "applies on next launch" to name the actual apply gate.

## Cookie model: SameSite + manual add/edit (M8.5.1)

- **SameSite is a local enum, not `cookie::SameSite`, on the public `CookieView`.** Re-exporting a dependency's enum through churl's own API ties a patch bump in that dependency to churl's semver (the same 0.5.0 `#[non_exhaustive]` lesson) — `churl_core::cookies::SameSite { Strict, Lax, None }` is churl's own, mapped to/from `cookie::SameSite` only at the `list()`/`upsert()` boundary. `CookieView` stays a plain struct literal (not `#[non_exhaustive]`), since in-workspace test code (the panel's render test) still constructs it directly.
- **A manually-entered cookie is always inserted as *persistent* (a far-future `Expires`) and *host-only*, via `insert_raw` against a synthesized `https://{domain}{path}` request URL.** A session cookie (no `Max-Age`/`Expires`) would evaporate before ever reaching `to_json`, and a hand-typed cookie is usually an auth token the user explicitly wants to stick. The URL is always `https` so a `Secure` cookie is never rejected at insert. **No `Domain=` attribute is set** — the cookie is scoped host-only (RFC 6265), sent back to the exact host the user typed and never to a subdomain. This is the browser default for a hand-typed host and the safe direction: silently broadening an auth token to every `*.{domain}` is the cross-origin over-scoping churl's redirect-strip policy (R3) exists to prevent. Subdomain-wide (`Domain=`) manual cookies are a deliberate non-goal for now. The store's own rejection (an unparseable host, a mismatch) is always surfaced as `CookieError`, never swallowed as a silent no-op — a cookie jar must never claim to hold something it doesn't.
- **Edit-that-changes-key = upsert-FIRST, then `delete_exact` the old coordinate, in the app handler (not inside `upsert` itself).** The store keys cookies on `(domain, path, name)`; `upsert` only REPLACES a same-key entry, so a changed domain/name/path leaves the old entry as a different row that must be removed. The removal uses the **path-precise** `ChurlCookieJar::delete_exact(domain, path, name)` (surfacing the store's own `(domain, path, name)` `remove`), **not** the domain+name-scoped `delete` (which wipes every path under that pair — fine for the "clear by name" list UI, but a silent-data-loss bug here: editing `sid@/app`'s path would also delete an untouched `sid@/admin` sibling). Ordering is upsert-then-delete: because the new key differs from the old by definition of "changed," `delete_exact` on the old coord can't touch the just-added row, and a *failed* upsert skips the delete entirely, so the original always survives (closing the earlier "original gone if the edited domain no longer parses" gap). Proven by a regression test that edits one path of a two-path same-name pair and asserts the sibling survives.
- **SameSite is stored and displayed but NOT enforced on churl's request-send path.** The cookie store does no `SameSite` filtering, and a CLI/TUI request has no cross-site request context to filter against — so `SameSite` here is advisory round-trip metadata (shown in the list, editable in the form, persisted), not an emission control. Nobody should assume a `SameSite=Strict` manual cookie is withheld from any request churl makes.

## Foundational choices

- **Name: churl** — "curl" in a TUI shell; short, not taken on crates.io at decision time.
- **Default branch: master** — matches the host platform default.
- **Platform paths: `dirs` crate** — config at `<config_dir>/churl/config.toml`, state at `<data_dir>/churl/state.sqlite`; hand-rolling XDG-vs-macOS-vs-Windows resolution isn't worth the bugs.
- **Vim-modal editing: edtui** — `tui-textarea` is unmaintained; edtui is active and ships modal vim out of the box.
- **Fuzzy engine: `nucleo-matcher`** (not the threaded `nucleo` crate) — same scoring engine, synchronous API; the threaded streaming matcher is overkill for workspace-sized lists and would drag a worker-pool lifecycle into the TUI.
- **HTTP mocking: wiremock; snapshot tests: insta + ratatui `TestBackend`** — wiremock exercises the full reqwest stack without network; insta gives deterministic pixel-level layout-regression assertions.
